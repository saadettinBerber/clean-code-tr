firstName, lastName, street, houseNumber, city, state ve zipcode değişkenlerini bir arada gördüğünüzde, bunların bir adresi oluşturduğunu anlarsınız. Ancak sadece state değişkenini tek başına görseydiniz, bunun bir adresin parçası olduğunu otomatik olarak varsayar mıydınız? Imagine that you have variables named firstName, lastName, street, houseNumber, city, state, and zipcode. Taken together it’s pretty clear they form an address. But what if you just saw the state variable being used alone? Would you automatically assume it was part of an address?
Önekler (prefixes) kullanarak bağlam ekleyebilirsiniz: addrFirstName, addrLastName, addrState vb. En azından okuyucu bu değişkenlerin daha büyük bir yapının parçası olduğunu anlar. Elbette daha iyi bir çözüm, Address adında bir sınıf (class) oluşturmaktır; böylece derleyici bile değişkenlerin daha büyük bir kavramın parçası olduğunu bilir. You can add context by using prefixes: addrFirstName, addrLastName, addrState, and so on. At least the reader will understand that these variables are part of a larger structure. Of course, a better solution is to create a class named Address. Then, even the compiler knows that the variables belong to a bigger concept.
Listing 2-1'deki fonksiyonu inceleyin. Değişkenlerin anlamlı bir bağlama ihtiyacı var mı? Consider the function in Listing 2-1. Do the variables need a more meaningful context?
Listing 2-1: Net olmayan bağlama sahip değişkenler / Variables with unclear context Listing 2-1: Variables with unclear context 
private void printGuessStatistics(char candidate, int count) {
    String number;
    String verb;
    String pluralModifier;
    if (count == 0) {
        number = "no";
        verb = "are";
        pluralModifier = "s";
    } else if (count == 1) {
        number = "1";
        verb = "is";
        pluralModifier = "";
    } else {
        number = Integer.toString(count);
        verb = "are";
        pluralModifier = "s";
    }
    String guessMessage = String.format(
        "There %s %s %s%s", verb, number, candidate, pluralModifier
    );
    print(guessMessage);
}
Fonksiyonun biraz uzun olduğu ve değişkenlerin en başta tanımlandığı görülüyor. Fonksiyonun ismi bir bağlam sağlasa da, kodu okurken number, verb ve pluralModifier değişkenlerinin "tahmin istatistikleri" (guess statistics) mesajının bir parçası olduğunu anlamak için tüm metodu analiz etmeniz gerekir. Bağlamın (context) örtük (implicit) olduğu ortadadır; onu açık (explicit) hale getirmeliyiz. The function is a bit overlong and the variables are used throughout. While the function name provides a context, you have to read the entire method to know that number, verb, and pluralModifier are part of the guess statistics message. The context is implicit; we must make it explicit.
Bunu yapmanın bir yolu, GuessStatisticsMessage adında bir sınıf oluşturmak ve bu üç değişkeni bu sınıfın alanları (fields) haline getirmektir. Bu sayede bağlamın parçası olan değişkenler net bir şekilde gruplanmış olur. One way to do this is to create a class named GuessStatisticsMessage and make the three variables fields of this class. This provides a clear context for the variables.

Anlamlı Bağlam (Meaningful Context)Meaningful Context

Kavram AçıklamasıConcept Explanation

Değişkenler tek başlarına muğlak olabilir. state isminde bir değişken, bir adresin eyaletini mi, bir siparişin durumunu mu yoksa bir makinenin o anki halini mi temsil ediyor? Değişkeni anlamlı bir yapının (sınıf, fonksiyon) içine koyarak ona bir "ev" ve dolayısıyla net bir anlam kazandırırız.Variables can be ambiguous on their own. Does a variable named state represent an address state, an order status, or a machine's current state? By putting the variable inside a meaningful structure (class, function), we give it a "home" and thus a clear meaning.

Kötü Örnek (Before)Bad Example (Before)

KÖTÜ — Bağlamı anlamak için kodun tamamını okumak gerekiyorBAD — Must read entire code to understand context 
public void logStatus(String msg, int code) {
    // code burada ne kodu? HTTP mi? Hata mı?
    System.out.println("Log: " + msg + " [" + code + "]");
}

İyi Örnek (After)Good Example (After)

İYİ — Açık bağlam (Explicit Context)GOOD — Explicit Context 
public class HttpResponse {
    private String message;
    private int statusCode; // Bağlam netleşti

    public void log() {
        System.out.println("Response: " + message + " (Status: " + statusCode + ")");
    }
}
Pratik İpucuPractical TipOkuyucunun bağlamı (context) "kazıyıp çıkarmasına" (dig out) izin vermeyin. Bağlamı kodun yapısıyla (sınıf hiyerarşisi vb.) göz önüne serin.Don't make the reader "dig out" the context. Lay it out through the structure of your code (class hierarchy, etc.).

Address Sınıfı Çözümü (Address Class Solution)Address Class Solution

Kavram AçıklamasıConcept Explanation

Birbirleriyle ilişkili değişkenler (street, city, zip) genellikle bir "kavramsal bütün" oluşturur. Bunları ayrı ayrı taşımak yerine bir sınıf altında toplamak (Data Clump refactoring), hem bağlamı güçlendirir hem de kodun taşınabilirliğini artırır.Related variables (street, city, zip) often form a "conceptual whole." Instead of carrying them separately, grouping them under a class (Data Clump refactoring) both strengthens context and increases code portability.

İyi ÖrnekGood Example

İYİ — Nesne tabanlı bağlamGOOD — Object-oriented context 
public class MailingAddress {
    private String street;
    private String city;
    private String postalCode;

    public String formatForLabel() {
        return street + ", " + city + " " + postalCode;
    }
}
Pratik İpucuPractical TipEğer bir grup değişken sürekli birlikte metotlara parametre olarak gönderiliyorsa, bu bir "Data Clump" (Veri Kümesi) kokusudur. Bunları hemen bir sınıfa dönüştürün.If a group of variables is constantly passed together as parameters to methods, it's a "Data Clump" smell. Convert them into a class immediately.
