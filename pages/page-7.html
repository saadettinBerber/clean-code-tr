Temiz kod yazmak, zahmetli bir süreçte edinilen bir "temizlik" duygusuyla uygulanan sayısız küçük tekniğin disiplinli kullanımını gerektirir. Bu "kod duygusu" (code-sense) anahtardır. Bazılarımız onunla doğar. Bazılarımız onu edinmek için savaşmak zorundadır. Kod duygusu yalnızca kodun iyi mi kötü mü olduğunu görmemizi sağlamakla kalmaz, aynı zamanda kötü kodu temiz koda dönüştürmek için disiplinimizi uygulama stratejisini de gösterir. Writing clean code requires the disciplined use of a myriad little techniques applied through a painstakingly acquired sense of "cleanliness." This "code-sense" is the key. Some of us are born with it. Some of us have to fight to acquire it. Not only does it let us see whether code is good or bad, but it also shows us the strategy for applying our discipline to transform bad code into clean code.
"Kod duygusu" (code-sense) olmayan bir programcı dağınık bir modüle (module) bakıp karmaşayı fark edebilir ama ne yapacağı hakkında hiçbir fikri olmaz. "Kod duygusu" olan bir programcı ise dağınık bir modüle bakıp seçenekleri ve varyasyonları görür. "Kod duygusu", o programcının en iyi varyasyonu seçmesine ve davranışı koruyan dönüşümlerin (behavior preserving transformations) bir dizisini planlayarak mevcut durumdan hedeflenen duruma ulaşmasına rehberlik eder. A programmer without "code-sense" can look at a messy module and recognize the mess but will have no idea what to do about it. A programmer with "code-sense" will look at a messy module and see options and variations. The "code-sense" will help that programmer choose the best variation and guide him or her to plot a sequence of behavior preserving transformations to get from here to there.
Kısacası, temiz kod yazan bir programcı, boş bir ekranı bir dizi dönüşümden (transformations) geçirerek zarif bir şekilde kodlanmış (elegantly coded) bir sisteme dönüştürebilen bir sanatkârdır. In short, a programmer who writes clean code is an artist who can take a blank screen through a series of transformations until it is an elegantly coded system.
Temiz Kod Nedir? / What Is Clean Code? What Is Clean Code?
Muhtemelen programcı sayısı kadar tanım vardır. Bu yüzden çok tanınmış ve derin deneyime sahip bazı programcılara ne düşündüklerini sordum. There are probably as many definitions as there are programmers. So I asked some very well-known and deeply experienced programmers what they thought.
Bjarne Stroustrup, C++'ın yaratıcısı ve The C++ Programming Language kitabının yazarı Bjarne Stroustrup, inventor of C++ and author of The C++ Programming Language
Kodumun zarif (elegant) ve verimli (efficient) olmasını severim. Mantık, hataların (bugs) saklanmasını zorlaştıracak kadar açık olmalı; bağımlılıklar (dependencies) bakımı kolaylaştırmak için minimum düzeyde tutulmalı; hata yönetimi (error handling) açıkça ifade edilmiş bir stratejiye göre eksiksiz olmalı; ve performans, insanları ilkesiz optimizasyonlarla (unprincipled optimizations) kodu kirletmeye kışkırtmamak için optimale yakın olmalıdır. Temiz kod bir şeyi iyi yapar. I like my code to be elegant and efficient. The logic should be straightforward to make it hard for bugs to hide, the dependencies minimal to ease maintenance, error handling complete according to an articulated strategy, and performance close to optimal so as not to tempt people to make the code messy with unprincipled optimizations. Clean code does one thing well.
Bjarne "zarif" (elegant) kelimesini kullanıyor. Bu oldukça güçlü bir kelime! MacBook®'umdaki sözlük şu tanımları veriyor: görünüm veya tavır bakımından hoş derecede zarif ve şık; hoş derecede yaratıcı ve basit. "Hoş" (pleasing) kelimesine yapılan vurguya dikkat edin. Görünüşe göre Bjarne, temiz kodun okunmasının hoş olduğunu düşünüyor. Kodu okumak, iyi yapılmış bir müzik kutusu ya da iyi tasarlanmış bir araba gibi sizi gülümsetmelidir. Bjarne uses the word "elegant." That's quite a word! The dictionary in my MacBook® provides the following definitions: pleasingly graceful and stylish in appearance or manner; pleasingly ingenious and simple. Notice the emphasis on the word "pleasing." Apparently Bjarne thinks that clean code is pleasing to read. Reading it should make you smile the way a well-crafted music box or well-designed car would.
Bjarne ayrıca verimlilikten (efficiency) iki kez bahsediyor. C++'ın yaratıcısından geldiği düşünüldüğünde belki bu bizi şaşırtmamalı; ama bence bunun salt hız arzusundan ötesi var. Boşa harcanan döngüler zarafetle bağdaşmaz, hoş değildir. Ve şimdi Bjarne'nin kullandığı kelimeye dikkat edin — Bjarne also mentions efficiency—twice. Perhaps this should not surprise us coming from the inventor of C++; but I think there's more to it than the sheer desire for speed. Wasted cycles are inelegant, they are not pleasing. Kod Duygusu (Code-Sense)Code-Sense

Kavram AçıklamasıConcept Explanation

Martin'e göre "kod duygusu" (code-sense), temiz kod yazmanın anahtarıdır. Bu yeti yalnızca kodun iyi mi kötü mü olduğunu görmenizi sağlamakla kalmaz; aynı zamanda kötü kodu temiz koda dönüştürmek için hangi adımları atacağınızı da gösterir. Kod duygusu olan bir programcı bir karmaşayı gördüğünde seçenekleri ve varyasyonları fark eder — kod duygusu olmayan programcı ise sadece karmaşayı fark edip çaresiz kalır.According to Martin, "code-sense" is the key to writing clean code. This sense not only lets you see whether code is good or bad; it also shows you the strategy for transforming bad code into clean code. A programmer with code-sense sees options and variations when looking at a mess — a programmer without it only recognizes the mess but feels helpless.

KÖTÜ — Karmaşayı fark ediyor ama ne yapacağını bilmiyorBAD — Recognizes the mess but doesn't know what to do

    }

İyi Örnek (After)Good Example (After)

İYİ — Kod duygusuyla dönüştürülmüş: her adım davranışı koruyarakGOOD — Transformed with code-sense: each step preserving behavior 
class ReportBuilder {
    constructor(data) {
        this.pipeline = [...data];
    }

    filterBy(field, value) {
        this.pipeline = this.pipeline.filter(item => item[field] === value);
        return this;
    }

    sortBy(field) {
        this.pipeline.sort((a, b) => a[field] > b[field] ? 1 : -1);
        return this;
    }

    build(strategy) {
        return strategy.generate(this.pipeline);
    }
}

const report = new ReportBuilder(salesData)
    .filterBy('region', 'EMEA')
    .sortBy('revenue')
    .build(new SummaryStrategy());

Extract Method, Replace Conditional with Polymorphism, Introduce Parameter Object gibi refactoring tekniklerini pratik edin.To develop code-sense, do "refactoring katas": deliberately take a messy function and apply small transformations, ensuring tests pass at each step. Practice techniques like Extract Method, Replace Conditional with Polymorphism, Introduce Parameter Object.

Zarif Kod (Elegant Code)Elegant Code

Kavram AçıklamasıConcept Explanation

Bjarne Stroustrup temiz kodu tanımlarken "zarif" (elegant) kelimesini kullanır. Martin bu kelimeyi sözlükten açar: "görünüm veya tavır bakımından hoş derecede zarif ve şık; hoş derecede yaratıcı ve basit." Zarif kod okunması hoş olan koddur — iyi yapılmış bir müzik kutusu gibi sizi gülümsetir. Aynı zamanda verimlidir (efficient): boşa harcanan döngüler zarafetle bağdaşmaz.Bjarne Stroustrup uses the word "elegant" when defining clean code. Martin looks it up: "pleasingly graceful and stylish in appearance or manner; pleasingly ingenious and simple." Elegant code is pleasing to read — it makes you smile like a well-crafted music box. It's also efficient: wasted cycles are inelegant.

Kötü Örnek (Before)Bad Example (Before)

KÖTÜ — Ne zarif ne de verimliBAD — Neither elegant nor efficient 
def find_duplicates(items):
    duplicates = []
    for i in range(len(items)):
        for j in range(len(items)):
            if i != j and items[i] == items[j]:
                if items[i] not in duplicates:
                    duplicates.append(items[i])
    return duplicates

# O(n²) karmaşıklık, okunması zor, niyeti belirsiz
İç içe döngü hem verimsiz (O(n²)) hem de okunması güç. Kodun amacı (niyet) iç içe geçmiş mantığın arkasına gizlenmiş durumda.Nested loops are both inefficient (O(n²)) and hard to read. The code's intent is buried behind tangled logic.

İyi Örnek (After)Good Example (After)

İYİ — Zarif: basit, verimli, niyeti açıkGOOD — Elegant: simple, efficient, clear intent 
def find_duplicates(items):
    seen = set()
    return {item for item in items if item in seen or seen.add(item)}

# O(n) karmaşıklık, tek satır, niyet kristal netliğinde
# "Daha önce gördüklerimi bul" — hepsi bu
Pratik İpucuPractical TipZarafet testi: Kodunuzu sesli okuyun. Eğer bir cümle gibi akıyorsa, zariftir. Eğer kendinize "bu ne yapıyor?" diye soruyorsanız, henüz zarif değildir. Bjarne'nin dediği gibi: kod sizi gülümsetmeli.Elegance test: read your code aloud. If it flows like a sentence, it's elegant. If you ask yourself "what does this do?", it's not elegant yet. As Bjarne says: code should make you smile.

Bir Şeyi İyi Yap (Does One Thing Well)Does One Thing Well

Kavram AçıklamasıConcept Explanation

Bjarne Stroustrup'un temiz kod tanımı şu cümleyle biter: "Temiz kod bir şeyi iyi yapar." Bu, Tek Sorumluluk İlkesinin (Single Responsibility Principle) özüdür. Her fonksiyon, her sınıf, her modül tek bir amaca odaklanmalıdır. Birden fazla şey yapmaya çalışan kod karmaşıklaşır, test edilmesi zorlaşır ve değişikliklere kırılgan hale gelir.Bjarne Stroustrup's clean code definition ends with: "Clean code does one thing well." This is the essence of the Single Responsibility Principle. Every function, class, and module should focus on a single purpose. Code that tries to do multiple things becomes complex, hard to test, and fragile to changes.

Kötü Örnek (Before)Bad Example (Before)

KÖTÜ — Bir fonksiyon, üç sorumlulukBAD — One function, three responsibilities 
public class UserManager {
    public boolean registerUser(String email, String password) {
        // 1. Validasyon
        if (email == null || !email.contains("@")) return false;
        if (password.length() < 8) return false;

        // 2. Veritabanına kayıt
        Connection conn = DriverManager.getConnection(DB_URL);
        PreparedStatement stmt = conn.prepareStatement(
            "INSERT INTO users (email, password) VALUES (?, ?)");
        stmt.setString(1, email);
        stmt.setString(2, hashPassword(password));
        stmt.executeUpdate();
        conn.close();

        // 3. Hoş geldin e-postası gönder
        Transport transport = Session.getInstance(props).getTransport();
        transport.connect();
        MimeMessage msg = new MimeMessage(session);
        msg.setSubject("Hoş geldiniz!");
        msg.setText("Kayıt başarılı...");
        transport.sendMessage(msg, msg.getAllRecipients());
        transport.close();

        return true;
    }
}
Bu fonksiyon validasyon, veritabanı kaydı ve e-posta gönderimi olmak üzere üç farklı şey yapıyor. Herhangi birinde değişiklik gerektiğinde tüm fonksiyonu anlamak ve riske atmak zorundasınız.This function does three different things: validation, database saving, and email sending. Changing any one requires understanding and risking the entire function.

İyi Örnek (After)Good Example (After)

İYİ — Her bileşen bir şeyi iyi yaparGOOD — Each component does one thing well 
public class UserRegistration {
    private final UserValidator validator;
    private final UserRepository repository;
    private final WelcomeMailer mailer;

    public User register(RegistrationRequest request) {
        validator.validate(request);
        User user = repository.save(request.toUser());
        mailer.sendWelcome(user);
        return user;
    }
}

public class UserValidator {
    public void validate(RegistrationRequest request) {
        requireValidEmail(request.email());
        requireStrongPassword(request.password());
    }
}

public class UserRepository {
    public User save(User user) { /* sadece kayıt */ }
}

public class WelcomeMailer {
    public void sendWelcome(User user) { /* sadece e-posta */ }
}
Pratik İpucuPractical TipFonksiyonunuzun ne yaptığını tek cümlede açıklayın. Eğer cümlede "ve" veya "veya" kullanmak zorunda kalıyorsanız, fonksiyonunuz birden fazla şey yapıyor demektir. "Bu fonksiyon kullanıcıyı doğrular VE kaydeder VE e-posta gönderir" → üç fonksiyon olmalı.Describe what your function does in one sentence. If you need "and" or "or" in that sentence, your function does more than one thing. "This function validates the user AND saves AND sends email" → should be three functions.
