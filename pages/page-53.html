Yorumlar / Comments Comments
"Kötü kodu yorum satırı ekleyerek kurtarmaya çalışma — onu yeniden yaz." — Brian W. Kernighan ve P. J. Plaugher1 "Don't comment bad code—rewrite it." — Brian W. Kernighan and P. J. Plaugher1
Doğru yere yerleştirilmiş bir yorum kadar yararlı hiçbir şey olamaz. Gereksiz, dogmatik yorumlar kadar bir modülü kirletecek hiçbir şey yoktur. Yanlış bilgi yayan eski ve çürümüş bir yorum kadar zararlı hiçbir şey olamaz. Nothing can be quite so helpful as a well-placed comment. Nothing can clutter up a module more than frivolous dogmatic comments. Nothing can be quite so damaging as an old crufty comment that propagates lies and misinformation.
Yorumlar, Schindler'in Listesi gibi değildir. Onlar "salt iyilik" değildir. Gerçek şu ki: yorumlar en iyi ihtimalle zorunlu bir kötülüktür. Eğer programlama dillerimiz yeterince ifade edici olsaydı ya da niyetimizi bu dilleri ustalıkla kullanarak aktarabilseydik, yorumlara neredeyse hiç — belki de hiç — ihtiyacımız kalmazdı. Comments are not like Schindler's List. They are not "pure good." Indeed, comments are, at best, a necessary evil. If our programming languages were expressive enough, or if we had the talent to subtly wield those languages to express our intent, we would not need comments very much—perhaps not at all.

Zorunlu Kötülük (Necessary Evil)Necessary Evil

Kavram AçıklamasıConcept Explanation

Yorumlar, programcının niyetini kodla ifade edememesinin bir itirafıdır. "Zorunlu kötülük" metaforu şunu söyler: yorumlar bazen kaçınılmazdır, ama hiçbir zaman kutlanmamalıdır. Her yorum yazıldığında, aslında bir başarısızlık belgelenmiş olur. Comments are an admission that the programmer could not express their intent in code. The "necessary evil" metaphor says: comments are sometimes unavoidable, but should never be celebrated. Every time a comment is written, a failure is documented.

Kötü Örnek (Before)Bad Example (Before)

KÖTÜ — Yorumla ifade edilmeye çalışılan niyetBAD — Intent expressed through comment 
// Kullanıcı aktif ve premium üyeyse indirim uygula
if (user.status == 1 && user.type == 2) {
    applyDiscount(order);
}

İyi Örnek (After)Good Example (After)

İYİ — Niyet kodun içindeGOOD — Intent expressed in code 
if (user.isActiveAndPremium()) {
    applyDiscount(order);
}
Pratik İpucuPractical Tip Bir yorum yazmak üzereyken durun ve sorun: "Bu yorumu koda nasıl dönüştürebilirim?" Çoğu zaman iyi bir metot adı yorumun yerini alır. Before writing a comment, pause and ask: "How can I turn this comment into code?" Often a good method name replaces the comment entirely.

İfade Edici Kod (Expressive Code)Expressive Code

Kavram AçıklamasıConcept Explanation

İfade gücü yüksek kod (Expressive Code), programcının niyetini doğrudan kodun kendi yapısıyla aktarabilmesidir. Değişken isimleri, metot isimleri ve sınıf yapıları o kadar açık olmalıdır ki ekstra açıklamaya gerek kalmasın. Expressive code is code that communicates the programmer's intent directly through its own structure. Variable names, method names, and class structures should be so clear that no extra explanation is needed.

Kötü Örnek (Before)Bad Example (Before)

KÖTÜ — Sihirli sayılar ve belirsiz isimlerBAD — Magic numbers and vague names 
// 86400 = bir günün saniyesi
int d = 86400;
// kullanıcının son 7 günde aktif olup olmadığını kontrol et
if (now - user.lastSeen < d * 7) { ... }

İyi Örnek (After)Good Example (After)

İYİ — Kendini açıklayan sabitler ve metotGOOD — Self-explanatory constants and method 
static final int SECONDS_PER_DAY = 86400;
if (user.wasActiveInLastDays(7)) { ... }
Pratik İpucuPractical Tip Kodunuzu bir meslektaşınıza okutun. Eğer herhangi bir kısmı açıklamak zorunda kalıyorsanız, o kısım yeterince ifade edici değildir — yorumla değil, yeniden adlandırmayla düzeltin. Have a colleague read your code. If you need to explain any part, that part is not expressive enough — fix it by renaming, not by adding a comment.
