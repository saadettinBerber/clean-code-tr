SetupTeardownIncluder (devam) / SetupTeardownIncluder (cont.) SetupTeardownIncluder (cont.)
Kodun devamında, sınıfın asıl iş mantığını içeren özel (private) metotlar yer almaktadır. render(boolean isSuite) metodu, sınıfın ana giriş noktasıdır ve tüm iş akışını üst seviyede koordine eder. Dikkat ederseniz, bu metot yalnızca birkaç satırdan oluşur ve ne yaptığını açıkça ifade eder. The continuation of the code contains the private methods that hold the actual business logic. The render(boolean isSuite) method is the main entry point and coordinates the entire workflow at a high level. Notice that this method consists of only a few lines and clearly expresses what it does.
SetupTeardownIncluder.java (devam) Listing 3-7 (devam) 
private String render(boolean isSuite) throws Exception {
    this.isSuite = isSuite;
    if (isTestPage())
        includeSetupAndTeardownPages();
    return pageData.getHtml();
}

private boolean isTestPage() throws Exception {
    return pageData.hasAttribute("Test");
}

private void includeSetupAndTeardownPages() throws Exception {
    includeSetupPages();
    includePageContent();
    includeTeardownPages();
    updatePageContent();
}

private void includeSetupPages() throws Exception {
    if (isSuite)
        includeSuiteSetupPage();
    includeSetupPage();
}

private void includeSuiteSetupPage() throws Exception {
    include(SuiteResponder.SUITE_SETUP_NAME, "-setup");
}

private void includeSetupPage() throws Exception {
    include("SetUp", "-setup");
}

private void includePageContent() throws Exception {
    newPageContent.append(pageData.getContent());
}

private void includeTeardownPages() throws Exception {
    includeTeardownPage();
    if (isSuite)
        includeSuiteTeardownPage();
}

private void includeTeardownPage() throws Exception {
    include("TearDown", "-teardown");
}

private void includeSuiteTeardownPage() throws Exception {
    include(SuiteResponder.SUITE_TEARDOWN_NAME, "-teardown");
}
Bu metotların her biri yalnızca tek bir iş yapmaktadır. Her metot, adıyla uyumlu tek bir sorumluluğa sahiptir: includeSetupPages() yalnızca kurulum sayfalarını ekler, includeTeardownPages() yalnızca temizlik sayfalarını ekler. Bu, Tek Sorumluluk (Single Responsibility) ilkesinin metot seviyesindeki mükemmel bir uygulamasıdır. Each of these methods does only one thing. Each method has a single responsibility consistent with its name: includeSetupPages() only includes setup pages, includeTeardownPages() only includes teardown pages. This is a perfect application of the Single Responsibility principle at the method level.
Ayrıca Basamak Kuralına (Stepdown Rule) da uyulduğuna dikkat edin: kod yukarıdan aşağıya bir gazete makalesi gibi okunabilir. Üst seviye metotlar önce gelir, ayrıntılar aşağıya doğru açılır. render → includeSetupAndTeardownPages → includeSetupPages → includeSuiteSetupPage şeklinde bir soyutlama hiyerarşisi oluşur. Also notice how the Stepdown Rule is followed: the code reads top-down like a newspaper article. Higher-level methods come first, details unfold downward. A hierarchy of abstraction forms: render → includeSetupAndTeardownPages → includeSetupPages → includeSuiteSetupPage.

Metot Seviyesinde Tek Sorumluluk (Method-Level SRP)Method-Level Single Responsibility

Kavram AçıklamasıConcept Explanation

Tek Sorumluluk İlkesi (SRP) sadece sınıflar için değil, metotlar için de geçerlidir. Her metot yalnızca bir şey yapmalı ve bunu adıyla açıkça ifade etmelidir. Bir metodun ne yaptığını özetlemek için "ve" kullanıyorsanız, o metot muhtemelen bölünmelidir.The Single Responsibility Principle (SRP) applies not only to classes but also to methods. Every method should do only one thing and express it clearly through its name. If you need the word "and" to summarize what a method does, it probably needs to be split.

Kötü Örnek (Before)Bad Example (Before)

KÖTÜ — Birden fazla sorumlulukBAD — Multiple responsibilities 
function processOrder(order) {
    // Doğrulama
    if (!order.items || order.items.length === 0)
        throw new Error("Empty order");
    // Hesaplama
    let total = 0;
    for (const item of order.items)
        total += item.price * item.qty;
    // Veritabanı
    db.save({ ...order, total });
    // E-posta
    sendConfirmation(order.email, total);
}

İyi Örnek (After)Good Example (After)

İYİ — Her metot tek işGOOD — Each method does one thing 
function processOrder(order) {
    validateOrder(order);
    const total = calculateTotal(order.items);
    persistOrder(order, total);
    notifyCustomer(order.email, total);
}

function validateOrder(order) { /* ... */ }
function calculateTotal(items) { /* ... */ }
function persistOrder(order, total) { /* ... */ }
function notifyCustomer(email, total) { /* ... */ }
Pratik İpucuPractical TipBir metotta yorum başlıkları görüyorsanız (// Doğrulama, // Hesaplama, // Kaydet), her bölüm ayrı bir metot olmalıdır. Yorum başlığı, "burada yeni bir sorumluluk başlıyor" demektir.If you see comment headers in a method (// Validate, // Calculate, // Save), each section should be a separate method. A comment header means "a new responsibility starts here."

Basamak Kuralı (Stepdown Rule)Stepdown Rule

Kavram AçıklamasıConcept Explanation

Basamak Kuralı (Stepdown Rule), kodu bir gazete makalesi gibi organize etmeyi önerir: önce üst seviye özet, sonra detaylar. Her fonksiyonun altındaki fonksiyonlar, bir soyutlama seviyesi daha düşük olmalıdır. Okuyucu yukarıdan aşağıya inerken giderek daha fazla detayla karşılaşır.The Stepdown Rule suggests organizing code like a newspaper article: first a high-level summary, then details. Functions below each function should be one level of abstraction lower. As the reader moves top-down, they encounter progressively more detail.

Kötü Örnek (Before)Bad Example (Before)

KÖTÜ — Karışık soyutlama seviyeleriBAD — Mixed abstraction levels 
class ReportGenerator {
    // Düşük seviye detay en üstte
    private String formatCurrency(double v) {
        return String.format("$%.2f", v);
    }
    // Yüksek seviye metot altta
    public Report generate() { ... }
    // Orta seviye arada
    private List fetchData() { ... }
}

İyi Örnek (After)Good Example (After)

İYİ — Yukarıdan aşağıya soyutlamaGOOD — Top-down abstraction 
class ReportGenerator {
    // Seviye 1: En yüksek soyutlama
    public Report generate() {
        List data = fetchData();
        Table table = buildTable(data);
        return formatReport(table);
    }
    // Seviye 2: Orta soyutlama
    private List fetchData() { ... }
    private Table buildTable(...) { ... }
    private Report formatReport(...) { ... }
    // Seviye 3: Düşük seviye detay
    private String formatCurrency(...) { ... }
}
Pratik İpucuPractical TipDosyanızı açtığınızda ilk gördüğünüz metot, "bu sınıf ne yapıyor?" sorusunun cevabı olmalıdır. Detaylar aşağıda olsun — bir gazete makalesinin manşetinden gövdeye inmesi gibi.The first method you see when opening a file should answer "what does this class do?" Details come below — like going from a newspaper headline to the body text.

Oluşturulmuş Metot (Composed Method)Composed Method

Kavram AçıklamasıConcept Explanation

Oluşturulmuş Metot (Composed Method) deseni, bir metodu yalnızca diğer metotlara yapılan çağrılardan oluşturma tekniğidir. includeSetupAndTeardownPages() bunun harika bir örneğidir: kendi içinde herhangi bir düşük seviye iş yapmaz, sadece diğer metotları doğru sırayla çağırır.The Composed Method pattern is a technique of building a method entirely from calls to other methods. includeSetupAndTeardownPages() is a great example: it doesn't do any low-level work itself, it just calls other methods in the right order.

Kötü Örnek (Before)Bad Example (Before)

KÖTÜ — Tek monolitik metotBAD — Single monolithic method 
def deploy(app):
    # 50+ satır: build, test, package, upload,
    # verify, notify... hepsi tek metotta
    subprocess.run(["npm", "run", "build"])
    subprocess.run(["npm", "test"])
    # ... 40 satır daha

İyi Örnek (After)Good Example (After)

İYİ — Oluşturulmuş metotGOOD — Composed method 
def deploy(app):
    build_application(app)
    run_tests(app)
    create_package(app)
    upload_to_server(app)
    verify_deployment(app)
    notify_team(app)
Pratik İpucuPractical TipÜst seviye bir metot "ne yapılacağını" anlatmalı, "nasıl yapılacağını" değil. includeSetupAndTeardownPages okuyucuya tam olarak ne olduğunu söyler — detaylar merak edenler alt metotlara inebilir.A high-level method should tell "what" to do, not "how" to do it. includeSetupAndTeardownPages tells the reader exactly what happens — those curious about details can drill into the sub-methods.
