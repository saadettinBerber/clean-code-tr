Yorumlar Kötü Kodu Telafi Etmez / Comments Do Not Make Up for Bad Code Comments Do Not Make Up for Bad Code
Yorum yazmak için en yaygın motivasyonlardan biri kötü koddur. Bir modül yazar ve onun kafa karıştırıcı ve düzensiz olduğunu biliriz. Bir karmaşa olduğunun farkındayızdır. Bunun üzerine kendi kendimize şunu söyleriz: "Ah, buna bir yorum eklesem iyi olur!" Hayır! Onu temizlesen daha iyi olur! One of the more common motivations for writing comments is bad code. We write a module and we know it is confusing and disorganized. We know it's a mess. So we say to ourselves, "Ooh, I'd better comment that!" No! You'd better clean it!
Az yorumlu, net ve ifade gücü yüksek bir kod; çok yorumlu, karmaşık ve dağınık bir koddan çok daha üstündür. Yarattığın karmaşayı açıklamak için zaman harcamak yerine, o karmaşayı temizlemek için zaman harca. Clear and expressive code with few comments is far superior to cluttered and complex code with lots of comments. Rather than spend your time writing the comments that explain the mess you've made, spend it cleaning that mess.
Kendinizi Kodla Açıklayın / Explain Yourself in Code Explain Yourself in Code
Elbette kodun açıklama için yetersiz kaldığı zamanlar olur. Ne yazık ki pek çok programcı bunu, kodun hiçbir zaman iyi bir açıklama aracı olmadığı anlamında yorumlamıştır. Bu açıkça yanlıştır. Hangisini görmek isterdiniz? Bunu mu: There are certainly times when code makes a poor vehicle for explanation. Unfortunately, many programmers have taken this to mean that code is seldom, if ever, a good means for explanation. This is patently false. Which would you rather see? This:
❌ Yorumla ifade ❌ Expressed with comment 
// Çalışan tam zamanlı yardımlara
// hak kazanıp kazanmadığını kontrol et
if ((employee.flags & HOURLY_FLAG)
    && (employee.age > 65))
✅ Kodla ifade ✅ Expressed with code 
if (employee
    .isEligibleForFullBenefits())
Niyetinizin büyük bölümünü kod aracılığıyla açıklamak yalnızca birkaç saniyelik düşünme gerektirir. Çoğu durumda yapılması gereken şey basitçe şudur: yazmak istediğiniz yorumla aynı şeyi söyleyen bir fonksiyon oluşturmak. It takes only a few seconds of thought to explain most of your intent in code. In many cases it's simply a matter of creating a function that says the same thing as the comment you want to write.
İyi Yorumlar / Good Comments Good Comments
Bazı yorumlar zorunlu ya da faydalıdır. Harcadıkları bit'leri hak ettiklerini düşündüğüm birkaçına bakacağız. Ancak şunu aklınızda tutun: gerçek anlamda iyi olan tek yorum, yazmamak için bir yol bulduğunuz yorumdur. Some comments are necessary or beneficial. We'll look at a few that I consider worthy of the bits they consume. Keep in mind, however, that the only truly good comment is the comment you found a way not to write.
Yasal Yorumlar (Legal Comments) Legal Comments
Zaman zaman kurumsal kodlama standartları, yasal nedenlerle belirli yorumlar yazmamızı zorunlu kılar. Örneğin, telif hakkı ve yazarlık bildirimleri her kaynak dosyasının başına eklenmesi gereken makul ve zorunlu şeylerdir. Sometimes our corporate coding standards force us to write certain comments for legal reasons. For example, copyright and authorship statements are necessary and reasonable things to put into a comment at the start of each source file.
Örneğin FitNesse'de her kaynak dosyasının başına koyduğumuz standart yorum başlığı budur. IDE'mizin bu yorumu otomatik olarak daraltarak dağınıklık yaratmasını engellediğini söylemekten memnuniyet duyuyorum: Here, for example, is the standard comment header that we put at the beginning of every source file in FitNesse. I am happy to say that our IDE hides this comment from acting as clutter by automatically collapsing it.
Yasal yorum örneği Legal comment example 
// Copyright (C) 2003,2004,2005 by Object Mentor, Inc. All rights reserved.
// Released under the terms of the GNU General Public License version 2 or later.
Bu tür yorumlar birer sözleşme ya da hukuki metin olmamalıdır. Mümkün olduğunda, tüm şart ve koşulları yorumun içine yazmak yerine standart bir lisans veya başka bir harici belgeye referans verin. Comments like this should not be contracts or legal tomes. Where possible, refer to a standard license or other external document rather than putting all the terms and conditions into the comment.

Yorum Yerine TemizleClean Instead of Comment

Kavram AçıklamasıConcept Explanation

Kodu yorumla "kurtarmaya" çalışmak en sık yapılan hatalardan biridir. Yorum dağınık kodu temizlemez — sadece üzerine bir örtü çeker. Asıl çözüm temizlemektir. Trying to "rescue" code with a comment is one of the most common mistakes. A comment doesn't clean up messy code — it just puts a cover over it. The real solution is to clean it.

Kötü Örnek (Before)Bad Example (Before)

KÖTÜ — Yorumla kurtarılan karmaşaBAD — Mess saved by comment 
// Kullanıcı rolü 1 (admin) veya 2 (editor) ise
// ve durum 'aktif' ise panele erişim ver
if((u.r==1||u.r==2)&&u.s.equals("aktif")) grantAccess();

İyi Örnek (After)Good Example (After)

İYİ — Yorum kalktı, kod konuşuyorGOOD — No comment, code speaks 
if (user.hasEditorialAccess() && user.isActive()) {
    grantAccess();
}
Pratik İpucuPractical Tip Bir yorum yazmak istediğinizde kendinize şunu sorun: "Bu yorumun amacını bir metot ya da değişken adıyla ifade edebilir miyim?" Evet ise — refactor edin, yorum yazmayın. When you want to write a comment, ask yourself: "Can I express the purpose of this comment with a method or variable name?" If yes — refactor, don't comment.

Kendinizi Kodla Açıklayın (Explain Yourself in Code)Explain Yourself in Code

Kavram AçıklamasıConcept Explanation

Niyeti açıklamak için yorum satırı eklemek yerine, o niyeti doğrudan koda yansıtın. Bir koşulu açıklayan yorum, genellikle o koşulu sayan bir boolean metoduna dönüştürülebilir. Instead of adding a comment to explain intent, reflect that intent directly in the code. A comment explaining a condition can often be turned into a boolean method that expresses that condition.

Kötü Örnek (Before)Bad Example (Before)

KÖTÜ — Yorum açıklıyorBAD — Comment explains 
// Sipariş ücretsiz kargo hakkına sahipse
if (order.total >= 150 && order.isPremiumMember
    && !order.hasPhysicalItems()) {
    addFreeShipping(order);
}

İyi Örnek (After)Good Example (After)

İYİ — Kod açıklıyorGOOD — Code explains 
if (order.isEligibleForFreeShipping()) {
    addFreeShipping(order);
}
Pratik İpucuPractical Tip "Bu satırı anlamak için yoruma gerek var mı?" sorusunu sık sık sorun. Cevap evetse, yorum eklemek yerine kodu daha açık hale getirin. Frequently ask "Do I need a comment to understand this line?" If yes, make the code clearer instead of adding a comment.

Yasal Yorumlar (Legal Comments)Legal Comments

Kavram AçıklamasıConcept Explanation

Telif hakkı ve lisans bildirimleri, yazılım projelerinde yasal zorunluluk nedeniyle eklenmesi gereken yorumlardır. Bu tür yorumlar "iyi yorum" kategorisine girer; ancak mümkün olduğunca kısa tutulmalı ve detaylar harici belgeye bırakılmalıdır. Copyright and license notices are comments that must be added to software projects due to legal requirements. These types of comments fall into the "good comment" category; however, they should be kept as short as possible and details should be left to an external document.

Kötü Örnek (Before)Bad Example (Before)

KÖTÜ — Tüm lisans metni yoruma gömülmüşBAD — Full license text embedded in comment 
// BU YAZILIM ŞU KOŞULLAR ALTINDA... (50 satır devam eder)
// 1. Yeniden dağıtım...
// 2. İkili formda...
// 3. Ne ABD hükümeti...
// ...

İyi Örnek (After)Good Example (After)

İYİ — Kısa yorum, detay için referansGOOD — Short comment, reference for details 
// Copyright (C) 2024 Acme Corp. All rights reserved.
// Licensed under the MIT License. See LICENSE file for details.
Pratik İpucuPractical Tip Projenizdeki tüm yasal bildirimleri bir LICENSE dosyasında tutun ve kaynak dosyalarında sadece kısa bir referans satırı bırakın. Bu hem okunabilirliği artırır hem de güncellemeleri kolaylaştırır. Keep all legal notices in a LICENSE file in your project and leave only a short reference line in source files. This both improves readability and makes updates easier.
