Gürültü Yorumu / Noise Comments Noise Comments
Bazen yorumlar hiçbir şey söylememektedir. Yalnızca gürültüdürler. Aşağıdaki yorumlar o kadar aşikar ki hiç sorgulanmadan geçilirler. Ve geçilmeleri gerekir: Sometimes you see comments that are nothing but noise. They restate the obvious and provide no new information. These comments are so noisy that we learn to ignore them. The further we ignore them, the more likely we are to miss real information among them.
Gürültü yorum örnekleri Noise comment examples Java 
/**
 * Varsayılan kurucu
 */
protected AnnualDateRule() {
}

/** Gün */
private int dayOfMonth;

/**
 * Gün içi ay değeri döndürür.
 * @return gün içi ay değeri
 */
public int getDayOfMonth() {
  return dayOfMonth;
}
Bu tür yorumlar o kadar anlamsız ki programcılar onları okumayı öğrenir — tıpkı görmezden geldikleri reklamlar gibi. Sonunda gerçek yorumların da aynı şekilde görmezden gelinmesine yol açarlar. Aşağıdakine ne diyorsunuz: These comments are so noisy that programmers learn to ignore them. As they learn to ignore irrelevant comments, they eventually ignore all comments—and there is real damage done. The following comment is so redundant it borders on absurdity:
En absürd gürültü yorumu Most absurd noise comment Java 
private void startSending()
{
  try
  {
    doSending();
  }
  catch(SocketException e)
  {
    // Gönderme iptal edildi. Bu normal.
  }
  catch(Exception e)
  {
    try
    {
      response.add(ErrorResponder.makeExceptionString(e));
      response.closeAll();
    }
    catch(Exception e1)
    {
      //Artık ne yapabiliriz ki?
    }
  }
}
İkinci catch bloğundaki yorum muhtemelen bir sinirlenme anında yazılmıştır. Programcı gerçekten o catch bloğunu boş bırakmaktan rahatsız olmuş; ama bir şeyler yazmak istememiş. Boş bir catch bloğunu anlamlı şekilde adlandırılmış bir metoda dönüştürmek yerine yorum bırakmıştır. The second catch block was authored in a moment of frustration. The programmer was apparently displeased about leaving the catch block empty but didn't want to take the time to create an appropriate exception class. So he vented his frustration in a comment. Better to have replaced the catch block with a named function or handled the exception appropriately.
Konum İşaretçisi / Position Markers Position Markers
Bazen programcılar bir kaynak dosyada belirli bir konumu işaretlemek ister. Şunun gibi: Sometimes programmers like to mark a particular position in a source file. For example, I recently found this in a program I was browsing:
Konum işaretçisi örneği Position marker example 
// Actions //////////////////////////////////
Bazen belirli bir fonksiyonlar topluluğunu bu tür bir banner altında gruplamak anlamlı olabilir. Ama genellikle bunlar gereksiz gürültüdür ve görmezden gelinmelidir. Bunları nadiren ve yalnızca faydası açıkça görülen yerlerde kullanın. Bu tür banner'ları sık kullandığınızda arka plan gürültüsü haline gelirler ve görmezden gelinirler. There are rare times when it makes sense to gather certain functions together beneath a banner like this. But in general they are clutter that should be eliminated—especially the noisy train of slashes at the end. Think of it this way. A banner is startling and obvious if you don't see banners very often. So use them very sparingly, and only when the benefit is significant. If you overuse banners, they'll fall into the background noise and be ignored.
Kapatma Parantezi Yorumu / Closing Brace Comments Closing Brace Comments
Bazen programcılar kapanan parantezlerin yanına özel yorumlar ekler. Bu, uzun ve derin iç içe geçmiş yapılarda anlam taşıyabilir; ancak yalnızca küçük ve kapsüllü fonksiyonlar yazdığımız için tercih ettiğimiz durumlarda bu yorumlar yalnızca dağınıklıktır. Kapatma parantezi yorumlarına ihtiyaç duyuyorsanız, bunu fonksiyonlarınızı kısaltmak için bir sinyal olarak değerlendirin: Sometimes programmers will put special comments on closing braces. Although this might make sense for long functions with deeply nested structures, it serves only to clutter the kind of small and encapsulated functions that we prefer. So if you find yourself wanting to mark your closing braces, try to shorten your functions instead.
Kapatma parantezi yorumları — kötü örnek Closing brace comments — bad example Java 
public class wc {
  public static void main(String[] args) {
    BufferedReader in = new BufferedReader(new InputStreamReader(System.in));
    String line;
    int lineCount = 0;
    int charCount = 0;
    int wordCount = 0;
    try {
      while ((line = in.readLine()) != null) {
        lineCount++;
        charCount += line.length();
        String words[] = line.split("\\W");
        wordCount += words.length;
      } //while
      System.out.println("wordCount = " + wordCount);
      System.out.println("lineCount = " + lineCount);
      System.out.println("charCount = " + charCount);
    } // try
    catch (IOException e) {
      System.err.println("Error:" + e.getMessage());
    } //catch
  } //main
}

Gürültü Yorumu (Noise Comment)Noise Comment

Kavram AçıklamasıConcept Explanation

Gürültü yorumları apaçık olanı tekrarlar ve okuyucuya hiçbir yeni bilgi vermez. Zamanla görmezden gelinen bu yorumlar, aralarına sıkışmış gerçek ve önemli yorumların da görmezden gelmesine yol açar. Noise comments repeat the obvious and provide no new information to the reader. Over time these ignored comments lead to real and important comments interspersed among them also being ignored.

Kötü Örnek (Before)Bad Example (Before)

KÖTÜ — Her satır yorumlanmış, hiçbiri değer katmıyorBAD — Every line commented, none adding value 
/** Kullanıcı adı */
String username;

/** Yaş */
int age;

/** Kullanıcı adını döndürür */
String getUsername() { return username; }

İyi Örnek (After)Good Example (After)

İYİ — Yorum yok, isimler açıkGOOD — No comments, names are clear 
String username;
int age;
String getUsername() { return username; }
Pratik İpucuPractical Tip Bir yorum yazmadan önce kendinize "Bu yorum olmasaydı kodu anlamak güçleşir miydi?" sorusunu sorun. Hayırsa — silmeyin, yazmayın. Before writing a comment, ask yourself "Would the code be harder to understand without this comment?" If no — don't write it.

Konum İşaretçisi (Position Marker)Position Marker

Kavram AçıklamasıConcept Explanation

Dosyada bölüm ayıracı olarak kullanılan // ====== benzeri satırlar başlangıçta dikkat çekiciyken, alışıla gelince arka plan gürültüsüne dönüşürler. İyi kod organizasyonu ve küçük fonksiyonlar bu işaretçilere olan ihtiyacı ortadan kaldırır. Lines like // ====== used as section dividers in files are eye-catching at first but turn into background noise once they become commonplace. Good code organization and small functions eliminate the need for these markers.

Kötü Örnek (Before)Bad Example (Before)

KÖTÜ — Orman gibi bir dosyada işaretçilerBAD — Markers in a forest-like file 
// ==================== INIT ====================
public void init() { ... }

// =================== ACTIONS ==================
public void processOrder() { ... }
public void cancelOrder() { ... }

// =================== HELPERS ==================
private void validate() { ... }

İyi Örnek (After)Good Example (After)

İYİ — Ayrı küçük sınıflar, işaretçi gereksizGOOD — Separate small classes, no markers needed 
class OrderInitializer { public void init() { ... } }
class OrderProcessor { public void process() { ... } }
class OrderValidator { public void validate() { ... } }
Pratik İpucuPractical Tip Dosyanız konum işaretçisi olmadan okunamıyorsa, bu dosyanın çok büyük ve tek sorumluluk ilkesini ihlal ettiğinin işaretidir. If your file can't be read without position markers, it's a sign that the file is too large and violating the single responsibility principle.

Kapatma Parantezi Yorumu (Closing Brace Comment)Closing Brace Comment

Kavram AçıklamasıConcept Explanation

Kapanan süslü parantezlerin yanında // while veya // try gibi yorumlar yazmak, fonksiyonun çok uzun ve karmaşık olduğunun belirtisidir. Çözüm yorum değil, fonksiyonu küçültmektir. Writing comments like // while or // try next to closing braces is a sign that the function is too long and complex. The solution is not a comment, but shortening the function.

Kötü Örnek (Before)Bad Example (Before)

KÖTÜ — Kapatma parantezi yorumlarıBAD — Closing brace comments while (hasNext()) { if (isValid()) { process(); } // if isValid } // while hasNext

İyi Örnek (After)Good Example (After)

İYİ — Küçük fonksiyon, yorum gereksizGOOD — Small function, no comment needed while (hasNext()) { processNextValidItem(); } private void processNextValidItem() { if (isValid()) process(); }
Pratik İpucuPractical Tip Kapatma parantezine yorum ekleme ihtiyacı hissediyorsanız, durdurun ve o döngünün gövdesini ayrı bir metoda çıkarın. İsimleyin ve yorumdan daha iyi bir belgeleme sağlayın. If you feel the need to add a comment to a closing brace, stop and extract the loop body into a separate method. Name it and provide better documentation than a comment.
