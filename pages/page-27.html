Eğer "her kavram için tek kelime" kuralına uyarsanız, örneğin add metoduna sahip birçok sınıfınız olabilir. Farklı add metotlarının parametre listeleri ve dönüş değerleri anlamsal olarak eşdeğer olduğu sürece her şey yolundadır. If you follow the “one word per concept” rule, you could end up with many classes that have, for example, an add method. As long as the parameter lists and return values of the various add methods are semantically equivalent, all is well.
Ancak birisi, aslında aynı anlamda ekleme yapmadığı halde "tutarlılık" adına add kelimesini kullanmaya karar verebilir. Diyelim ki mevcut iki değeri toplayarak veya birbirine bağlayarak yeni bir değer oluşturan birçok sınıfımızda add metodu var. Şimdi tek bir parametreyi bir koleksiyona (collection) yerleştiren bir metoda sahip yeni bir sınıf yazdığımızı varsayalım. Bu metoda add demeli miyiz? Diğer pek çok add metodumuz olduğu için tutarlı görünebilir, ancak bu durumda semantik (anlamsal yapı) farklıdır; bu yüzden insert veya append gibi bir isim kullanmalıyız. Yeni metoda add demek bir kelime oyunu (pun) olurdu. However one might decide to use the word add for “consistency” when he or she is not in fact adding in the same sense. Let’s say we have many classes where add will create a new value by adding or concatenating two existing values. Now let’s say we are writing a new class that has a method that puts its single parameter into a collection. Should we call this method add? It might seem consistent because we have so many other add methods, but in this case, the semantics are different, so we should use a name like insert or append instead. To call the new method add would be a pun.
Yazarlar olarak amacımız, kodumuzu anlaşılması mümkün olduğunca kolay hale getirmektir. Kodumuzun yoğun bir çalışma değil, hızlı bir göz gezdirme (skim) olmasını isteriz. Yazarın kendini açıkça ifade etmekten sorumlu olduğu popüler karton kapaklı kitap modelini (popular paperback model) kullanmak isteriz; okuyucunun anlamı kağıttan kazıyıp çıkarmak zorunda olduğu akademik modeli değil. Our goal, as authors, is to make our code as easy as possible to understand. We want our code to be a quick skim, not an intense study. We want to use the popular paperback model whereby the author is responsible for making himself clear and not the academic model where it is the scholar’s job to dig the meaning out of the paper.
Çözüm Alanı İsimlerini Kullanma / Use Solution Domain Names Use Solution Domain Names
Kodunuzu okuyacak kişilerin programcılar olacağını unutmayın. Bu yüzden bilgisayar bilimleri (CS) terimlerini, algoritma isimlerini, desen (pattern) isimlerini, matematik terimlerini ve benzerlerini kullanmaktan çekinmeyin. Her ismi problem alanından (problem domain) çekip çıkarmak akıllıca değildir; çünkü iş arkadaşlarımızın zaten farklı bir isimle bildikleri bir kavramın ne anlama geldiğini sormak için sürekli müşteriye koşmalarını istemeyiz. Remember that the people who read your code will be programmers. So go ahead and use computer science (CS) terms, algorithm names, pattern names, math terms, and so forth. It is not wise to draw every name from the problem domain because we don’t want our coworkers to have to run back and forth to the customer asking what every name means when they already know the concept by a different name.
AccountVisitor ismi, ZİYARETÇİ desenine (VISITOR pattern) aşina olan bir programcı için çok şey ifade eder. Hangi programcı bir İş Kuyruğu'nun (JobQueue) ne olduğunu bilmez? Programcıların yapmak zorunda olduğu pek çok teknik şey vardır. Bunlar için teknik isimler seçmek genellikle en uygun yoldur. The name AccountVisitor means a great deal to a programmer who is familiar with the VISITOR pattern. What programmer would not know what a JobQueue was? There are lots of very technical things that programmers have to do. Choosing technical names for those things is usually the most appropriate course.
Problem Alanı İsimlerini Kullanma / Use Problem Domain Names Use Problem Domain Names
Yaptığınız iş için bir "programcı dili" (programmer-eese) yoksa, problem alanından (problem domain) bir isim kullanın. En azından kodu koruyan programcı, bir alan uzmanına (domain expert) bunun ne anlama geldiğini sorabilir. When there is no “programmer-eese” for what you’re doing, use the name from the problem domain. At least the programmer who maintains your code can ask a domain expert what it means.
Çözüm ve problem alanı kavramlarını birbirinden ayırmak, iyi bir programcı ve tasarımcının görevinin bir parçasıdır. Problem alanı kavramlarıyla daha çok ilgili olan kodlar, problem alanından seçilmiş isimlere sahip olmalıdır. Separating solution and problem domain concepts is part of the job of a good programmer and designer. The code that has more to do with problem domain concepts should have names drawn from the problem domain.
Anlamlı Bağlam Ekleme / Add Meaningful Context Add Meaningful Context
Kendi başlarına anlamlı olan çok az isim vardır; çoğu değildir. Bunun yerine, isimleri iyi adlandırılmış sınıflar, fonksiyonlar veya ad alanları (namespaces) içine yerleştirerek okuyucu için bir bağlam (context) oluşturmanız gerekir. Her şey başarısız olduğunda, ismin önüne bir önek (prefix) eklemek son çare olarak gerekli olabilir. There are a few names which are meaningful in and of themselves—most are not. Instead, you need to place names in context for your reader by enclosing them in well-named classes, functions, or namespaces. When all else fails, then prefixing the name may be necessary as a last resort.

Kelime Oyunlarından Kaçınma (Avoid Punning)Avoid Punning

Kavram AçıklamasıConcept Explanation

Aynı kelimeyi iki farklı amaç için kullanmak bir kelime oyunudur. "Tutarlılık" adına farklı anlamlara gelen eylemler için aynı kelimeyi seçmek okuyucuyu yanıltır. Örneğin, iki sayıyı toplamak ile bir öğeyi listeye eklemek farklı anlamsal eylemlerdir.Using the same word for two different purposes is punning. Choosing the same word for actions with different meanings for the sake of "consistency" misleads the reader. For example, adding two numbers and inserting an item into a list are different semantic actions.

Kötü Örnek (Before)Bad Example (Before)

KÖTÜ — "add" kelimesinin aşırı yüklenmesiBAD — Overloading the word "add" 
public class Calculator {
    public int add(int a, int b) { return a + b; }
}

public class UserList {
    // Koleksiyona ekleme yaparken de 'add' kullanılmış
    public void add(User user) { users.add(user); }
}
Burada UserList.add metodu matematiksel bir toplama yapmıyor, sadece koleksiyona bir öğe yerleştiriyor. Diğer sınıflardaki add metotlarıyla "benzer" görünse de işlevi farklıdır.Here, UserList.add doesn't perform a mathematical addition; it just inserts an item into a collection. Even if it looks "consistent" with other add methods, its function is different.

İyi Örnek (After)Good Example (After)

İYİ — Anlamsal olarak doğru terimlerGOOD — Semantically correct terms 
public class Calculator {
    public int add(int a, int b) { return a + b; }
}

public class UserList {
    public void insert(User user) { users.add(user); }
    // veya
    public void append(User user) { users.add(user); }
}
Pratik İpucuPractical TipKodunuzu bir "paperback" (karton kapaklı popüler roman) gibi yazın. Okuyucunun anlamı çözmek için dedektiflik yapmasına gerek kalmamalı. Kelimelerin yaygın teknik anlamlarına sadık kalın.Write your code like a "paperback" novel. The reader shouldn't have to play detective to figure out the meaning. Stick to common technical meanings of words.

Çözüm Alanı İsimleri (Solution Domain Names)Solution Domain Names

Kavram AçıklamasıConcept Explanation

Kodunuzu programcılar okur. Bu yüzden teknik terimleri (algoritmalar, desenler, veri yapıları) kullanmaktan çekinmeyin. Bir programcıya "Müşteri Hesabı Veri Erişim Nesnesi" demek yerine "AccountDAO" demek çok daha hızlı ve net bir iletişim sağlar.Programmers read your code. So don't hesitate to use technical terms (algorithms, patterns, data structures). Saying "AccountDAO" to a programmer is much faster and clearer than saying "Customer Account Data Access Object."

İyi ÖrneklerGood Examples

İYİ — Teknik terimlerin kullanımıGOOD — Use of technical terms 
// Programcılar bu isimlerin ne anlama geldiğini hemen anlar:
class AccountObserver { ... }
class EmailPriorityQueue { ... }
public void sortWithQuickSort() { ... }
public UserFactory userFactory;
Pratik İpucuPractical TipEğer bir tasarım deseni (design pattern) kullanıyorsanız, o desenin adını sınıf ismine dahil edin. Örneğin, Account yerine AccountProxy veya AccountDecorator kullanmak, kodun mimari rolünü anında açıklar.If you are using a design pattern, include its name in the class name. For example, using AccountProxy or AccountDecorator instead of just Account immediately clarifies the code's architectural role.

Problem Alanı İsimleri (Problem Domain Names)Problem Domain Names

Kavram AçıklamasıConcept Explanation

Yaptığınız işin teknik bir karşılığı yoksa, o işin yapıldığı alanın (finans, sağlık, oyun vb.) terminolojisini kullanın. Bu, teknik olmayan ekip üyeleri ve alan uzmanlarıyla aynı dili konuşmanızı sağlar ve bakım aşamasında anlam karmaşasını önler.If there is no technical equivalent for what you're doing, use the terminology of the domain (finance, health, gaming, etc.) where the work is done. This allows you to speak the same language as non-technical team members and domain experts, preventing confusion during maintenance.

ÖrnekExample

Bir bankacılık uygulamasında teknik bir kuyruk yapısı kurmuyorsanız, "bekleyen işlemler" için PendingTransactions demek, OperationBuffer demekten daha anlamlıdır.In a banking application, unless you are building a technical queue structure, saying PendingTransactions for "waiting operations" is more meaningful than saying OperationBuffer.
// Problem alanından isimler:
class InsurancePolicy { ... }
class ClaimAdjustment { ... }
public void applyDiscount(Voucher voucher) { ... }
Pratik İpucuPractical Tipİyi bir tasarımcı, çözüm alanı (teknik) ve problem alanı (iş) kavramlarını nerede ayıracağını bilir. Kullanıcı arayüzüne ve iş kurallarına yakın kodlar problem alanı isimlerini kullanmalıdır.A good designer knows where to separate solution domain (technical) and problem domain (business) concepts. Code close to the UI and business rules should use problem domain names.

Anlamlı Bağlam (Meaningful Context)Meaningful Context

Kavram AçıklamasıConcept Explanation

Değişken isimleri tek başlarına yetersiz kalabilir. Örneğin state değişkeni bir adresin eyaletini mi, yoksa bir nesnenin durumunu mu temsil ediyor? İsmi bir sınıfın veya fonksiyonun içine yerleştirerek ona bir bağlam kazandırın.Variable names alone can be insufficient. For example, does the state variable represent an address's state or an object's status? Give it context by placing the name inside a class or function.

Kötü Örnek (Before)Bad Example (Before)

KÖTÜ — Bağlamdan kopuk değişkenlerBAD — Variables disconnected from context 
public void process(String firstName, String lastName, String state) {
    // 'state' burada ne anlama geliyor? 
    // Sipariş durumu mu? Eyalet mi?
}

İyi Örnek (After)Good Example (After)

İYİ — Sınıf ile sağlanan bağlamGOOD — Context provided by class 
public class Address {
    private String firstName;
    private String lastName;
    private String state; // Artık eyalet olduğu açık.
}

public void process(Address address) { ... }
Pratik İpucuPractical TipEğer değişkenlerin önüne sürekli aynı öneki (örn: addrState, addrZip) ekleme ihtiyacı duyuyorsanız, bu değişkenleri kendi sınıflarına taşımanın zamanı gelmiş demektir.If you feel the need to constantly add the same prefix to variables (e.g., addrState, addrZip), it's time to move these variables into their own class.
