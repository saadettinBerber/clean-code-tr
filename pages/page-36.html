Yukarıdan Aşağıya Kod Okumak: Basamak Kuralı / Reading Code from Top to Bottom: The Stepdown Rule Reading Code from Top to Bottom: The Stepdown Rule
34. sayfadaki getHtml fonksiyonunu tekrar inceleyin. Dikkat edin, soyutlama seviyesi bir kerede iki seviye düşüyor. Bu, okurken zihinsel bir sıçrama gerektirir. Bunu okurken önce bir seviye aşağı, sonra bir düşük seviye detay gördükten sonra tekrar yukarı çıkmamız gerekir. Bu zihinsel sıçrama zorlayıcıdır. Look again at the getHtml function on page 34. Notice that the abstraction level drops by two levels at once. This requires a mental leap when reading it. As we read it, we have to drop down one level, then after seeing some low-level detail, jump back up again. This mental jumping is hard.
Kodumuzun üstten alta doğru bir anlatı gibi okunmasını istiyoruz [KP]. Her fonksiyonun, bir sonraki soyutlama seviyesindeki fonksiyonları takip etmesini istiyoruz, böylece programı okurken bir fonksiyon seviyesinden diğerine düşeriz. Bunu Basamak Kuralı (Stepdown Rule) olarak adlandırıyoruz. We want the code to read like a top-down narrative [KP]. We want every function to be followed by those at the next level of abstraction so that we can read the program, dropping one level of abstraction at a time as we read down the list of functions. I call this The Stepdown Rule.
Başka bir deyişle, aynı seviyede bir sonraki adımı gerçekleştirmek için ihtiyacımız olan detayları anlamak istiyoruz. Bu, bir makaleyi okumak gibidir. Gazeteyi alırsınız ve bir makaleyi okumaya başlarsınız. İlk cümle genel konunun bir özeti verir, bu sizi bir sonraki paragrafa götürür ve bu paragraf biraz daha fazla detay verir. Okudukça detaylar açıklanır. To say this differently, we want to be able to read the program as though it were a set of TO paragraphs, each of which is describing the current level of abstraction and referencing subsequent TO paragraphs at the next level down. It turns out to be very difficult for humans to hold more than about three levels of nesting in working memory. Thus, each function should lead into the next by doing one thing at the current level of abstraction.
Bu, fonksiyonlarımızı oluştururken, bir sonraki soyutlama seviyesindeki fonksiyonları çağırdığımızdan emin olmak anlamına gelir. Böylece kodu okurken, bir fonksiyonu okurken gözümüzü bir sonraki fonksiyona kaydırırız ve bu fonksiyon bir sonraki seviyedir. Kodumuzu bu şekilde yazarsak, Basamak Kuralı'nı takip ederiz. This means that as we compose our functions, the functions that are called should be at the next level of abstraction. Thus, as we read down the list of functions in a module, we see the abstraction level decreasing step by step. We want to avoid the jarring alternation of high and low levels that makes the code hard to read.
Tabii ki bu kuralı takip etmek zordur. İnsanlar bir seferde yaklaşık üç seviyelik iç içe geçmeyi çalışma belleğinde (working memory) tutmakta zorlanırlar. Bu nedenle, her fonksiyon mevcut soyutlama seviyesinde tek bir şey yaparak bir sonrakine yol göstermelidir. Of course this is hard. People can typically hold only about three levels of nesting in working memory. Therefore, each function should lead into the next by doing one thing at the current level of abstraction.
Switch İfadeleri / Switch Statements Switch Statements
switch ifadeleriyle ilgili sorun, yapılacak N şeyin olduğunu belirtmek zorunda olmalarıdır. Ne yazık ki, N şey yapmak için kullanılan switch ifadelerini tamamen önleyemeyiz, ancak her switch ifadesinin düşük seviyeli bir sınıfta görünmesini ve asla tekrarlanmamasını sağlayabiliriz. Bunu bir çok biçimli (polymorphic) nesne oluşturmak için kullanırız. It's hard to make a small switch statement. Even a switch statement with only two cases is larger than I'd like a single block or function to be. It's also hard to make a switch statement that does one thing. By their nature, switch statements always do N things. Unfortunately we can't always avoid switch statements, but we can make sure that each switch statement is buried in a low-level class and is never repeated. We do this, of course, with polymorphism.
Liste 3-4'te gösterildiği gibi, switch ifadesi bir abstract factory'nin (soyut fabrika) bodrum katına gömülebilir ve hiçbir zaman görünmez. Factory, switch ifadesini kullanarak uygun somut türevleri (concrete derivatives) Employee arayüzünün (interface) arkasına yerleştirir. Consider Listing 3-4. It shows a switch statement that can be buried in the basement of an abstract factory and never seen. The factory uses the switch statement to create appropriate instances of the derivatives of Employee based on the EmployeeType code.

Basamak Kuralı (Stepdown Rule)Stepdown Rule

Kavram AçıklamasıConcept Explanation

Kod, yüksek seviyeden düşük seviyeye doğru "aşağı doğru" okunmalıdır. Her fonksiyon, bir sonraki soyutlama seviyesindeki fonksiyonları çağırmalıdır. Bu, gazete makalesi okuma düzenine benzer - önce özet, sonra detaylar.Code should be read "top down" from high level to low level. Each function should call functions at the next level of abstraction. This is similar to reading a newspaper article - first the summary, then the details.

Kötü Örnek (Before)Bad Example (Before)

KÖTÜ — Seviyeler karışıkBAD — Levels mixed 
def processOrder(order):
    # Yüksek seviye
    validateOrder(order)
    
    # Düşük seviye - KARMAŞIK!
    for item in order.items:
        db.execute("INSERT...", item.id)
    
    # Tekrar yüksek seviye
    sendConfirmation(order)

İyi Örnek (After)Good Example (After)

İYİ — Aşamalı seviye düşüşüGOOD — Step-by-step level drop 
def processOrder(order):  # Seviye 1: İş akışı
    validateOrder(order)
    saveToDatabase(order)
    sendConfirmation(order)

def saveToDatabase(order):  # Seviye 2: Veritabanı
    for item in order.items:
        insertItem(item)

def insertItem(item):  # Seviye 3: SQL
    db.execute("INSERT...", item.id)
Pratik İpucuPractical TipKodunuzu bir gazete makalesi gibi organize edin: en üstte özet (yüksek seviye fonksiyon), aşağıya doğru detaylar (yardımcı fonksiyonlar).Organize your code like a newspaper article: summary at the top (high-level function), details below (helper functions).

Switch ve Çok Biçimlilik (Switch and Polymorphism)Switch and Polymorphism

Kavram AçıklamasıConcept Explanation

Switch ifadeleri N şey yapar. Ancak abstract factory'nin bodrum katına gömülerek gizlenebilir. Factory, çok biçimlilik (polymorphism) kullanarak switch mantığını soyutlar ve her tür için uygun somut sınıfı oluşturur.Switch statements do N things. But they can be buried in the basement of an abstract factory. The factory uses polymorphism to abstract away the switch logic and creates appropriate concrete classes for each type.

Kötü Örnek (Before)Bad Example (Before)

KÖTÜ — Her yerde switchBAD — Switch everywhere 
def calculatePay(employee):
    switch employee.type:
        case HOURLY:
            return employee.hours * employee.rate
        case SALARIED:
            return employee.salary / 12
        case COMMISSION:
            return employee.base + employee.sales * 0.1

def calculateBonus(employee):
    switch employee.type:  # Tekrar!
        case HOURLY: ...

İyi Örnek (After)Good Example (After)

İYİ — Factory + PolymorphismGOOD — Factory + Polymorphism 
abstract class Employee:
    abstract def calculatePay()
    abstract def calculateBonus()

class HourlyEmployee(Employee):
    def calculatePay(self): ...

# Factory - switch sadece burada
def createEmployee(type):
    switch type:
        case HOURLY: return new HourlyEmployee()
Pratik İpucuPractical TipSwitch ifadesini abstract factory'de gizleyin. İş mantığını her tür için ayrı sınıflara dağıtın. "Open/Closed Principle" - yeni tür eklemek için sadece yeni sınıf ekleyin, mevcut kodu değiştirmeyin.Hide the switch statement in an abstract factory. Distribute business logic into separate classes for each type. "Open/Closed Principle" - add new types by adding new classes, don't change existing code.

Kırık Pencereler (Broken Windows)Broken Windows

Kavram AçıklamasıConcept Explanation

"Kırık Pencereler" teorisi, Pragmatik Programcılar kitabından gelir: bir binada bir pencere kırık bırakılırsa, diğer pencereler de kırılır. Kötü kod da öyledir - bir kez başladı mı, daha fazla kötü kod çeker. Her zaman temiz tutun!The "Broken Windows" theory comes from The Pragmatic Programmer: if one window in a building is left broken, others will be broken too. Bad code is the same - once it starts, it attracts more bad code. Always keep it clean!

Kötü Örnek (Before)Bad Example (Before)

KÖTÜ — "Sonra düzeltirim"BAD — "I'll fix it later" 
def processData(data):
    # FIXME: Geçici çözüm, sonra düzelt
    if data is None:
        data = {}  # Kırık pencere!
    
    # TODO: Refactor needed
    result = []
    for d in data:  # Karmaşık iç içe döngüler
        for x in d:
            for y in x:
                result.append(y)
    return result

İyi Örnek (After)Good Example (After)

İYİ — Hemen düzeltGOOD — Fix immediately 
def processData(data):
    if not data:
        raise ValueError("Data required")
    
    return [extractValues(item) for item in data]

def extractValues(item):
    # Her seviye için ayrı fonksiyon
    return [y for x in item for y in x]
Pratik İpucuPractical Tip"İzci Kuralı"nı (Boy Scout Rule) uygulayın: kodu bulduğunuzdan daha temiz bırakın. Küçük düzeltmeler bile birikerek büyük fark yaratır.Apply the "Boy Scout Rule": leave the code cleaner than you found it. Even small fixes accumulate to make a big difference.
