Çok Fazla Bilgi / Too Much Information Too Much Information
İlginç tarihsel tartışmalar veya alakasız ayrıntılar yorumlarınıza eklemeyin. Aşağıdaki yorum, base64 kodlama ve kod çözme işlevini test etmek için tasarlanmış bir modülden alınmıştır. RFC numarası dışında bu kodu okuyan birinin yorumda yer alan anlaşılmaz bilgilere hiçbir ihtiyacı yoktur: Don't put interesting historical discussions or irrelevant descriptions of details into your comments. The comment below was extracted from a module designed to test that a function could encode and decode base64. Other than the RFC number, someone reading this code has no need for the arcane information contained in the comment.
Çok fazla bilgi içeren yorumToo much information in comment
/*
  RFC 2045 - Multipurpose Internet Mail Extensions (MIME)
  Part One: Format of Internet Message Bodies
  section 6.8.  Base64 Content-Transfer-Encoding
  The encoding process represents 24-bit groups of input bits as output
  strings of 4 encoded characters. Proceeding from left to right, a
  24-bit input group is formed by concatenating 3 8-bit input groups.
  These 24 bits are then treated as 4 concatenated 6-bit groups, each
  of which is translated into a single digit in the base64 alphabet.
  When encoding a bit stream via the base64 encoding, the bit stream
  must be presumed to be ordered with the most-significant-bit first.
  ...
*/
Bağlantısız Yorum / Inobvious Connection Inobvious Connection
Bir yorum ile açıkladığı kod arasındaki bağlantı açık olmalıdır. Bir yorum yazma zahmetine katlanıyorsanız, en azından okuyucunun hem yoruma hem de koda bakıp yorumun ne hakkında olduğunu anlayabilmesini istersiniz. Apache commons'tan şu örneği ele alalım: The connection between a comment and the code it describes should be obvious. If you are going to the trouble to write a comment, then at least you'd like the reader to be able to look at the comment and the code and understand what the comment is talking about. Consider, for example, this comment drawn from apache commons:
Bağlantısız yorum — "filtre byte" nedir?Inobvious connection — what is a "filter byte"?
         /*
          * tüm pikselleri (ve filtre byte'larını) tutabilecek kadar büyük
          * bir array'la başlayın ve başlık bilgisi için 200 byte fazladan ekleyin
          */
         this.pngBytes = new byte[(((this.width + 1) * this.height * 3) + 200)];
Filtre byte nedir? +1 ile mi ilgili? *3 ile mi? İkisiyle birden mi? Bir piksel bir byte mıdır? Neden 200? Bir yorumun amacı, kendini açıklamayan kodu açıklamaktır. Bir yorumun kendi açıklamasına ihtiyaç duyması ne yazık ki gayet piyanktır. What is a filter byte? Does it relate to the +1? Or to the *3? Both? Is a pixel a byte? Why 200? The purpose of a comment is to explain code that does not explain itself. It is a pity when a comment needs its own explanation.
Fonksiyon Başlığı / Function Headers Function Headers
Kısa fonksiyonlar fazla açıklamaya ihtiyaç duymaz. Tek bir şey yapan küçük bir fonksiyon için iyi seçilmiş bir isim, genellikle bir yorum başlığından çok daha iyidir. Short functions don't need much description. A well-chosen name for a small function that does one thing is usually better than a comment header.
Herkese Açık Olmayan Kodlarda Javadoc / Javadocs in Nonpublic Code Javadocs in Nonpublic Code
Javadoc'lar herkese açık API'lar için ne kadar yararlıysa, herkese sunulmayan kodlar için o kadar zararlıdır. Bir sistemdeki sınıflar ve fonksiyonlar için Javadoc sayfaları oluşturmak genellikle faydalı değildir; Javadoc yorumlarının eklediği ekstra biçimsellik ise çoğunlukla boş gürültüden ibaret kalır ve dikkati dağıtır. As useful as javadocs are for public APIs, they are anathema to code that is not intended for public consumption. Generating javadoc pages for the classes and functions inside a system is not generally useful, and the extra formality of the javadoc comments amounts to little more than cruft and distraction.
Bu Sayfadaki Kavramlar Concepts on This Page

Çok Fazla Bilgi (Too Much Information)Too Much Information

Kavram AçıklamasıConcept Explanation

Yorumlar kısa ve amaca yönelik olmalıdır. Tarihsel arka plan, RFC veya standart detayları, matematiksel kanıtlar gibi bilgiler yoruma değil, ayrı bir dokümana aittir. Comments should be brief and purposeful. Historical background, RFC or standard details, mathematical proofs and similar information belong in separate documentation, not in comments.

Kötü Örnek (Before)Bad Example (Before)

KÖTÜ — Yorumda tam RFC metniBAD — Full RFC text in comment 
// RFC 2045, Section 6.8: Base64 Content-Transfer-Encoding
// The encoding process represents 24-bit groups...
// (15 satır devam eder)
public String encodeBase64(byte[] data) { ... }

İyi Örnek (After)Good Example (After)

İYİ — Kısa yorum, referans dışarıdaGOOD — Short comment, reference external 
// Base64 kodlaması (RFC 2045 Section 6.8)
public String encodeBase64(byte[] data) { ... }
Pratik İpucuPractical Tip Bir yorumun iki satırdan uzun olması gerekiyorsa, önce bu bilgiyi kodda (iyi isimler, sabitler) veya harici dokümantasyonda ifade etmeye çalışın. If a comment needs to be more than two lines, first try to express this information in the code (good names, constants) or in external documentation.

Bağlantısız Yorum (Inobvious Connection)Inobvious Connection

Kavram AçıklamasıConcept Explanation

Bir yorum anlaşılması için ek açıklamaya ihtiyaç duyuyorsa, başarısız olmuş demektir. Okuyucu yorumu okuduktan sonra hâlâ "filtre byte ne demek?" diye soruyorsa, yorum değer katmaktan çok kafa karıştırıyor. If a comment requires additional explanation to be understood, it has failed. If after reading the comment the reader is still asking "what does a filter byte mean?", the comment is contributing confusion rather than value.

Kötü Örnek (Before)Bad Example (Before)

KÖTÜ — Kim anlar?BAD — Who understands this? 
// 3 baytlık grupları 4 karakterlik sekizliğe dönüştürür
return ((n >> 18) & 0x3f) | (((n >> 12) & 0x3f) << 8)
     | (((n >> 6) & 0x3f) << 16) | ((n & 0x3f) << 24);

İyi Örnek (After)Good Example (After)

İYİ — İsimlendirilmiş adımlarGOOD — Named steps int b0 = (n >> 18) & 0x3f; int b1 = (n >> 12) & 0x3f; int b2 = (n >> 6) & 0x3f; int b3 = n & 0x3f; return b0 | (b1 << 8) | (b2 << 16) | (b3 << 24);
Pratik İpucuPractical Tip Bir yorum okuyucunun kafasında soru işareti bırakıyorsa, o soruların cevabını koda taşıyın — yorum değil, adlandırma kullanın. If a comment leaves a question mark in the reader's mind, move the answer to those questions into the code — use naming, not comments.

Fonksiyon Başlığı (Function Headers)Function Headers

Kavram AçıklamasıConcept Explanation

Küçük ve tek amacı olan fonksiyonlar, yorum başlığına ihtiyaç duymaz. Fonksiyonun adı yeterince açıklayıcıysa yorum artık gereksizdir. İyi seçilmiş bir isim, her zaman yorumdan önce gelir. Small functions with a single purpose don't need a comment header. If the function name is sufficiently descriptive, the comment is already unnecessary. A well-chosen name always comes before comments.

Kötü Örnek (Before)Bad Example (Before)

KÖTÜ — Yorum gereksiz, fonksiyon adı yeterliBAD — Comment unnecessary, function name is enough 
/**
 * Kullanıcıyı geçersiz kılar ve oturumu kapatır.
 */
public void invalidateUserAndLogout(User user) { ... }

İyi Örnek (After)Good Example (After)

İYİ — İyi isimlendirilmiş, yorum yokGOOD — Well named, no comment public void invalidateUserAndLogout(User user) { ... }
Pratik İpucuPractical Tip Bir fonksiyona Javadoc yazmak istediğinizde, önce şunu deneyin: fonksiyon adını ve parametrelerini o kadar açık yapın ki Javadoc gereksiz hale gelsin. When you want to write Javadoc for a function, first try this: make the function name and parameters so clear that the Javadoc becomes unnecessary.
