Argüman Nesneleri / Argument Objects Argument Objects
Bir fonksiyon iki veya üçten fazla argümana ihtiyaç duyuyorsa, bu argümanların bazılarının kendi sınıflarına sarılması gerektiğinin işaretidir. Örneğin, aşağıdaki iki bildirimi karşılaştırın: When a function seems to need more than two or three arguments, it is likely that some of those arguments ought to be wrapped into a class of their own. Consider, for example, the difference between the two following declarations:
Argüman Nesnesi Karşılaştırması Java 
Circle makeCircle(double x, double y, double radius);
Circle makeCircle(Point center, double radius);
Nesneler oluşturarak argüman sayısını azaltmak hile gibi görünebilir, ama değildir. Değişken grupları birlikte geçiriliyorsa —yukarıdaki örnekteki x ve y gibi— büyük olasılıkla kendi adını hak eden bir kavramın parçasıdırlar. Reducing the number of arguments by creating objects out of them may seem like cheating, but it's not. When groups of variables are passed together, the way x and y are in the example above, they are likely part of a concept that deserves a name of its own.
Argüman Listeleri / Argument Lists Argument Lists
Bazen bir fonksiyona değişken sayıda argüman geçirmek isteriz. Örneğin, String.format metodunu ele alalım: Sometimes we want to pass a variable number of arguments into a function. Consider, for example, the String.format method:
String.format Örneği Java 
String.format("%s worked %.2f hours.", name, hours);
Eğer değişken argümanlar —yukarıdaki örnekte olduğu gibi— hepsi aynı şekilde işleniyorsa, bunlar List türünde tek bir argümana eşdeğerdir. Bu mantıkla String.format aslında ikili (dyadic) bir fonksiyondur. Gerçekten de aşağıdaki bildirim açıkça ikiliye işaret eder. If the variable arguments are all treated identically, as they are in the example above, then they are equivalent to a single argument of type List. By that reasoning, String.format is actually dyadic. Indeed, the declaration of String.format as shown below is clearly dyadic.
String.format Bildirimi Java 
public String format(String format, Object... args)
Dolayısıyla aynı kurallar geçerlidir. Değişken argüman alan fonksiyonlar tekli (monadic), ikili (dyadic), hatta üçlü (triadic) olabilir. Ancak bundan daha fazla argüman vermek hata olur. So all the same rules apply. Functions that take variable arguments can be monads, dyads, or even triads. But it would be a mistake to give them more arguments than that.
Değişken Argümanlı Fonksiyon Türleri Java 
void monad(Integer... args);
void dyad(String name, Integer... args);
void triad(String name, int count, Integer... args);
Fiiller ve Anahtar Kelimeler / Verbs and Keywords Verbs and Keywords
Bir fonksiyon için iyi isimler seçmek, fonksiyonun amacını ve argümanların sırasını ve niyetini açıklamada uzun bir yol kat eder. Tekli (monadic) bir fonksiyonda, fonksiyon ve argüman çok güzel bir fiil/isim çifti oluşturmalıdır. Örneğin, write(name) oldukça çağrışımlıdır. Bu "isim" neyse, "yazılıyor" denmektedir. Daha da iyi bir isim writeField(name) olabilir; bu bize "isim" denen şeyin bir "alan" (field) olduğunu söyler. Choosing good names for a function can go a long way toward explaining the intent of the function and the order and intent of the arguments. In the case of a monad, the function and argument should form a very nice verb/noun pair. For example, write(name) is very evocative. Whatever this "name" thing is, it is being "written." An even better name might be writeField(name), which tells us that the "name" thing is a "field."
Bu son örnek, fonksiyon isminin anahtar kelime (keyword) formuna bir örnektir. Bu formda argüman isimlerini fonksiyon isminin içine kodlarız. Örneğin, assertEquals yerine assertExpectedEqualsActual(expected, actual) yazmak, argüman sırasını hatırlama sorununu büyük ölçüde ortadan kaldırır. This last is an example of the keyword form of a function name. Using this form we encode the names of the arguments into the function name. For example, assertEquals might be better written as assertExpectedEqualsActual(expected, actual). This strongly mitigates the problem of having to remember the ordering of the arguments.

Argüman Nesneleri (Argument Objects)Argument Objects

Kavram AçıklamasıConcept Explanation

Birlikte gezinen argümanlar muhtemelen bir kavramın parçasıdır ve kendi sınıflarına sarılmalıdır. Bu, argüman sayısını azaltır ve kodu daha okunabilir hale getirir.Arguments that travel together are likely part of a concept and should be wrapped into their own class. This reduces argument count and makes code more readable.

Kötü Örnek (Before)Bad Example (Before)

KÖTÜ — Çok fazla ilkel argümanBAD — Too many primitive arguments 
function createUser(firstName, lastName, email, age, city, country) {
    // 6 argüman - hangi sırada?
    return { firstName, lastName, email, age, city, country };
}

createUser("Ali", "Yılmaz", "ali@mail.com", 28, "İstanbul", "TR");

İyi Örnek (After)Good Example (After)

İYİ — Nesnelerle gruplandırılmışGOOD — Grouped with objects 
function createUser(personalInfo, address) {
    return { ...personalInfo, ...address };
}

const personalInfo = { firstName: "Ali", lastName: "Yılmaz", email: "ali@mail.com", age: 28 };
const address = { city: "İstanbul", country: "TR" };
createUser(personalInfo, address);
Pratik İpucuPractical Tip3'ten fazla argüman gördüğünüzde kendinize sorun: "Bu argümanlar birlikte bir kavram mı oluşturuyor?" Eğer öyleyse, onları bir nesneye sarın.When you see more than 3 arguments, ask yourself: "Do these arguments form a concept together?" If so, wrap them into an object.

Anahtar Kelime Formu (Keyword Form)Keyword Form

Kavram AçıklamasıConcept Explanation

Fonksiyon isminin içine argüman isimlerini kodlamak, argüman sırasını hatırlama sorununu ortadan kaldırır. Bu özellikle çok argümanlı fonksiyonlarda faydalıdır.Encoding argument names into the function name eliminates the problem of remembering argument order. This is especially useful for multi-argument functions.

Kötü Örnek (Before)Bad Example (Before)

KÖTÜ — Argüman sırası belirsizBAD — Ambiguous argument order 
def copy(source, destination):
    shutil.copy(source, destination)

# Hangisi kaynak, hangisi hedef?
copy("/tmp/a.txt", "/home/b.txt")

İyi Örnek (After)Good Example (After)

İYİ — Argümanlar isimde kodlanmışGOOD — Arguments encoded in name 
def copySourceToDestination(source, destination):
    shutil.copy(source, destination)

# Sıra açık!
copySourceToDestination("/tmp/a.txt", "/home/b.txt")
Pratik İpucuPractical TipBirden fazla argümanlı fonksiyonlarda, argüman sırasının kafa karıştırdığı yerlerde anahtar kelime formunu kullanın. assertEquals yerine assertExpectedEqualsActual çok daha açıktır.For multi-argument functions where argument order is confusing, use the keyword form. assertExpectedEqualsActual is much clearer than assertEquals.

Değişken Argümanlar (Variable Arguments)Variable Arguments

Kavram AçıklamasıConcept Explanation

Değişken sayıda argüman alan fonksiyonlar, aynı şekilde işlenen argümanlar tek bir liste olarak değerlendirilerek sınıflandırılır. Böylece argüman sayısı kuralları geçerliliğini korur.Functions with variable arguments are classified by treating identically-processed arguments as a single list. This way, argument count rules still apply.

Kötü Örnek (Before)Bad Example (Before)

KÖTÜ — Sabit sayıda argüman zorlamasıBAD — Forcing fixed argument count 
function logMessage(level, msg1, msg2, msg3, msg4) {
    // Kullanılmayan argümanlar null geçilir
    console.log(level, msg1, msg2 || "", msg3 || "", msg4 || "");
}

logMessage("ERROR", "Disk dolu", null, null, null);

İyi Örnek (After)Good Example (After)

İYİ — Rest parametresi ileGOOD — Using rest parameter 
function logMessage(level, ...messages) {
    // level + messages = ikili (dyadic)
    console.log(level, messages.join(" "));
}

logMessage("ERROR", "Disk dolu");
logMessage("WARN", "Bellek", "azalıyor", "dikkat!");
Pratik İpucuPractical TipDeğişken argümanlar kullanırken bile toplam mantıksal argüman sayısını 3'ün altında tutmaya çalışın. Rest parametresi tek bir liste argümanı olarak sayılır.Even when using variable arguments, try to keep the total logical argument count below 3. A rest parameter counts as a single list argument.
