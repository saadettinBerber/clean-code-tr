Çıkış Argümanları / Output Arguments Output Arguments
Argümanlar en doğal olarak bir fonksiyona girdi (input) olarak yorumlanır. Birkaç yıldan fazla programlama deneyiminiz varsa, eminim girdi değil aslında çıktı (output) olan bir argüman karşısında bir an duraksadığınız olmuştur. Örneğin: Arguments are most naturally interpreted as inputs to a function. If you have been programming for more than a few years, I'm sure you've done a double-take on an argument that was actually an output rather than an input. For example:
appendFooter Belirsizliği Java 
appendFooter(s);
Bu fonksiyon s'yi bir şeye footer olarak mı ekliyor? Yoksa s'ye bir footer mı ekliyor? s bir girdi mi yoksa çıktı mı? Fonksiyon imzasına bakmak uzun sürmez: Does this function append s as the footer to something? Or does it append some footer to s? Is s an input or an output? It doesn't take long to look at the function signature and see:
Fonksiyon İmzası Java 
public void appendFooter(StringBuffer report)
Bu, konuyu açıklığa kavuşturur, ama fonksiyonun bildirimini kontrol etmek pahasına. Fonksiyon imzasını kontrol etmeye zorlayan her şey bilişsel bir kesinti (cognitive break) yaratır ve bundan kaçınılmalıdır. This clarifies the issue, but only at the expense of checking the declaration of the function. Anything that forces you to check the function signature is equivalent to a double-take. It's a cognitive break and should be avoided.
Nesne yönelimli programlama öncesi dönemde bazen çıkış argümanlarına ihtiyaç duyuluyordu. Ancak nesne yönelimli dillerde çıkış argümanlarına olan ihtiyacın büyük kısmı ortadan kalkmıştır çünkü this bir çıkış argümanı olarak hareket etmek üzere tasarlanmıştır. Başka bir deyişle, appendFooter'ın şu şekilde çağrılması daha iyi olurdu: In the days before object oriented programming it was sometimes necessary to have output arguments. However, much of the need for output arguments disappears in OO languages because this is intended to act as an output argument. In other words, it would be better for appendFooter to be invoked as:
Nesne Yönelimli Yaklaşım Java 
report.appendFooter();
Genel olarak çıkış argümanlarından (output arguments) kaçınılmalıdır. Fonksiyonunuz bir şeyin durumunu değiştirmek zorundaysa, ait olduğu nesnenin durumunu değiştirsin. In general output arguments should be avoided. If your function must change the state of something, have it change the state of its owning object.
Komut Sorgu Ayrımı / Command Query Separation Command Query Separation
Fonksiyonlar ya bir şey yapmalı ya da bir şeye cevap vermeli, ama ikisini birden yapmamalıdır. Ya fonksiyonunuz bir nesnenin durumunu değiştirmeli ya da o nesne hakkında bilgi döndürmelidir. İkisini birden yapmak genellikle kafa karışıklığına yol açar. Örneğin, şu fonksiyonu ele alalım: Functions should either do something or answer something, but not both. Either your function should change the state of an object, or it should return some information about that object. Doing both often leads to confusion. Consider, for example, the following function:
Belirsiz set Fonksiyonu Java 
public boolean set(String attribute, String value);
Bu fonksiyon, adı verilen bir niteliğin (attribute) değerini ayarlar ve başarılıysa true, böyle bir nitelik yoksa false döndürür. Bu, şöyle garip ifadelere yol açar: This function sets the value of a named attribute and returns true if it is successful and false if no such attribute exists. This leads to odd statements like this:
Kafa Karıştırıcı Kullanım Java 
if (set("username", "unclebob"))...
Bunu okuyucunun bakış açısından düşünün. Ne anlama geliyor? "username" niteliği daha önce "unclebob" olarak ayarlanmış mı diye mi soruyor? Yoksa "username" niteliği başarıyla "unclebob" olarak ayarlandı mı diye mi soruyor? Çağrıdan anlamı çıkarmak zordur çünkü "set" kelimesinin fiil mi yoksa sıfat mı olduğu belli değildir. Imagine this from the point of view of the reader. What does it mean? Is it asking whether the "username" attribute was previously set to "unclebob"? Or is it asking whether the "username" attribute was successfully set to "unclebob"? It's hard to infer the meaning from the call because it's not clear whether the word "set" is a verb or an adjective.
Gerçek çözüm, komutu sorgudan ayırarak belirsizliğin oluşmasını engellemektir: The real solution is to separate the command from the query so that the ambiguity cannot occur:
Komut ve Sorgu Ayrılmış Java 
if (attributeExists("username")) {
    setAttribute("username", "unclebob");
    ...
}

Çıkış Argümanları (Output Arguments)Output Arguments

Kavram AçıklamasıConcept Explanation

Çıkış argümanları, fonksiyonun girdi yerine çıktı olarak kullandığı argümanlardır. Nesne yönelimli programlamada this zaten bu amaca hizmet ettiği için çıkış argümanlarına gerek kalmaz.Output arguments are arguments that a function uses as output rather than input. In OOP, this already serves this purpose, so output arguments are unnecessary.

Kötü Örnek (Before)Bad Example (Before)

KÖTÜ — Çıkış argümanıBAD — Output argument 
function addItemsToList(items, targetList) {
    // targetList girdi mi çıktı mı?
    for (const item of items) {
        targetList.push(transform(item));
    }
}

const result = [];
addItemsToList(rawData, result);  // result değiştiriliyor

İyi Örnek (After)Good Example (After)

İYİ — Dönüş değeri ileGOOD — Using return value 
function transformItems(items) {
    return items.map(item => transform(item));
}

const result = transformItems(rawData);  // Açık ve temiz
Pratik İpucuPractical TipBir fonksiyonun bir şeyin durumunu değiştirmesi gerekiyorsa, ait olduğu nesnenin metodunu kullanın: report.appendFooter() daha iyidir appendFooter(report)'dan.If a function needs to change something's state, use a method on the owning object: report.appendFooter() is better than appendFooter(report).

Komut Sorgu Ayrımı (Command Query Separation)Command Query Separation

Kavram AçıklamasıConcept Explanation

Komut Sorgu Ayrımı (CQS) ilkesine göre fonksiyonlar ya bir eylem gerçekleştirmeli (komut) ya da bilgi döndürmeli (sorgu) ama ikisini birden yapmamalıdır. Bu ilke, kodun okunabilirliğini ve öngörülebilirliğini artırır.According to Command Query Separation (CQS), functions should either perform an action (command) or return information (query), but not both. This principle increases code readability and predictability.

Kötü Örnek (Before)Bad Example (Before)

KÖTÜ — Hem komut hem sorguBAD — Both command and query 
class Stack:
    def pop(self):
        # Hem değeri döndürür hem de durumu değiştirir
        item = self.items[-1]
        self.items = self.items[:-1]
        return item

# "pop" bir eylem mi yoksa bir soru mu?
if stack.pop() == "target":
    # Eşleşmezse eleman geri gelmez!
    ...

İyi Örnek (After)Good Example (After)

İYİ — Ayrılmış komut ve sorguGOOD — Separated command and query 
class Stack:
    def peek(self):
        # Sorgu: sadece bakar, değiştirmez
        return self.items[-1]

    def remove_top(self):
        # Komut: sadece siler, döndürmez
        self.items.pop()

if stack.peek() == "target":
    stack.remove_top()  # Güvenli ve açık
Pratik İpucuPractical TipBir fonksiyon yazarken kendinize sorun: "Bu bir soru mu yoksa bir eylem mi?" Eğer ikisi de ise, ikiye bölün. Sorgular yan etkisiz (side-effect-free) olmalı, komutlar ise hiçbir şey döndürmemelidir.When writing a function, ask: "Is this a question or an action?" If it's both, split it. Queries should be side-effect-free, commands should return nothing.

Bilişsel Kesinti (Cognitive Break)Cognitive Break

Kavram AçıklamasıConcept Explanation

Bilişsel kesinti, bir kod satırını anlamak için okuyucunun başka bir yere bakması gerektiğinde oluşan akış bozulmasıdır. Fonksiyon imzalarını kontrol etmeye zorlamak, belgelere başvurmak veya başka dosyalara atlamak gibi durumlar bilişsel kesintilere neden olur.A cognitive break occurs when understanding a line of code requires the reader to look elsewhere. Checking function signatures, consulting documentation, or jumping to other files all cause cognitive breaks.

Kötü Örnek (Before)Bad Example (Before)

KÖTÜ — Anlam için imza gerekliBAD — Signature needed for meaning 
// Bu ne yapıyor? Belgeyi mi formatlar, formatı mı belgeler?
format(document);

// Diske mi yazıyor, listeye mi ekliyor?
save(users, file);

// Sonucu döndürür mü, yerinde mi değiştirir?
sort(data);

İyi Örnek (After)Good Example (After)

İYİ — İmza gerektirmeyen açıklıkGOOD — Self-evident without signatures 
document.formatAsMarkdown();

saveUsersToFile(users, filePath);

const sortedData = sortByDate(data);
Pratik İpucuPractical TipKodunuzu okuyucunun gözüyle inceleyin. Eğer bir satırı anlamak için başka bir yere bakmanız gerekiyorsa, isim seçimi veya yapı yeterince açık değildir. Amacınız "sıfır atlama" (zero-jump) okunabilirlik olmalıdır.Review your code through the reader's eyes. If understanding a line requires looking elsewhere, the naming or structure isn't clear enough. Aim for "zero-jump" readability.
