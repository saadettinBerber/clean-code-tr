Öncül ve İlkeler / Prequel and Principles Prequel and Principles
Birçok açıdan bu kitap, 2002 yılında yazdığım Agile Software Development: Principles, Patterns, and Practices (PPP) başlıklı kitabın bir "öncülü"dür. PPP kitabı, nesne yönelimli tasarım ilkeleri (object-oriented design principles) ve profesyonel geliştiricilerin kullandığı birçok pratikle ilgilenir. Eğer PPP'yi okumadıysanız, bu kitabın anlattığı hikâyeyi devam ettirdiğini görebilirsiniz. Eğer zaten okuduysanız, o kitabın duygu ve düşüncelerinin birçoğunun burada kod düzeyinde yankılandığını fark edeceksiniz. In many ways this book is a "prequel" to a book I wrote in 2002 entitled Agile Software Development: Principles, Patterns, and Practices (PPP). The PPP book concerns itself with the principles of object-oriented design, and many of the practices used by professional developers. If you have not read PPP, then you may find that it continues the story told by this book. If you have already read it, then you'll find many of the sentiments of that book echoed in this one at the level of code.
Bu kitapta çeşitli tasarım ilkelerine (design principles) ara ara atıflar bulacaksınız. Bunlar arasında Tek Sorumluluk İlkesi (Single Responsibility Principle — SRP), Açık Kapalı İlkesi (Open Closed Principle — OCP) ve Bağımlılık Tersine Çevirme İlkesi (Dependency Inversion Principle — DIP) gibi ilkeler yer almaktadır. Bu ilkeler PPP'de derinlemesine anlatılmıştır. In this book you will find sporadic references to various principles of design. These include the Single Responsibility Principle (SRP), the Open Closed Principle (OCP), and the Dependency Inversion Principle (DIP) among others. These principles are described in depth in PPP.
Sonuç / Conclusion Conclusion
Sanat üzerine yazılmış kitaplar sizi sanatçı yapmayı vaat etmez. Yapabilecekleri tek şey, diğer sanatçıların kullandığı araçları, teknikleri ve düşünce süreçlerini size sunmaktır. Bu kitap da size iyi bir programcı olmayı vaat edemez. Size "kod duygusu" (code-sense) kazandırmayı vaat edemez. Yapabileceği tek şey, iyi programcıların düşünce süreçlerini ve kullandıkları hileleri, teknikleri ve araçları göstermektir. Books on art don't promise to make you an artist. All they can do is give you some of the tools, techniques, and thought processes that other artists have used. So too this book cannot promise to make you a good programmer. It cannot promise to give you "code-sense." All it can do is show you the thought processes of good programmers and the tricks, techniques, and tools that they use.
Tıpkı bir sanat kitabı gibi, bu kitap da ayrıntılarla dolu olacaktır. Çok sayıda kod göreceksiniz. İyi kod göreceksiniz ve kötü kod göreceksiniz. Kötü kodun iyi koda dönüştürüldüğünü göreceksiniz. Sezgisel kurallar (heuristics), disiplinler ve teknikler listeleri göreceksiniz. Örnek üstüne örnek göreceksiniz. Bundan sonrası size kalmış. Just like a book on art, this book will be full of details. There will be lots of code. You'll see good code and you'll see bad code. You'll see bad code transformed into good code. You'll see lists of heuristics, disciplines, and techniques. You'll see example after example. After that, it's up to you.
Bir performansına giderken yolunu kaybeden konser kemancısıyla ilgili eski şakayı hatırlayın. Köşedeki yaşlı bir adama durdu ve Carnegie Hall'a nasıl gideceğini sordu. Yaşlı adam kemancıya ve kolunun altına sıkıştırdığı kemana baktı ve dedi ki: "Pratik, evlat. Pratik!" Remember the old joke about the concert violinist who got lost on his way to a performance? He stopped an old man on the corner and asked him how to get to Carnegie Hall. The old man looked at the violinist and the violin tucked under his arm, and said: "Practice, son. Practice!"

Tasarım İlkeleri — SOLID (Design Principles)Design Principles — SOLID

Kavram AçıklamasıConcept Explanation

SOLID, nesne yönelimli tasarımın beş temel ilkesinin kısaltmasıdır. Bu sayfada üçüne atıfta bulunulur: SRP (Tek Sorumluluk İlkesi) — her sınıf yalnızca bir nedenden dolayı değişmelidir; OCP (Açık Kapalı İlkesi) — sınıflar genişlemeye açık, değişikliğe kapalı olmalıdır; DIP (Bağımlılık Tersine Çevirme İlkesi) — üst seviye modüller alt seviye modüllere değil, soyutlamalara bağımlı olmalıdır.SOLID is an acronym for five fundamental principles of object-oriented design. Three are referenced on this page: SRP (Single Responsibility Principle) — a class should have only one reason to change; OCP (Open Closed Principle) — classes should be open for extension, closed for modification; DIP (Dependency Inversion Principle) — high-level modules should depend on abstractions, not low-level modules.

Kötü Örnek (Before) — SRP İhlaliBad Example (Before) — SRP Violation

KÖTÜ — Sınıf birden fazla sorumluluk taşıyorBAD — Class has multiple responsibilities 
class UserService:
    def authenticate(self, username, password):
        # Kullanıcı doğrulama
        user = db.query(f"SELECT * FROM users WHERE name='{username}'")
        return check_password(user, password)

    def generate_monthly_report(self, user_id):
        # Aylık rapor oluşturma
        data = db.query(f"SELECT * FROM orders WHERE user={user_id}")
        return create_pdf(data)

    def send_welcome_email(self, email):
        # E-posta gönderme
        smtp = connect_smtp()
        smtp.send(email, "Welcome!", "...")
Bu sınıf üç farklı sorumluluk taşıyor: kimlik doğrulama, rapor oluşturma ve e-posta gönderme. Herhangi birindeki değişiklik tüm sınıfı etkiler.This class carries three different responsibilities: authentication, report generation, and email sending. A change in any one affects the entire class.

İyi Örnek (After) — SRP UygulanmışGood Example (After) — SRP Applied

İYİ — Her sınıf tek bir sorumluluk taşıyorGOOD — Each class has a single responsibility 
class AuthenticationService:
    def authenticate(self, username, password):
        user = self.user_repo.find_by_name(username)
        return self.password_checker.verify(user, password)

class ReportService:
    def generate_monthly_report(self, user_id):
        data = self.order_repo.find_by_user(user_id)
        return self.pdf_generator.create(data)

class NotificationService:
    def send_welcome_email(self, email):
        self.mailer.send(email, "Welcome!", "...")
Pratik İpucuPractical TipBir sınıfın ne yaptığını tek cümleyle anlatamıyorsanız, muhtemelen birden fazla sorumluluğu vardır. "Bu sınıf X yapar VE Y yapar" diyorsanız, o "VE" kelimesi bir SRP ihlali sinyalidir.If you can't describe what a class does in a single sentence, it probably has more than one responsibility. If you say "This class does X AND Y," that "AND" is a signal of an SRP violation.

Pratik ve Zanaat (Practice & Craftsmanship)Practice & Craftsmanship

Kavram AçıklamasıConcept Explanation

Kitap okumak, iyi bir programcı olmak için yeterli değildir — tıpkı sanat kitaplarının sizi sanatçı yapmadığı gibi. Carnegie Hall fıkrasının mesajı açıktır: ustalık ancak pratikle gelir. "Kod duygusu" (code-sense), yani temiz kodu kirli koddan ayırt etme ve onu düzeltme yetisi, ancak yoğun pratikle kazanılır. Okumak araçları verir, pratik ise beceriyi geliştirir.Reading a book is not enough to become a good programmer — just as art books don't make you an artist. The message of the Carnegie Hall joke is clear: mastery comes only through practice. "Code-sense" — the ability to distinguish clean code from dirty code and fix it — is acquired only through intense practice. Reading gives you the tools; practice develops the skill.

Kötü Örnek (Before)Bad Example (Before)

KÖTÜ — "Okudum, biliyorum" yaklaşımıBAD — "I read it, so I know it" approach 
// "Clean Code okudum, artık temiz kod yazarım"
// Sonuç: Kitaptaki örnekleri ezberlemek ama pratiğe dökmemek

function processData(d) {
    let r = [];
    for (let i = 0; i < d.length; i++) {
        if (d[i].s === 'A') {
            r.push({ n: d[i].n, v: d[i].p * 1.18 });
        }
    }
    return r;
}
Kitabı okumuş olsa bile, geliştirici hâlâ tek harfli değişkenler, belirsiz koşullar ve sihirli sayılar kullanıyor. Bilgi pratiğe dönüşmemiş.Even though the developer may have read the book, they still use single-letter variables, unclear conditions, and magic numbers. Knowledge hasn't been converted to practice.

İyi Örnek (After)Good Example (After)

İYİ — Pratik yaparak içselleştirilmiş temiz kodGOOD — Clean code internalized through practice 
const TAX_RATE = 0.18;

function calculatePricesWithTax(products) {
    return products
        .filter(product => product.status === 'Active')
        .map(product => ({
            name: product.name,
            priceWithTax: product.price * (1 + TAX_RATE)
        }));
}
Pratik İpucuPractical TipHer gün küçük bir "kata" (kod egzersizi) yapın. Eski kodunuzu açıp bir fonksiyonu yeniden düzenleyin. Pratik, mükemmelliğe giden tek yoldur — Carnegie Hall'a giden yol gibi.Do a small "kata" (code exercise) every day. Open your old code and refactor one function. Practice is the only path to mastery — just like the road to Carnegie Hall.

Kod Duygusu (Code-Sense)Code-Sense

Kavram AçıklamasıConcept Explanation

Kod duygusu, bir geliştiricinin koda bakarak onun temiz mi kirli mi olduğunu sezgisel olarak anlayabilmesi ve kirli kodu temize dönüştürmek için stratejiler üretebilmesi yetisidir. Bölüm 1'in başından beri bu kavram işlenmiştir: bazı insanlar doğuştan bu sezgiye sahipken, çoğumuz bunu pratik ve deneyimle kazanırız. Bu kitap o sürece rehberlik etmeyi amaçlar.Code-sense is a developer's ability to intuitively recognize whether code is clean or dirty, and to devise strategies for transforming dirty code into clean code. This concept has been discussed since the beginning of Chapter 1: while some people have this intuition naturally, most of us acquire it through practice and experience. This book aims to guide that process.

Kötü Örnek (Before)Bad Example (Before)

KÖTÜ — Kod duygusu gelişmemiş: "Çalışıyorsa dokunma"BAD — Undeveloped code-sense: "If it works, don't touch it" 
public class Util {
    public static Object proc(Object x, int m) {
        if (m == 1) {
            return ((String) x).toUpperCase();
        } else if (m == 2) {
            return ((String) x).length();
        } else if (m == 3) {
            return ((List) x).size() > 0;
        }
        return null;
    }
}
Bu kodu gören biri "çalışıyorsa sorun yok" diyebilir. Ancak kod duygusu gelişmiş bir geliştirici hemen birçok sorun tespit eder: belirsiz isimler, sihirli sayılar, tür güvenliği eksikliği, tek sorumluluk ihlali.Someone seeing this code might say "if it works, no problem." But a developer with code-sense immediately spots multiple issues: unclear names, magic numbers, lack of type safety, single responsibility violation.

İyi Örnek (After)Good Example (After)

İYİ — Kod duygusu ile yeniden düzenlenmişGOOD — Refactored with code-sense 
public class StringFormatter {
    public String toUpperCase(String text) {
        return text.toUpperCase();
    }

    public int lengthOf(String text) {
        return text.length();
    }
}

public class CollectionValidator {
    public boolean isNonEmpty(List<?> items) {
        return items != null && !items.isEmpty();
    }
}
Pratik İpucuPractical TipKod duygunuzu geliştirmek için kod incelemeleri (code review) yapın — hem başkalarının kodunu inceleyin, hem de kendi eski kodunuza geri dönün. "Bu kodu 6 ay sonra anlayabilir miyim?" sorusu, kod duygunuzun en iyi litmus testidir.To develop your code-sense, do code reviews — both review others' code and revisit your own old code. "Can I understand this code 6 months from now?" is the best litmus test for your code-sense.
