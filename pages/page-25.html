Zihinsel Eşleştirmeden Kaçınmak / Avoid Mental Mapping Avoid Mental Mapping
Okuyucular, isimlerinizi halihazırda bildikleri başka isimlere zihinsel olarak tercüme etmek zorunda kalmamalıdır. Bu sorun genellikle, ne problem alanı (problem domain) terimlerini ne de çözüm alanı (solution domain) terimlerini kullanmamayı tercih etmekten kaynaklanır. Readers shouldn’t have to mentally translate your names into other names they already know. This problem generally arises from a choice to use neither problem domain terms nor solution domain terms.
Bu, tek harfli değişken isimleriyle (single-letter variable names) ilgili bir sorundur. Elbette, kapsamı (scope) çok darsa ve başka hiçbir isimle çakışmıyorsa, bir döngü sayacı (loop counter) i, j veya k olarak adlandırılabilir (asla l değil!). Bunun nedeni, döngü sayaçları için bu tek harfli isimlerin geleneksel olmasıdır. Ancak, diğer çoğu bağlamda tek harfli bir isim kötü bir seçimdir; bu sadece okuyucunun gerçek kavramla zihinsel olarak eşleştirmesi gereken bir yer tutucudur (placeholder). Bir ismi c olarak kullanmak için, a ve b zaten alınmış olmasından daha kötü bir neden olamaz. This is a problem with single-letter variable names. Certainly a loop counter may be named i or j or k (though never l!) if its scope is very small and no other names can conflict with it. This is because those single-letter names for loop counters are traditional. However, in most other contexts a single-letter name is a poor choice; it’s just a placeholder that the reader must mentally map to the actual concept. There can be no worse reason for using the name c than because a and b were already taken.
Genelde programcılar oldukça zeki insanlardır. Zeki insanlar bazen zihinsel hokkabazlık (mental juggling) yeteneklerini sergileyerek zekalarıyla hava atmayı severler. Sonuçta, r harfinin URL'nin host ve şema (scheme) kısımları çıkarılmış küçük harfli hali olduğunu güvenilir bir şekilde hatırlayabiliyorsanız, açıkça çok zeki olmalısınız. In general programmers are pretty smart people. Smart people sometimes like to show off their smarts by demonstrating their mental juggling abilities. After all, if you can reliably remember that r is the lower-cased version of the url with the host and scheme removed, then you must clearly be very smart.
Zeki bir programcı ile profesyonel bir programcı arasındaki farklardan biri, profesyonelin açıklığın kral (clarity is king) olduğunu anlamasıdır. Profesyoneller yeteneklerini iyilik için kullanırlar ve başkalarının anlayabileceği kodlar yazarlar. One difference between a smart programmer and a professional programmer is that the professional understands that clarity is king. Professionals use their powers for good and write code that others can understand.
Sınıf İsimleri / Class Names Class Names
Sınıflar ve nesneler; Customer, WikiPage, Account ve AddressParser gibi isim (noun) veya isim tamlaması şeklinde adlara sahip olmalıdır. Bir sınıfın adında Manager, Processor, Data veya Info gibi kelimelerden kaçının. Bir sınıf adı fiil (verb) olmamalıdır. Classes and objects should have noun or noun phrase names like Customer, WikiPage, Account, and AddressParser. Avoid words like Manager, Processor, Data, or Info in the name of a class. A class name should not be a verb.
Metot İsimleri / Method Names Method Names
Metotlar; postPayment, deletePage veya save gibi fiil (verb) veya fiil tamlaması şeklinde adlara sahip olmalıdır. Erişimciler (accessors), değiştiriciler (mutators) ve yüklemler (predicates); değerlerine göre adlandırılmalı ve JavaBean standartlarına uygun olarak get, set ve is önekleriyle başlatılmalıdır. Methods should have verb or verb phrase names like postPayment, deletePage, or save. Accessors, mutators, and predicates should be named for their value and prefixed with get, set, and is according to the javabean standard. 
string name = employee.getName();
customer.setName("mike");
if (paycheck.isPosted())...
Yapıcılar (constructors) aşırı yüklendiğinde (overloaded), argümanları betimleyen isimlere sahip statik fabrika metotlarını (static factory methods) kullanın. Örneğin: When constructors are overloaded, use static factory methods with names that describe the arguments. For example, 
Complex fulcrumPoint = Complex.FromRealNumber(23.0);
şu kullanımdan genellikle daha iyidir: is generally better than 
Complex fulcrumPoint = new Complex(23.0);
Karşılık gelen yapıcıları (constructors) private yaparak bu metotların kullanımını zorunlu kılmayı düşünebilirsiniz. Consider enforcing their use by making the corresponding constructors private.

Zihinsel Eşleştirme (Mental Mapping)Mental Mapping

Kavram AçıklamasıConcept Explanation

Kod okuyucusu, gördüğü bir ismi zihninde asıl anlamına dönüştürmek için çaba sarf etmemelidir. Tek harfli isimler (döngü sayaçları hariç) veya sadece yazarın bildiği kısaltmalar zihinsel yük yaratır. Profesyonel bir programcı, zekasını karmaşık eşleştirmeleri hatırlamak için değil, herkesin anlayabileceği netlikte kod yazmak için kullanır.A code reader should not have to spend effort mentally converting a name into its actual meaning. Single-letter names (except for loop counters) or abbreviations known only to the author create mental burden. A professional programmer uses their intelligence not to remember complex mappings, but to write code clear enough for everyone to understand.

Kötü Örnek (Before)Bad Example (Before)

KÖTÜ — Zihinsel eşleştirme gerektiren isimlerBAD — Names requiring mental mapping 
for (int i = 0; i < n; i++) {
    var r = getR(u[i]);
    // r neydi? u neydi? r -> result, u -> urls?
}
Burada okuyucu sürekli r'nin "sonuç" ve u'nun "URL listesi" olduğunu hatırlamak zorundadır. Bu bir "zihinsel hokkabazlık"tır.Here, the reader must constantly remember that r is "result" and u is "URL list." This is "mental juggling."

İyi Örnek (After)Good Example (After)

İYİ — Açık ve net isimlerGOOD — Clear and explicit names 
for (String url : urlList) {
    String pageContent = fetchPageContent(url);
    // sayfa içeriği doğrudan anlaşılıyor
}
Pratik İpucuPractical TipSadece sizin bildiğiniz "kısayol" isimlerden kaçının. Kodunuzu, projeye bugün katılan birinin sözlüğe bakmadan anlayabileceği şekilde yazın. "Clarity is King" (Açıklık Kraldır).Avoid "shortcut" names known only to you. Write your code so that someone joining the project today can understand it without looking at a glossary. "Clarity is King."

Sınıf İsimlendirme (Class Naming)Class Naming

Kavram AçıklamasıConcept Explanation

Sınıf isimleri nesneleri temsil ettiği için isim (noun) veya isim tamlaması olmalıdır. Sınıf isimlerinde fiil kullanılmamalıdır. Ayrıca Data, Info, Manager gibi çok genel ve belirsiz kelimelerden kaçınılmalıdır, çünkü bu kelimeler sınıfın sorumluluğunu netleştirmez.Since class names represent objects, they should be nouns or noun phrases. Verbs should not be used in class names. Additionally, very general and vague words like Data, Info, Manager should be avoided because they don't clarify the class's responsibility.

Kötü Örnek (Before)Bad Example (Before)

KÖTÜ — Belirsiz veya hatalı sınıf isimleriBAD — Vague or incorrect class names 
class CleanData { } // Fiil gibi duruyor (Verbal)
class UserInfo { }  // Info gürültü kelimesidir
class Processor { } // Çok genel, neyi işliyor?
UserInfo yerine sadece User demek çoğu zaman yeterlidir. Processor gibi isimler sınıfın "her şeyi yapan" dev bir yapıya dönüşmesine neden olabilir.Saying just User instead of UserInfo is often sufficient. Names like Processor can lead to the class becoming a "do-it-all" giant structure.

İyi Örnek (After)Good Example (After)

İYİ — Net isim tamlamalarıGOOD — Clear noun phrases 
class User { }
class AddressParser { }
class PaymentAccount { }
Pratik İpucuPractical TipEğer bir sınıfı isimlendirmekte zorlanıyorsanız, muhtemelen o sınıf birden fazla iş yapıyordur. Sınıfı küçülttüğünüzde daha spesifik ve anlamlı bir isim bulmak kolaylaşacaktır.If you're having trouble naming a class, it probably does more than one thing. When you make the class smaller, finding a specific and meaningful name will become easier.

Metot İsimlendirme (Method Naming)Method Naming

Kavram AçıklamasıConcept Explanation

Metotlar eylemleri temsil eder, bu yüzden fiil (verb) veya fiil tamlaması olmalıdırlar. Erişimciler için get, değiştiriciler için set ve mantıksal (boolean) kontroller için is standartlarına uyulmalıdır.Methods represent actions, so they should be verbs or verb phrases. Standards like get for accessors, set for mutators, and is for logical (boolean) checks should be followed.

Kötü Örnek (Before)Bad Example (Before)

KÖTÜ — Eylem belirtmeyen isimlerBAD — Names not indicating action 
user.name();      // Ne yapıyor? Get mi set mi?
account.status(); // Durumu mu dönüyor yoksa güncelliyor mu?
if(order.posted)  // Boolean mı metot mu?

İyi Örnek (After)Good Example (After)

İYİ — Fiil odaklı isimlerGOOD — Verb-oriented names 
user.getName();
account.updateStatus(newStatus);
if(order.isPosted())
Pratik İpucuPractical TipBir metot ismini okuduğunuzda bir soru sormuş gibi (isReady?) veya bir emir vermiş gibi (saveUser!) hissetmelisiniz. Eğer metot isminde "and", "or" gibi bağlaçlar geçiyorsa, metot muhtemelen birden fazla iş yapıyordur.When you read a method name, it should feel like asking a question (isReady?) or giving a command (saveUser!). If connectors like "and" or "or" appear in the method name, the method is likely doing more than one thing.

Statik Fabrika Metotları (Static Factory Methods)Static Factory Methods

Kavram AçıklamasıConcept Explanation

Yapıcılar (constructors) aşırı yüklendiğinde (farklı parametrelerle birden fazla yapıcı), hangisinin ne amaçla kullanıldığını anlamak zorlaşabilir. Statik fabrika metotları, niyet belirten isimleri sayesinde nesne oluşturma sürecini çok daha açık hale getirir.When constructors are overloaded (multiple constructors with different parameters), it can be difficult to understand which one is used for what purpose. Static factory methods make the object creation process much clearer thanks to their intention-revealing names.

Kötü Örnek (Before)Bad Example (Before)

KÖTÜ — Kafa karıştırıcı overloaded yapıcılarBAD — Confusing overloaded constructors 
Duration d1 = new Duration(60);   // Saniye mi? Dakika mı?
Duration d2 = new Duration(1, 30); // Saat-dakika mı? Dakika-saniye mi?

İyi Örnek (After)Good Example (After)

İYİ — Kendini açıklayan fabrika metotlarıGOOD — Self-explaining factory methods 
Duration d1 = Duration.fromSeconds(60);
Duration d2 = Duration.fromMinutesAndSeconds(1, 30);
Pratik İpucuPractical TipNesne oluştururken sadece new kullanmak yerine, eğer varsa sınıfın sunduğu statik fabrika metotlarını arayın. Bu metotlar genellikle from..., of..., valueOf... veya newInstance... gibi öneklerle başlar.Instead of just using new when creating objects, look for static factory methods offered by the class if available. These methods usually start with prefixes like from..., of..., valueOf..., or newInstance....
