Kırık Pencereler Teorisi (Broken Windows Theory)Broken Windows Theory

Kavram AçıklamasıConcept Explanation

Pragmatik Programcılar (Dave Thomas ve Andy Hunt) tarafından yazılıma uyarlanan bu metafor, bir binadaki tek bir kırık pencerenin tüm binayı çürümeye (decay) sürükleyebileceğini anlatır. Kodda da aynıdır: tek bir kötü kod parçası, "zaten bozuk, biraz daha bozulsa ne olacak?" zihniyetini tetikler. Kötü kod, daha fazla kötü kodu kışkırtır. Kimsenin umursamadığı izlenimi verir ve ekip standartları hızla erir.Adapted to software by the Pragmatic Programmers (Dave Thomas and Andy Hunt), this metaphor explains that a single broken window in a building can drag the entire structure into decay. The same applies to code: one piece of bad code triggers the mindset of "it's already broken, a little more won't matter." Bad code tempts more bad code. It gives the impression nobody cares, and team standards erode rapidly.

Kötü Örnek (Before)Bad Example (Before)

KÖTÜ — İlk "kırık pencere" ve ardından gelen çürümeBAD — The first "broken window" and the decay that follows 
class OrderService:
    def process(self, order):
        # TODO: validasyon ekle (6 ay önce yazıldı)
        
        # Hızlı düzeltme: bazen None geliyor
        if order is not None:

                for item in order.items:

                if order.customer.vip:

                elif order.country == "FR":
                    total = total * 1.20  # Fransa KDV

                db.save(order)  # hata yönetimi? biri halletsin

Then came "whatever", "don't touch", "add later", "someone else will handle it", and finally except: pass. Each new developer said "it's already broken" and added their own broken window.

İYİ — Kırık pencere yok: her detay özenle ele alınmışGOOD — No broken windows: every detail handled with care 
class OrderService:
    def __init__(self, validator, tax_calculator, repository, notifier):
        self._validator = validator
        self._tax_calculator = tax_calculator
        self._repository = repository
        self._notifier = notifier

    def process(self, order: Order) -> ProcessedOrder:
        self._validator.validate(order)
        
        subtotal = order.calculate_subtotal()
        discount = order.customer.discount_strategy.apply(subtotal)
        tax = self._tax_calculator.calculate(order.country, subtotal - discount)
        
        processed = order.finalize(subtotal, discount, tax)
        self._repository.save(processed)
        self._notifier.send_confirmation(processed)
        
        return processed
Pratik İpucuPractical Tip"İzci Kuralı"nı (Boy Scout Rule) uygulayın: kodu bulduğunuzdan daha temiz bırakın. Bir dosyada çalışırken küçük bir kırık pencere görürseniz (belirsiz isimlendirme, eksik hata yönetimi, sihirli sayı), hemen düzeltin. Kırık pencerelerin birikmesine asla izin vermeyin — ilk kırık pencere en tehlikelisidir.Apply the "Boy Scout Rule": leave the code cleaner than you found it. When working in a file and you spot a small broken window (unclear naming, missing error handling, magic number), fix it immediately. Never let broken windows accumulate — the first one is the most dangerous.

Detaylara Dikkat (Attention to Detail)Attention to Detail

Kavram AçıklamasıConcept Explanation

Bjarne Stroustrup, hata yönetiminin (error handling) eksiksiz olması gerektiğini vurgular. Kısaltılmış hata yönetimi, bellek sızıntıları (memory leaks), yarış durumları (race conditions) ve tutarsız isimlendirme (inconsistent naming) — bunların hepsi detayları geçiştirmenin farklı biçimleridir. Temiz kod, detaylara yakın ilgi gösterir. Her hata durumu düşünülmüş, her sınır koşulu ele alınmış, her isimlendirme tutarlıdır.Bjarne Stroustrup emphasizes that error handling should be complete. Abbreviated error handling, memory leaks, race conditions, and inconsistent naming — all are different forms of glossing over details. Clean code exhibits close attention to detail. Every error case is considered, every edge condition is handled, every naming convention is consistent.

Kötü Örnek (Before)Bad Example (Before)

KÖTÜ — Detaylar geçiştirilmiş: sızıntılar, yarış durumları, tutarsız isimlendirmeBAD — Details glossed over: leaks, race conditions, inconsistent naming 
public class CacheManager {
    private Map<String, Object> cache = new HashMap<>();
    private Map<String, Long> TTL = new HashMap<>();  // tutarsız isimlendirme

    public Object Get(String key) {  // tutarsız: Get vs put
        return cache.get(key);  // TTL kontrolü yok = bellek sızıntısı
    }

    public void put(String k, Object v, long ttl) {  // k, v belirsiz
        cache.put(k, v);
        TTL.put(k, System.currentTimeMillis() + ttl);
        // Thread-safe değil = yarış durumu
    }

    public void deleteOld() {  // ne zaman çağrılacak? kim çağıracak?
        for (String key : TTL.keySet()) {  // ConcurrentModificationException riski
            if (TTL.get(key) < System.currentTimeMillis()) {
                cache.remove(key);
                TTL.remove(key);  // iterasyon sırasında silme!
            }
        }
    }
}
Dört farklı detay ihmali: (1) Tutarsız isimlendirme (Get vs put, TTL vs cache), (2) TTL kontrolsüz okuma = bellek sızıntısı, (3) Thread güvenliği yok = yarış durumu, (4) İterasyon sırasında koleksiyon değiştirme = çalışma zamanı hatası.Four different detail oversights: (1) Inconsistent naming (Get vs put, TTL vs cache), (2) Reading without TTL check = memory leak, (3) No thread safety = race condition, (4) Modifying collection during iteration = runtime error.

public class ExpiringCache<T> {
    private final ConcurrentHashMap<String, CacheEntry<T>> entries;
    private final ScheduledExecutorService cleaner;

    public Optional<T> get(String key) {
        CacheEntry<T> entry = entries.get(key);
        if (entry == null || entry.isExpired()) {
            entries.remove(key);
            return Optional.empty();
        }
        return Optional.of(entry.value());
    }

    public void put(String key, T value, Duration timeToLive) {
        entries.put(key, new CacheEntry<>(value, Instant.now().plus(timeToLive)));
    }

    private void evictExpiredEntries() {
        entries.entrySet().removeIf(e -> e.getValue().isExpired());
    }

    private record CacheEntry<T>(T value, Instant expiresAt) {
        boolean isExpired() { return Instant.now().isAfter(expiresAt); }
    }
}
Pratik İpucuPractical TipKod yazarken şu kontrol listesini uygulayın: (1) İsimlendirme tutarlı mı? (2) Her hata durumu ele alındı mı? (3) Kaynaklar düzgün serbest bırakılıyor mu? (4) Eşzamanlı erişim güvenli mi? (5) Sınır koşulları (null, boş koleksiyon, negatif değer) test edildi mi? Bir tane bile "sonra hallederim" cevabı varsa, o bir kırık penceredir.Apply this checklist when coding: (1) Is naming consistent? (2) Is every error case handled? (3) Are resources properly released? (4) Is concurrent access safe? (5) Are edge cases (null, empty collection, negative value) tested? If even one answer is "I'll handle it later", that's a broken window.

Düzyazı Gibi Kod (Code as Prose)Code as Prose

Kavram AçıklamasıConcept Explanation

Grady Booch, temiz kodun "iyi yazılmış bir düzyazı (prose) gibi okunması" gerektiğini söyler. Martin bu metaforu genişletir: iyi bir roman gibi, temiz kod da problemdeki gerilimleri ortaya koymalı, onları bir doruk noktasına taşımalı ve okuyucuya "Aaa! Tabii ki!" dedirtmelidir. Kod yazarken, okuyucunuzu bir hikâyede olduğu gibi yönlendirmelisiniz — her satır bir sonrakini doğal olarak izlemelidir.Grady Booch says clean code should "read like well-written prose." Martin extends this metaphor: like a good novel, clean code should expose the tensions in the problem, build them to a climax, and give the reader an "Aha! Of course!" moment. When writing code, you should guide your reader as in a story — each line should naturally follow the previous one.

Kötü Örnek (Before)Bad Example (Before)

KÖTÜ — Okunamayan "şifreli" kod: hikâye anlatmıyorBAD — Unreadable "cryptic" code: tells no story 
function proc(d, t) {
    const r = [];
    const n = new Date();
    for (let i = 0; i < d.length; i++) {
        const x = d[i];
        if (x.s === 1 && x.e > n) {
            if (t === 1) {
                if (x.p > 100) r.push(x);
            } else if (t === 2) {
                if (x.c >= 4.0) r.push(x);
            }
        }
    }
    r.sort((a, b) => b.p - a.p);
    return r.slice(0, 10);
}
Bu kodu okumak bir bulmacayı çözmek gibi. d, t, x, s, e, p, c — hiçbir değişkenin ne anlama geldiğini bilmiyorsunuz. Kod bir hikâye anlatmıyor; şifre çözmeye zorluyor.Reading this code is like solving a puzzle. d, t, x, s, e, p, c — you have no idea what any variable means. The code doesn't tell a story; it forces you to decrypt.

İYİ — Düzyazı gibi okunur: bir hikâye anlatırGOOD — Reads like prose: tells a story 
function findTopProducts(products, searchStrategy) {
    const activeProducts = products
        .filter(product => product.isActive() && !product.isExpired());
    
    const matchingProducts = searchStrategy.apply(activeProducts);
    
    const topTen = matchingProducts
        .sort(byPriceDescending)
        .slice(0, 10);
    
    return topTen;
}
