Niyet Belirten İsimler Kullanın / Use Intention-Revealing Names Use Intention-Revealing Names
İsimlerin niyeti açığa çıkarması gerektiğini söylemek kolaydır. Size vurgulamak istediğimiz şey, bu konuda ciddi olduğumuzdur. İyi isimler seçmek zaman alır ama kazandırdığı zaman harcananı fazlasıyla karşılar. Bu yüzden isimlerinize özen gösterin ve daha iyisini bulduğunuzda değiştirin. Kodunuzu okuyan herkes (siz dahil) daha mutlu olacaktır. It is easy to say that names should reveal intent. What we want to impress upon you is that we are serious about this. Choosing good names takes time but saves more than it takes. So take care with your names and change them when you find better ones. Everyone who reads your code (including you) will be happier if you do.
Bir değişkenin (variable), fonksiyonun (function) veya sınıfın (class) adı, tüm büyük soruları cevaplamalıdır. Size neden var olduğunu, ne yaptığını ve nasıl kullanıldığını söylemelidir. Eğer bir isim yorum (comment) gerektiriyorsa, o isim niyetini açığa çıkarmıyor demektir. The name of a variable, function, or class, should answer all the big questions. It should tell you why it exists, what it does, and how it is used. If a name requires a comment, then the name does not reveal its intent. 
int d; // elapsed time in days
d ismi hiçbir şey ifade etmez. Geçen zamanı, günleri ya da herhangi bir şeyi çağrıştırmaz. Neyin ölçüldüğünü ve ölçüm birimini belirten bir isim seçmeliyiz: The name d reveals nothing. It does not evoke a sense of elapsed time, nor of days. We should choose a name that specifies what is being measured and the unit of that measurement: 
int elapsedTimeInDays;
int daysSinceCreation;
int daysSinceModification;
int fileAgeInDays;
Niyeti açığa çıkaran isimler seçmek, kodu anlamayı ve değiştirmeyi çok daha kolay hale getirir. Bu kodun amacı nedir? Choosing names that reveal intent can make it much easier to understand and change code. What is the purpose of this code? 
public List<int[]> getThem() {
    List<int[]> list1 = new ArrayList<int[]>();
    for (int[] x : theList)
        if (x[0] == 4)
            list1.add(x);
    return list1;
}
Bu kodun ne yaptığını anlamak neden zor? Karmaşık ifadeler (complex expressions) yok. Boşluklar ve girinti (spacing and indentation) makul. Yalnızca üç değişken ve iki sabit var. Süslü sınıflar veya çok biçimli metotlar (polymorphic methods) bile yok — sadece bir dizi listesi (ya da öyle görünüyor). Why is it hard to tell what this code is doing? There are no complex expressions. Spacing and indentation are reasonable. There are only three variables and two constants. There aren't even any fancy classes or polymorphic methods, just a list of arrays (or so it seems).
Sorun kodun basitliği (simplicity) değil, kodun örtüklüğüdür (implicity — bir ifade türetmek gerekirse): bağlamın (context) kodun içinde açıkça belirtilmemiş olma derecesidir. Kod, örtük olarak şu soruların yanıtlarını bilmemizi gerektirir: The problem isn't the simplicity of the code but the implicity of the code (to coin a phrase): the degree to which the context is not explicit in the code itself. The code implicitly requires that we know the answers to questions such as:
  1. theList içinde ne tür şeyler var? What kinds of things are in theList?
  2. theList içindeki bir öğenin sıfırıncı alt indisinin (zeroth subscript) önemi nedir? What is the significance of the zeroth subscript of an item in theList?
  3. 4 değerinin önemi nedir? What is the significance of the value 4?
  4. Döndürülen listeyi nasıl kullanırım? How would I use the list being returned?

Niyet Belirten İsimler (Intention-Revealing Names)Intention-Revealing Names

Kavram AçıklamasıConcept Explanation

Niyet belirten isimler (Intention-Revealing Names), bir değişkenin, fonksiyonun veya sınıfın adının yalnızca ona bakarak amacını anlamamızı sağlayan isimlerdir. İyi bir isim üç soruyu cevaplamalıdır: Neden var? Ne yapıyor? Nasıl kullanılıyor? Eğer bir ismin yanına yorum eklemeniz gerekiyorsa, isim yeterince iyi değildir.Intention-Revealing Names are names that let us understand the purpose of a variable, function, or class just by looking at it. A good name should answer three questions: Why does it exist? What does it do? How is it used? If you need to add a comment next to a name, the name isn't good enough.

Kötü Örnek (Before)Bad Example (Before)

KÖTÜ — İsimler niyeti gizliyorBAD — Names hide intent 
function process(arr) {
    const res = [];
    for (const item of arr) {
        if (item.s === 1) {
            res.push(item);
        }
    }
    return res;
}

const d = process(data);  // ???
process ne işler? arr neler içerir? s === 1 ne anlama gelir? d ne tutacak? Her bir satır için zihinsel çözümleme yapmak gerekiyor.What does process do? What does arr contain? What does s === 1 mean? What will d hold? You need to mentally decode each line.

İyi Örnek (After)Good Example (After)

İYİ — İsimler niyeti açıkça ortaya koyuyorGOOD — Names clearly reveal intent 
const TASK_STATUS_ACTIVE = 1;

function getActiveTasks(allTasks) {
    const activeTasks = [];
    for (const task of allTasks) {
        if (task.status === TASK_STATUS_ACTIVE) {
            activeTasks.push(task);
        }
    }
    return activeTasks;
}

const activeTasks = getActiveTasks(projectTasks);
Pratik İpucuPractical Tipİsim seçerken "Bu ismi 3 ay sonra tekrar gördüğümde hâlâ anlayacak mıyım?" testini uygulayın. Yorum yazmayı düşündüğünüz her yerde, önce ismi iyileştirmeyi deneyin — yoruma gerek kalmamalı.When choosing a name, apply the "Will I still understand this name 3 months from now?" test. Wherever you're tempted to write a comment, try improving the name first — no comment should be necessary.

Kodun Örtüklüğü (Implicity of Code)Implicity of Code

Kavram AçıklamasıConcept Explanation

Kodun örtüklüğü (implicity), kodun anlaşılması için gerekli bağlamın (context) kodun kendisinde açıkça yer almama derecesidir. Bir kod parçası teknik olarak basit olabilir — az değişken, az satır, karmaşık ifade yok — ama yine de anlaşılmaz olabilir çünkü okuyucunun bilmesi gereken bağlam kodda değil, yazarın kafasındadır.Implicity of code is the degree to which the context needed to understand the code is not explicitly present in the code itself. A piece of code can be technically simple — few variables, few lines, no complex expressions — but still be incomprehensible because the context the reader needs is not in the code, but in the author's head.

Kötü Örnek (Before)Bad Example (Before)

KÖTÜ — Basit ama tamamen örtükBAD — Simple but completely implicit 
def check(items):
    return [i for i in items if i[2] > 30 and i[0] == 'X']
Kod bir satır ve "basit" görünüyor ama: items nedir? i[2] ne anlama gelir? 30 neyin sınırı? 'X' ne durumu ifade eder? Tüm bağlam örtüktür.The code is one line and looks "simple" but: What are items? What does i[2] mean? What's the threshold 30? What does 'X' represent? All context is implicit.

İyi Örnek (After)Good Example (After)

İYİ — Bağlam açıkça kodda yer alıyorGOOD — Context is explicit in the code 
MINIMUM_AGE_DAYS = 30
STATUS_EXPIRED = 'X'

def find_expired_old_records(records):
    return [
        record for record in records
        if record.age_in_days > MINIMUM_AGE_DAYS
        and record.status == STATUS_EXPIRED
    ]
Pratik İpucuPractical TipBir fonksiyon yazdıktan sonra, kodunuzu bir "yabancı gözüyle" okuyun. Eğer herhangi bir satırda "Bu ne demek?" sorusu akla geliyorsa, o bilgi açıkça (explicit) kodda yer almalıdır. Sihirli sayıları (magic numbers) adlandırılmış sabitlerle, indisleri anlamlı alanlarla (fields) değiştirin.After writing a function, read your code with "stranger's eyes." If the question "What does this mean?" comes to mind at any line, that information should be explicit in the code. Replace magic numbers with named constants, indices with meaningful fields.
