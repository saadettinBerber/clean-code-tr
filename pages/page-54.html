Yorumlar Neden Yalan Söyler? / Why Comments Lie Why Comments Lie
Yorumların doğru kullanımı, kendimizi kodla ifade etmedeki başarısızlığımızı telafi etmektir. "Başarısızlık" kelimesini bilinçli olarak kullandığıma dikkat edin. Gerçekten bunu kastediyorum. Yorumlar her zaman bir başarısızlıktır. Onlara ihtiyaç duyarız, çünkü kendimizi onlarsız nasıl ifade edeceğimizi her zaman bulamayız; ancak onların varlığı kutlanacak bir şey değildir. The proper use of comments is to compensate for our failure to express ourself in code. Note that I used the word failure. I meant it. Comments are always failures. We must have them because we cannot always figure out how to express ourselves without them, but their use is not a cause for celebration.
Öyleyse kendinizi yorum yazmak zorunda hissettiğinizde, duraksayın ve şunu düşünün: kendimi kodun içinde ifade etmenin bir yolu yok mu? Her seferinde kodu konuşturduğunuzda kendinizi tebrik edebilirsiniz. Her yorum yazdığınızda ise yüzünüzü ekşitip ifade yetersizliğinizi kabullenmek durumundasınız. So when you find yourself in a position where you need to write a comment, think it through and see whether there isn't some way to turn the tables and express yourself in code. Every time you express yourself in code, you should pat yourself on the back. Every time you write a comment, you should grimace and feel the failure of your ability of expression.
Peki neden yorumlara bu kadar karşıyım? Çünkü onlar yalan söyler. Her zaman değil, kasıtlı olarak da değil; ama çok sık yalan söylerler. Bir yorum ne kadar eskiyse ve açıkladığı koddan ne kadar uzaktaysa, yanlış olma ihtimali o kadar yüksektir. Sebep basittir: programcılar onları gerçekçi bir şekilde güncel tutamazlar. Why am I so down on comments? Because they lie. Not always, and not intentionally, but too often. The older a comment is, and the farther away it is from the code it describes, the more likely it is to be just plain wrong. The reason is simple. Programmers can't realistically maintain them.
Kod değişir ve evrim geçirir. Parçalar bir yerden başka bir yere taşınır. O parçalar ikiye bölünür, çoğalır ve tekrar bir araya gelerek melez yapılar oluşturur. Ne yazık ki yorumlar onlara her zaman eşlik edemez — hiç edemezler de. Çoğu zaman yorumlar, açıkladıkları koddan kopar ve giderek azalan doğrulukla asılsız açıklamalar haline gelirler. Örneğin şu yoruma ve açıklanmak istediği satıra bakın: Code changes and evolves. Chunks of it move from here to there. Those chunks bifurcate and reproduce and come together again to form chimeras. Unfortunately the comments don't always follow them—can't always follow them. And all too often the comments get separated from the code they describe and become orphaned blurbs of ever-decreasing accuracy. For example, look what has happened to this comment and the line it was intended to describe:
Yorum ile kodun birbirinden uzaklaşması örneği Java 
MockRequest request;
private final String HTTP_DATE_REGEXP =
    "[SMTWF][a-z]{2}\\,\\s[0-9]{2}\\s[JFMASOND][a-z]{2}\\s"+
    "[0-9]{4}\\s[0-9]{2}\\:[0-9]{2}\\:[0-9]{2}\\sGMT";
private Response response;
private FitNesseContext context;
private FileResponder responder;
private Locale saveLocale;
// Example: "Tue, 02 Apr 2003 22:18:49 GMT"
Sonradan eklenen diğer örnek değişkenler, HTTP_DATE_REGEXP sabiti ile ona ait açıklama yorumu arasına sıkışmıştır. Yorum artık doğrudan kendi açıkladığı satırın yanında değildir. Other instance variables that were probably added later were interposed between the HTTP_DATE_REGEXP constant and its explanatory comment.
Programcıların yorumları yüksek bir doğruluk ve alaka düzeyinde tutacak kadar disiplinli olması gerektiği söylenebilir. Buna katılıyorum — olmaları gerekir. Ama o enerjiyi yorumları güncel tutmak yerine, kodu baştan yoruma gerek duymayacak kadar açık ve ifade edici yazmaya harcamayı tercih ederim. It is possible to make the point that programmers should be disciplined enough to keep the comments in a high state of repair, relevance, and accuracy. I agree, they should. But I would rather that energy go toward making the code so clear and expressive that it does not need the comments in the first place.
Hatalı yorumlar, hiç yorum olmamaktan çok daha kötüdür. Yanıltırlar ve aldatırlar. Hiçbir zaman karşılanmayacak beklentiler yaratırlar. Artık geçerli olmayan eski kuralları dayatırlar. Inaccurate comments are far worse than no comments at all. They delude and mislead. They set expectations that will never be fulfilled. They lay down old rules that need not, or should not, be followed any longer.

Gerçek yalnızca tek bir yerde bulunabilir: kodun içinde. Yalnızca kod ne yaptığını gerçekten anlatır. Codun gerçek anlamda doğru bilginin tek kaynağıdır. Bu yüzden, yorumlar zaman zaman zorunlu olsa da, onları en aza indirmek için önemli bir çaba harcamalıyız. Truth can only be found in one place: the code. Only the code can truly tell you what it does. It is the only source of truly accurate information. Therefore, though comments are sometimes necessary, we will expend significant energy to minimize them.

Yorumlar Bir Başarısızlıktır (Comments as Failure)Comments as Failure

Kavram AçıklamasıConcept Explanation

Martin'e göre her yorum, programcının niyet ve mantığı kod aracılığıyla aktaramamasının bir itirafıdır. Bu bazen kaçınılmazdır, ama asla kutlanmamalıdır. "Yorum ekledim, ne iyi!" yerine "Yoruma ihtiyaç duymamak için kodu nasıl düzeltebilirim?" sorusu sorulmalıdır. According to Martin, every comment is an admission that the programmer could not express their intent and reasoning through code. This is sometimes unavoidable, but should never be celebrated. Instead of "I added a comment, great!" ask "How can I fix the code so I don't need the comment?"

Kötü Örnek (Before)Bad Example (Before)

KÖTÜ — Yorum kodu kurtarmaya çalışıyorBAD — Comment trying to rescue code 
// d = geçen gün sayısı
int d;

// görev bitiş tarihinden bugüne kaç gün geçtiğini hesapla
d = (endDate - today) / 86400;

İyi Örnek (After)Good Example (After)

İYİ — Kod kendini açıklıyorGOOD — Code explains itself 
int daysSinceTaskDeadline = (endDate - today) / SECONDS_PER_DAY;
Pratik İpucuPractical Tip Bir yorum yazmak üzereyken şunu sorun: "Bu yorumu bir değişken adı veya metot adıyla kodun içine gömebilir miyim?" Çoğunlukla yanıt "evet"tir. Before writing a comment, ask: "Can I embed this explanation into the code as a variable or method name?" Most of the time, the answer is yes.

Yalan Söyleyen Yorumlar (Lying Comments)Lying Comments

Kavram AçıklamasıConcept Explanation

Yorumlar zamanla koddan kopar, güncel tutulmaz ve yanlış bilgi içermeye başlar. Kasıtlı olmasa da bu "yalan söyleme" durumu okuyucuyu yanıltır. Eski bir yorum, anlayan olmayan bir kodu anlatmak yerine tam tersi bir beklenti yaratabilir. Comments drift away from code over time, are not kept up to date, and begin to contain false information. Although unintentional, this "lying" misleads readers. An old comment can create the exact opposite expectation rather than clarifying unclear code.

Kötü Örnek (Before)Bad Example (Before)

KÖTÜ — Yorum artık doğru değilBAD — Comment is no longer accurate 
// Kullanıcıyı e-posta ile bilgilendir ve kaydı sil
public void deactivateUser(User user) {
    // E-posta gönderimi kaldırıldı (artık webhook var)
    userRepository.delete(user); // ama yorum hâlâ e-posta diyor!
}

İyi Örnek (After)Good Example (After)

İYİ — Yorum kaldırıldı, kod konuşuyorGOOD — No comment, code speaks 
public void deactivateUser(User user) {
    webhookService.notifyDeactivation(user);
    userRepository.delete(user);
}
Pratik İpucuPractical Tip Kod incelemelerinde (code review) yorumları da gözden geçirin. Değişen kodla çelişen bir yorum görürseniz, önce yorumu silin — kodu düzeltmek ikinci adım. During code reviews, review comments too. If you see a comment that contradicts the changed code, delete the comment first — fixing the code is the second step.

Gerçek Yalnızca Kodda (Truth in Code)Truth in Code

Kavram AçıklamasıConcept Explanation

Martin'in en güçlü tespitlerinden biri: sistemin o andaki gerçek davranışını yalnızca kaynak kodu yansıtır. Dokümanlar, yorumlar ve wiki sayfaları güncel olmayabilir. Ancak çalışan kod asla yalan söylemez — yaptığı şeyi yapar. One of Martin's most powerful observations: only the source code reflects the true current behavior of the system. Documentation, comments, and wiki pages may be outdated. But running code never lies — it does exactly what it does.

Kötü Örnek (Before)Bad Example (Before)

KÖTÜ — Dokümana güvenmekBAD — Trusting documentation 
// Wiki: calculateTax() tüm ülkeleri destekler
// (ama gerçekte sadece TR ve US için yazılmış)
double tax = calculateTax(order, "DE"); // sonuç yanlış!

İyi Örnek (After)Good Example (After)

İYİ — Kodu oku, kendini açıklasınGOOD — Read the code, let it explain itself 
public double calculateTax(Order order, String countryCode) {
    if (!SUPPORTED_COUNTRIES.contains(countryCode))
        throw new UnsupportedCountryException(countryCode);
    // desteklenen ülkeler açıkça koda gömülü
    ...
}
Pratik İpucuPractical Tip Bir davranışı anlamak istediğinizde her zaman önce kodu okuyun, sonra yorumları. Çelişki varsa koda inanın — o sistemi çalıştıran tek gerçektir. When you want to understand behavior, always read the code first, then the comments. If there is a conflict, believe the code — it is the only truth running the system.
