Ward Cunningham'ın bana öğrettiği şeylerden biri şuydu: İsim verdiğinizde kodu anlamış olursunuz. Tersi de doğrudur: İsim verdiğinizde kodun ne yapması gerektiğini anlarsınız. İyi bir isim seçmek zordur çünkü iyi bir isim seçmek için kodun yapısını iyi anlamanız gerekir ve bu da kodun iyi yapılandırılmasını gerektirir. Eğer isim vermekte zorlanıyorsanız, kodun yapısını daha iyi hale getirmenin zamanı gelmiştir. One of the things that Ward Cunningham taught me was that when you name something, you understand it. The reverse is also true: When you name something, you understand what it should do. Choosing good names is hard because choosing good names requires you to understand the structure of the code well, and that requires the code to be well structured. If you're having trouble naming something, it's time to improve the structure of the code.
Bu kuralın istisnası test kodudur. Test kodu için açıklayıcı isimlerden vazgeçmeyin. Test fonksiyonlarının isimleri, test edilen kuralın ve koşulların açıklaması olmalıdır. Test fonksiyonu isimleri çok uzun olabilir. Örneğin: testUserCanAddItemToCartWhenLoggedInAndCartIsEmpty gibi bir isim tamamen kabul edilebilirdir. Testlerde okunabilirlik her şeyden önemlidir. The exception to this rule is test code. Don't skimp on descriptive names for test code. Test function names should be descriptions of the rule and conditions being tested. Test function names can be very long. For example: testUserCanAddItemToCartWhenLoggedInAndCartIsEmpty is perfectly acceptable. Readability is paramount in tests.
İsimlendirme kuralları Bölüm 2'de ele alındığı için burada tekrar etmeyeceğim. Ancak şunu vurgulamak istiyorum: Tutarlı bir kelime haznesi (consistent lexicon) kullanın. Fonksiyonlarınızda tutarlı bir kelime haznesi kullanın. Aynı şeyi yapmak için farklı kelimeler kullanmayın. Örneğin, fetch, retrieve ve get kelimelerini rastgele karıştırmayın. Aynı şekilde controller, manager ve driver kelimelerini de karıştırmayın. Tutarlılık hayati önem taşır. Since naming conventions were covered in Chapter 2, I won't repeat them here. But I want to emphasize: Use a consistent lexicon. Use a consistent vocabulary in your functions. Don't use different words for the same thing. For example, don't randomly mix fetch, retrieve, and get. Similarly, don't mix controller, manager, and driver. Consistency is vital.
Aynı isimleri farklı amaçlar için kullanmayın. Örneğin, bir yerde add kelimesini birbiriyle bağlama (concatenation) anlamında kullanırsanız, başka bir yerde matematiksel toplama anlamında kullanamazsınız. Çünkü bu durumda add kelimesi farklı şeyler ifade eder ve okuyucular kafası karışır. Bu bir kelime oyunu (pun) örneğidir ve kaçınılması gerekir. Don't use the same names for different purposes. For example, if you use the word add to mean concatenation in one place, you can't use it to mean mathematical addition somewhere else. Because in that case, the word add means different things and readers get confused. This is an example of a pun and should be avoided.
Fonksiyon Argümanları / Function Arguments Function Arguments
Bir fonksiyon için ideal argüman sayısı sıfırdır (niladic). Sonra bir (monadic) gelir, ardından iki (dyadic). Üç argüman (triadic) mümkünse kaçınılmalıdır. Üçten fazla argüman (polyadic) çok özel bir gerekçe gerektirir - ve yine de kullanılmamalıdır. The ideal number of arguments for a function is zero (niladic). Next comes one (monadic), followed by two (dyadic). Three arguments (triadic) should be avoided where possible. More than three arguments (polyadic) requires very special justification - and then shouldn't be used anyway.
Argümanlar, test edilebilirlik açısından da zordur. Argümanların tüm olası kombinasyonlarının doğru çalıştığından emin olmak için test senaryoları yazmanın zorluğunu hayal edin. Argüman sayısı arttıkça, test kombinasyonları üstel olarak artar. Arguments are hard from a testing point of view as well. Imagine the difficulty of writing test cases to ensure that all various combinations of arguments work properly. As the number of arguments increases, test combinations grow exponentially.
Çıkış argümanları (output arguments), giriş argümanlarından (input arguments) daha zor anlaşılır. Bir fonksiyonu okurken, bilginin argümanlar aracılığıyla fonksiyona girdiğini ve dönüş değeri aracılığıyla çıktığını varsayarız. Genellikle bilginin argümanlar aracılığıyla çıktığını beklemeyiz. Output arguments are harder to understand than input arguments. When we read a function, we assume information goes into the function through arguments and out through the return value. We don't usually expect information to be going out through the arguments.
Fonksiyonlarınızda sıfır argüman kullanmak her zaman mümkün olmayabilir, ancak bunu hedeflemelisiniz. Çok argümanlı fonksiyonlar genellikle çok fazla şey yapmaya çalışıyor demektir. Bir fonksiyonun çok sayıda argümana ihtiyaç duyması, fonksiyonun tek bir şey yapmadığının işaretidir. It may not always be possible to use zero arguments in your functions, but you should aim for it. Functions with many arguments usually mean they're trying to do too much. If a function needs many arguments, it's a sign that the function is not doing one thing.
Argüman sayısını azaltmanın bir yolu, argümanları bir nesne içinde gruplamaktır. Örneğin, Circle makeCircle(int x, int y, int radius) yerine Circle makeCircle(Point center, int radius) yazabilirsiniz. Bu, argüman sayısını üçten ikiye düşürür ve aynı zamanda kavramsal bütünlüğü artırır. One way to reduce the number of arguments is to group arguments into an object. For example, instead of Circle makeCircle(int x, int y, int radius) you can write Circle makeCircle(Point center, int radius). This reduces the argument count from three to two and also improves conceptual integrity.

Tutarlı Kelime Haznesi (Consistent Lexicon)Consistent Lexicon

Kavram AçıklamasıConcept Explanation

Aynı şeyi yapmak için farklı kelimeler kullanmayın. Kod tabanınızda tutarlı bir dil (lexicon) oluşturun. fetch, retrieve, get kelimelerini rastgele karıştırmayın. Controller, manager, driver kelimelerini tutarlı kullanın.Don't use different words for the same thing. Create a consistent language (lexicon) in your codebase. Don't randomly mix fetch, retrieve, get. Use controller, manager, driver consistently.

Kötü Örnek (Before)Bad Example (Before)

KÖTÜ — Tutarsız kelimelerBAD — Inconsistent words 
UserService.fetchUser(id)      # fetch
OrderService.retrieveOrder(id)  # retrieve
ProductService.getProduct(id)   # get - Aynı şey!

UserController # controller
OrderManager   # manager
ProductDriver  # driver - Ne farkı var?

İyi Örnek (After)Good Example (After)

İYİ — Tutarlı kelime haznesiGOOD — Consistent lexicon 
UserService.get(id)      # get - Hepsi get
OrderService.get(id)     # get
ProductService.get(id)   # get

UserController   # controller - HTTP istekleri
OrderController  # controller
ProductController # controller
Pratik İpucuPractical TipTakımınızla bir "sözlük" (lexicon) oluşturun. Her kavram için bir kelime seçin ve hep onu kullanın. Yeni geliştiricilere bu sözlüğü öğretin.Create a "lexicon" with your team. Choose one word for each concept and always use it. Teach this lexicon to new developers.

Argüman Sayısı (Argument Count)Argument Count

Kavram AçıklamasıConcept Explanation

İdeal: 0 argüman (niladic). Kabul edilebilir: 1 (monadic) veya 2 (dyadic). Kaçınılması gerekir: 3 (triadic). Kullanılmamalı: 3+ (polyadic). Çok argüman fonksiyonlar test edilmesi zor ve anlaşılması güçtür.Ideal: 0 arguments (niladic). Acceptable: 1 (monadic) or 2 (dyadic). Should be avoided: 3 (triadic). Should not be used: 3+ (polyadic). Many-argument functions are hard to test and understand.

Kötü Örnek (Before)Bad Example (Before)

KÖTÜ — Çok fazla argümanBAD — Too many arguments 
def createUser(
    firstName, lastName, email, phone,
    address, city, country, postalCode,
    isActive, isVerified, role
):  # 11 argüman!
    ...

İyi Örnek (After)Good Example (After)

İYİ — Nesne ile gruplamaGOOD — Grouped into object 
def createUser(userData):
    # Tek argüman - bir nesne
    ...

# Veya daha da iyi:
def createUser(name, contact, address, status):
    # 4 argüman - her biri bir nesne
Pratik İpucuPractical Tip3+ argüman gördüğünüzde, bunları bir nesne içinde gruplayıp gruplayemeyeceğinizi düşünün. "Bu argümanlar birlikte mi kullanılıyor?" sorusunu sorun.When you see 3+ arguments, consider if they can be grouped into an object. Ask: "Are these arguments used together?"

Argüman Nesneleri (Argument Objects)Argument Objects

Kavram AçıklamasıConcept Explanation

Birden fazla argüman bir kavramın parçasıysa, onları bir nesne içinde gruplayın. Bu hem argüman sayısını azaltır hem de kodun anlaşılırlığını artırır. Örneğin x, y koordinatları bir Point nesnesi olabilir.When multiple arguments are part of a concept, group them into an object. This reduces argument count and improves code clarity. For example, x, y coordinates can be a Point object.

Kötü Örnek (Before)Bad Example (Before)

KÖTÜ — Ayrı koordinatlarBAD — Separate coordinates 
def drawCircle(centerX, centerY, radius,
              colorR, colorG, colorB):
    # 6 argüman - karmaşık!
    ...

# Kullanım:
drawCircle(10, 20, 5, 255, 0, 0)
# Hangisi x? Hangisi renk?

İyi Örnek (After)Good Example (After)

İYİ — Kavramsal nesnelerGOOD — Conceptual objects 
def drawCircle(center, radius, color):
    # 3 argüman - her biri bir nesne
    ...

# Kullanım:
drawCircle(
    Point(10, 20),
    Radius(5),
    Color(255, 0, 0)
)
# Açık ve anlaşılır
Pratik İpucuPractical TipArgümanlar birlikte geziyorsa (her zaman birlikte kullanılıyorsa), bir nesne yapın. Bu aynı zamanda validation mantığını da nesneye taşımanızı sağlar.If arguments travel together (always used together), make them an object. This also allows you to move validation logic into the object.
