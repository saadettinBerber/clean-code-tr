Keskin Soyutlama (Crisp Abstraction)Crisp Abstraction

Kavram AçıklamasıConcept Explanation

Grady Booch'un "keskin soyutlama" (crisp abstraction) ifadesi büyüleyici bir oksimorondur: "keskin" (crisp) somutluğu çağrıştırırken, "soyutlama" (abstraction) tam tersini ifade eder. Martin'e göre bu ifade güçlü bir mesaj taşır: kodumuz olgusal (matter-of-fact) olmalı, spekülatif değil. Yalnızca gerekli olanı içermeli. Kararlı (decisive) olmalı — tereddüt ve gereksiz detay barındırmamalı. İyi bir soyutlama, somut bir biçimde sunulmalıdır.Grady Booch's phrase "crisp abstraction" is a fascinating oxymoron: "crisp" evokes concreteness, while "abstraction" means the opposite. According to Martin, this carries a powerful message: our code should be matter-of-fact, not speculative. It should contain only what's necessary. It should be decisive — no hesitation, no unnecessary detail. A good abstraction should be presented in a concrete, clear manner.

Kötü Örnek (Before)Bad Example (Before)

KÖTÜ — Bulanık soyutlama: spekülatif, tereddütlü, gereksiz detayla doluBAD — Fuzzy abstraction: speculative, hesitant, full of unnecessary detail 
class AbstractDataProcessorStrategyFactoryBase:
    """
    Bu sınıf gelecekte farklı veri işleme stratejileri
    için kullanılabilecek bir temel fabrika sağlar.
    Şimdilik sadece CSV destekleniyor ama JSON, XML,
    YAML ve belki Parquet de eklenebilir.
    """
    def maybe_process(self, data, options=None, context=None,
                      retry_count=3, fallback=None, 
                      enable_logging=True, dry_run=False):
        if data is not None:
            if options is not None:
                if context is not None:
                    if not dry_run:
                        return self._do_process_internal(data)
                    else:
                        return self._simulate(data)
                else:
                    return self._do_process_internal(data)
            else:
                return self._do_process_internal(data)
        return fallback
Burada her şey spekülatif ve tereddütlü. Sınıf adı ("maybe_process", "AbstractDataProcessorStrategyFactoryBase") soyutlamayı bulanıklaştırıyor. Kullanılmayan parametreler (retry_count, enable_logging) "belki lazım olur" zihniyetiyle eklenmiş. İç içe if'ler kararsızlığı yansıtıyor.Everything here is speculative and hesitant. The class name ("maybe_process", "AbstractDataProcessorStrategyFactoryBase") makes the abstraction fuzzy. Unused parameters (retry_count, enable_logging) were added with a "might need it" mentality. Nested ifs reflect indecisiveness.

İyi Örnek (After)Good Example (After)

İYİ — Keskin soyutlama: kararlı, olgusal, yalnızca gerekli olanı içerirGOOD — Crisp abstraction: decisive, matter-of-fact, contains only what's necessary 
class CsvParser:
    def parse(self, content: str) -> list[dict]:
        rows = content.strip().splitlines()
        headers = rows[0].split(',')
        return [
            dict(zip(headers, row.split(',')))
            for row in rows[1:]
        ]

class JsonParser:
    def parse(self, content: str) -> list[dict]:
        return json.loads(content)
Pratik İpucuPractical Tip"Keskin" soyutlama testi: sınıfınızın veya fonksiyonunuzun adını söylediğinizde, ne yaptığı hemen anlaşılmalıdır. "CsvParser.parse(content)" → "CSV içeriğini ayrıştır." Eğer ismi açıklamak için iki cümle gerekiyorsa, soyutlamanız yeterince keskin değildir. "Maybe", "Abstract", "Base", "Manager", "Helper" gibi bulanık kelimelerden kaçının."Crisp" abstraction test: when you say your class or function name, what it does should be immediately clear. "CsvParser.parse(content)" → "Parse CSV content." If you need two sentences to explain the name, your abstraction isn't crisp enough. Avoid fuzzy words like "Maybe", "Abstract", "Base", "Manager", "Helper".

Testsiz Kod Kirlidir (Code Without Tests Is Unclean)Code Without Tests Is Unclean

Kavram AçıklamasıConcept Explanation

"Big" Dave Thomas temizliği doğrudan testlere bağlar: "Testleri olmayan kod temiz değildir." Martin bunu güçlendirir: "Ne kadar zarif olursa olsun, ne kadar okunabilir olursa olsun, eğer testleri yoksa, kirlidir." Test Güdümlü Geliştirme (TDD) disiplini endüstriyi dönüştürmüştür. Testler yalnızca hataları bulmak için değil, kodun değiştirilebilir olmasını garanti altına almak için vardır — çünkü okunması kolay ama değiştirilmesi riskli kod, gerçekten temiz değildir."Big" Dave Thomas ties cleanliness directly to tests: "Code without tests is not clean." Martin reinforces this: "No matter how elegant, no matter how readable, if it hath not tests, it be unclean." The discipline of Test Driven Development (TDD) has transformed the industry. Tests exist not just to find bugs, but to guarantee that code is changeable — because code that's easy to read but risky to change isn't truly clean.

Kötü Örnek (Before)Bad Example (Before)

KÖTÜ — "Zarif" ama testsiz: değiştirmeye cesaret edilemezBAD — "Elegant" but untested: nobody dares to change it 
class ShippingCalculator {
    calculateCost(weight, destination, isExpress) {
        const baseRate = destination.startsWith('EU') ? 5.0 : 12.0;
        const weightFactor = Math.ceil(weight / 0.5) * 1.5;
        const expressSurcharge = isExpress ? baseRate * 0.75 : 0;
        
        return baseRate + weightFactor + expressSurcharge;
    }
}

// Güzel görünüyor ama...
// - Bu formül doğru mu? Kim doğrulayacak?
// - "EU" kontrolü yeterli mi? İngiltere ne olacak?
// - 0.5 ve 1.5 nereden geliyor? Değişirse?
// - İş kuralları değiştiğinde kim fark edecek?
Kod kısa ve okunabilir görünüyor — ama testleri yok. Formülün doğruluğunu hiçbir şey garanti etmiyor. İş kuralları değiştiğinde, kimse mevcut hesaplamanın bozulup bozulmadığını bilemez. "Zarif ama testsiz" = "kirli."The code looks short and readable — but has no tests. Nothing guarantees the formula is correct. When business rules change, nobody can tell if existing calculations break. "Elegant but untested" = "unclean."

İyi Örnek (After)Good Example (After)

İYİ — Testlerle korunan kod: güvenle değiştirilebilirGOOD — Code protected by tests: can be changed with confidence 
describe('ShippingCalculator', () => {
    it('applies EU base rate for European destinations', () => {
        expect(calculator.calculateCost(1.0, 'EU-DE', false))
            .toBe(5.0 + 3.0);
    });

    it('applies international rate for non-EU destinations', () => {
        expect(calculator.calculateCost(1.0, 'US-NY', false))
            .toBe(12.0 + 3.0);
    });

    it('adds 75% express surcharge on base rate', () => {
        expect(calculator.calculateCost(1.0, 'EU-FR', true))
            .toBe(5.0 + 3.0 + 3.75);
    });

    it('rounds weight up to nearest 0.5kg increment', () => {
        expect(calculator.calculateCost(0.3, 'EU-DE', false))
            .toBe(5.0 + 1.5);
        expect(calculator.calculateCost(0.7, 'EU-DE', false))
            .toBe(5.0 + 3.0);
    });
});
Pratik İpucuPractical TipTDD döngüsünü uygulayın: (1) Başarısız bir test yaz (Red), (2) Testi geçirecek minimum kodu yaz (Green), (3) Kodu temizle (Refactor). Bu döngü sayesinde kodunuz her zaman test kapsamındadır. Dave Thomas'ın dediği gibi: testleri olmayan kod, ne kadar güzel olursa olsun, temiz değildir.Apply the TDD cycle: (1) Write a failing test (Red), (2) Write minimal code to pass (Green), (3) Clean up (Refactor). With this cycle, your code always has test coverage. As Dave Thomas says: code without tests, no matter how beautiful, is not clean.

Okuryazar Programlama (Literate Programming)Literate Programming

Kavram AçıklamasıConcept Explanation

"Big" Dave Thomas, kodun "okuryazar" (literate) olması gerektiğini söyler — bu, Donald Knuth'un "okuryazar programlama" (literate programming) kavramına bir referanstır. Knuth'a göre programlar önce insanlar için, sonra bilgisayarlar için yazılmalıdır. Dave, dile bağlı olarak tüm gerekli bilginin yalnızca kodda açıkça ifade edilemeyeceğini vurgular. Kod, insanlar tarafından okunabilecek bir biçimde yazılmalıdır — bir makineye talimat vermekten öte, bir insana hikâye anlatmalıdır."Big" Dave Thomas says code should be "literate" — a reference to Donald Knuth's "literate programming" concept. According to Knuth, programs should be written primarily for humans, then for computers. Dave emphasizes that depending on the language, not all necessary information can be expressed clearly in code alone. Code should be composed in a form that makes it readable by humans — beyond giving instructions to a machine, it should tell a story to a person.

Kötü Örnek (Before)Bad Example (Before)

KÖTÜ — Makine için yazılmış: insan için okunamazBAD — Written for machines: unreadable for humans 
public List<int[]> getThem() {
    List<int[]> list1 = new ArrayList<>();
    for (int[] x : theList) {
        if (x[0] == 4)
            list1.add(x);
    }
    return list1;
}
Bu kod derlenebilir ve çalışır — ama ne yaptığını anlamak imkânsızdır. theList ne? x[0] ne anlama geliyor? 4 ne demek? Kod makineye talimat veriyor ama insana hiçbir şey anlatmıyor. "Okuryazar" değil.This code compiles and runs — but understanding what it does is impossible. What's theList? What does x[0] mean? What does 4 represent? The code instructs the machine but tells the human nothing. Not "literate."

İyi Örnek (After)Good Example (After)

İYİ — İnsan için yazılmış: kod kendi hikâyesini anlatırGOOD — Written for humans: code tells its own story 
public List<Cell> getFlaggedCells() {
    List<Cell> flaggedCells = new ArrayList<>();
    for (Cell cell : gameBoard) {
        if (cell.isFlagged())
            flaggedCells.add(cell);
    }
    return flaggedCells;
}
Pratik İpucuPractical TipKodunuzu okuryazar yapmak için şu kuralı uygulayın: her değişken, fonksiyon ve sınıf adı bir niyet ifade etmelidir. "theList" yerine "gameBoard", "x[0] == 4" yerine "cell.isFlagged()" yazın. Dave Thomas'ın dediği gibi: tüm gerekli bilgi kodda açıkça ifade edilemeyebilir, ama ifade edebildiğiniz kadarını mümkün olduğunca net ifade edin.To make code literate, apply this rule: every variable, function, and class name should express an intent. Write "gameBoard" instead of "theList", "cell.isFlagged()" instead of "x[0] == 4". As Dave Thomas says: not all necessary information can be expressed in code alone, but express what you can as clearly as possible.
