Sonuçlara Dair Uyarı / Warning of Consequences Warning of Consequences
Bazen diğer programcıları belirli sonuçlar konusunda uyarmak faydalıdır. Örneğin şu yorum, belirli bir test durumunun neden devre dışı bırakıldığını açıklar: Sometimes it is useful to warn other programmers about certain consequences. For example, here is a comment that explains why a particular test case is turned off:
Devre dışı bırakılmış test — uyarı yorumuDisabled test — warning comment
// Biraz zamanınız olmadıkça çalıştırmayın.
public void _testWithReallyBigFile()
{
  writeLinesToFile(10000000);
  response.setBody(testFile);
  response.readyToSend(this);
  String responseString = output.toString();
  assertSubString("Content-Length: 1000000000", responseString);
  assertTrue(bytesSent > 1000000000);
}
Bugün elbette @Ignore anotasyonunu uygun bir açıklama dizisiyle — @Ignore("Çalışması çok uzun sürer") gibi — kullanarak test durumunu devre dışı bırakırdık. Ama JUnit 4 öncesi günlerde metot adının başına alt çizgi eklemek yaygın bir gelenekti. Yorum, biraz alaycı olsa da mesajı gayet iyi aktarıyor. Nowadays, of course, we'd turn off the test case by using the @Ignore attribute with an appropriate explanatory string. @Ignore("Takes too long to run"). But back in the days before JUnit 4, putting an underscore in front of the method name was a common convention. The comment, while flippant, makes the point pretty well.
İşte daha çarpıcı bir örnek: Here's another, more poignant example:
Thread güvenliği uyarısıThread safety warning
public static SimpleDateFormat makeStandardHttpDateFormat()
{
  //SimpleDateFormat thread-safe değildir,
  //bu yüzden her örneği bağımsız oluşturmamız gerekir.
  SimpleDateFormat df = new SimpleDateFormat("EEE, dd MMM  yyyy HH:mm:ss z");
  df.setTimeZone(TimeZone.getTimeZone("GMT"));
  return df;
}
Bu problemi çözmenin daha iyi yolları olduğunu söyleyebilirsiniz. Belki haklısınız. Ama buradaki yorum son derece makuldür. Aşırı hevesli bir programcının verimliliği için statik bir başlatıcı kullanmasını önler. You might complain that there are better ways to solve this problem. I might agree with you. But the comment, as given here, is perfectly reasonable. It will prevent some overly eager programmer from using a static initializer in the name of efficiency.
TODO Yorumları / TODO Comments TODO Comments
Zaman zaman //TODO biçiminde "Yapılacaklar" notları bırakmak makul olabilir. Aşağıdaki örnekte TODO yorumu, fonksiyonun neden çürük bir uygulamaya sahip olduğunu ve gelecekte ne olması gerektiğini açıklar: It is sometimes reasonable to leave "To do" notes in the form of //TODO comments. In the following case, the TODO comment explains why the function has a degenerate implementation and what that function's future should be.
TODO yorum örneğiTODO comment example
//TODO-MdM bunlara gerek yok
// Ödeme modelini yaptığımızda bunun ortadan kalkmasını bekliyoruz
protected VersionInfo makeVersion() throws Exception
{
  return null;
}
TODO'lar, programcının yapılması gerektiğini düşündüğü ama şu an yapamadığı işlerdir. Kullanımdan kaldırılmış bir özelliği silme hatırlatması, başkasının bir probleme bakmasını isteme talebi, birisinin daha iyi bir isim bulmasını isteme ricası ya da planlanan bir olaya bağlı olarak yapılacak değişiklik hatırlatıcısı olabilir. Ne olursa olsun, TODO sisteme kötü kod bırakmak için bir mazeret değildir. TODOs are jobs that the programmer thinks should be done, but for some reason can't do at the moment. It might be a reminder to delete a deprecated feature or a plea for someone else to look at a problem. It might be a request for someone else to think of a better name or a reminder to make a change that is dependent on a planned event. Whatever else a TODO might be, it is not an excuse to leave bad code in the system.
Bugün çoğu iyi IDE, tüm TODO yorumlarını bulmak için özel hareketler ve özellikler sunar; bu yüzden kaybolma olasılıkları düşüktür. Yine de kodunuzun TODO ile dolu olmasını istemezsiniz. Bu yüzden düzenli aralıklarla TODO'ları tarayın ve kaldırabileceklerinizi temizleyin. Nowadays, most good IDEs provide special gestures and features to locate all the TODO comments, so it's not likely that they will get lost. Still, you don't want your code to be littered with TODOs. So scan through them regularly and eliminate the ones you can.
Önem Vurgulama / Amplification Amplification
Bir yorum, başka türlü önemsiz görünebilecek bir şeyin önemini vurgulamak için kullanılabilir: A comment may be used to amplify the importance of something that may otherwise seem inconsequential.
trim() çağrısının önemi vurgulanıyorImportance of trim() call amplified
String listItemContent = match.group(3).trim();
// trim gerçekten önemli. Elemanın başındaki
// boşlukları kaldırır; aksi halde öğe başka bir
// liste olarak tanınabilir.
new ListItemWidget(this, listItemContent, this.level + 1);
return buildList(text.substring(match.end()));
Herkese Açık API'lerde Javadoc / Javadocs in Public APIs Javadocs in Public APIs
İyi belgelenmiş herkese açık bir API kadar yararlı ve tatmin edici hiçbir şey yoktur. Standart Java kütüphanesinin javadoc'ları bunun en somut örneğidir. Onlarsız Java programları yazmak — en iyi ihtimalle — son derece güç olurdu. There is nothing quite so helpful and satisfying as a well-described public API. The javadocs for the standard Java library are a case in point. It would be difficult, at best, to write Java programs without them.
Herkese açık bir API yazıyorsanız, elbette bunun için iyi javadoc'lar yazmalısınız. Ancak bu bölümdeki tavsiyenin geri kalanını da aklınızda tutun. Javadoc'lar, diğer yorum türleri gibi yanıltıcı, yerel olmayan ve dürüst olmayan olabilir. If you are writing a public API, then you should certainly write good javadocs for it. But keep in mind the rest of the advice in this chapter. Javadocs can be just as misleading, nonlocal, and dishonest as any other kind of comment.

Sonuç Uyarı Yorumu (Warning of Consequences)Warning of Consequences

Kavram AçıklamasıConcept Explanation

Bir kod bloğunun veya metodun beklenmedik yan etkileri varsa, diğer geliştiricileri uyarmak için yorum kullanılabilir. Bu, thread güvenliği sorunları veya uzun çalışma süreleri gibi durumlar için geçerlidir. If a code block or method has unexpected side effects, comments can be used to warn other developers. This applies to situations like thread safety issues or long execution times.

Kötü Örnek (Before)Bad Example (Before)

KÖTÜ — Gizli paylaşımlı durum, uyarı yokBAD — Hidden shared state, no warning 
private static DateFormat formatter = new SimpleDateFormat("yyyy-MM-dd");

public String format(Date date) {
    return formatter.format(date); // race condition!
}

İyi Örnek (After)Good Example (After)

İYİ — Uyarı yorumu veya thread-safe çözümGOOD — Warning comment or thread-safe solution 
// UYARI: SimpleDateFormat thread-safe değil.
// Her thread kendi örneğini oluşturmalı.
public String format(Date date) {
    return new SimpleDateFormat("yyyy-MM-dd").format(date);
}
Pratik İpucuPractical Tip Uyarı yorumları yazdıktan sonra "Bunu mimari düzeyde çözebilir miyim?" diye sorun. Çoğu zaman thread-local, immutable nesne veya proper locking daha kalıcı çözümdür. After writing warning comments, ask "Can I solve this at the architectural level?" Often thread-local, immutable objects, or proper locking is a more permanent solution.

TODO Yorumları (TODO Comments)TODO Comments

Kavram AçıklamasıConcept Explanation

TODO yorumları, "Şu an yapamıyorum ama yapılması gerekiyor" anlamına gelir. Ancak bunlar kötü kodu meşrulaştırmaz ve birikmeye bırakılmamalıdır. Düzenli olarak temizlenmelidir. TODO comments mean "I can't do this right now, but it needs to be done." However, they don't justify bad code and should not be left to accumulate. They should be cleaned up regularly.

Kötü Örnek (Before)Bad Example (Before)

KÖTÜ — TODO birikmesi: kötü kod bahanesiBAD — TODO pile-up: bad code excuse 
// TODO: hata yönetimi ekle
// TODO: null kontrolü yap
// TODO: bu metodu parçala
// TODO: performansı iyileştir
public List<Order> processOrders(List<Order> orders) {
    return orders.stream().map(this::process).collect(toList());
}

İyi Örnek (After)Good Example (After)

İYİ — Odaklı, gerekçeli tek TODOGOOD — Focused, justified single TODO 
public List<Order> processOrders(List<Order> orders) {
    // TODO: #342 - paralel işleme eklenecek (yük testi sonrası)
    return orders.stream().map(this::process).collect(toList());
}
Pratik İpucuPractical Tip Her TODO'ya bir issue/ticket numarası ekleyin. Böylece takip sisteminizde gerçekten kaydedilmiş olur ve "asla" yerine gerçekten "sonra" yapılır. Add an issue/ticket number to each TODO. This way it's actually recorded in your tracking system and "later" really means later instead of never.

Önem Vurgulama (Amplification)Amplification

Kavram AçıklamasıConcept Explanation

Bazen görünürde önemsiz görünen bir satır kritik bir rol oynar. Bu tür durumlarda, bir yorumla o satırın önemini vurgulamak meşrudur. Dikkat: bu gerçekten kritik olanlar için geçerlidir; her satırı önemli ilan eden bir yorum enflasyonu yaratmayın. Sometimes a seemingly trivial line plays a critical role. In such cases, it is legitimate to use a comment to amplify the importance of that line. Note: this applies to what is truly critical; don't create a comment inflation that declares every line important.

Kötü Örnek (Before)Bad Example (Before)

KÖTÜ — Kritik detay gözden kaçabilirBAD — Critical detail can be missed 
String csv = rawInput.trim();
String[] fields = csv.split(",");
processFields(fields);

İyi Örnek (After)Good Example (After)

İYİ — Önem vurgulandıGOOD — Importance amplified 
// trim() kritik: baş/son boşluklar split(",")'e girseydi
// boş field'lar oluşurdu ve parser çökerdi
String csv = rawInput.trim();
String[] fields = csv.split(",");
processFields(fields);
Pratik İpucuPractical Tip Önem vurgulayan yorumlar, aynı zamanda o satırı bir metoda veya iyi isimlendirilmiş bir değişkene çıkarmanız gerektiğinin sinyalini de verebilir. Amplification comments can also signal that you should extract that line into a method or a well-named variable.
