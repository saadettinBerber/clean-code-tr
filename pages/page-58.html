⚠ Kötü Yorumlar ⚠ Bad Comments
Kötü Yorumlar / Bad Comments Bad Comments
Yorumların büyük çoğunluğu bu kategoriye girer. Genellikle kötü kodun koltuk değnekleri ya da bahaneleridir; yetersiz kararların meşrulaştırılma çabasıdır. Özetle, programcının kendi kendine bağıra bağıra konuşmasıdır. Most comments fall into this category. Usually they are crutches or excuses for poor code or justifications for insufficient decisions, amounting to little more than the programmer talking to himself.
Mırıldanma / Mumbling Mumbling
Sırf "yapmalıyım" hissiyle ya da süreç gerektirdiği için bir yorum eklemek, yamaya çekmektir. Bir yorum yazmaya karar verdikten sonra, onu yazabileceğiniz en iyi yorum haline getirmek için gereken zamanı harcayın. Plopping in a comment just because you feel you should or because the process requires it, is a hack. If you decide to write a comment, then spend the time necessary to make sure it is the best comment you can write.
Örneğin FitNesse'de bulduğum şu duruma bakın; bir yorum gerçekten faydalı olabilirdi. Ama yazar acelede ya da yeterince dikkatli değildi. Bu mırıldanması geride bir bulmaca bıraktı: Here, for example, is a case I found in FitNesse, where a comment might indeed have been useful. But the author was in a hurry or just not paying much attention. His mumbling left behind an enigma:
Mırıldanan yorum örneği Mumbling comment example Java 
public void loadProperties()
{
  try
  {
    String propertiesPath = propertiesLocation + "/" + PROPERTIES_FILE;
    FileInputStream propertiesStream = new FileInputStream(propertiesPath);
    loadedProperties.load(propertiesStream);
  }
  catch(IOException e)
  {
    // If the properties file is missing then load all defaults
  }
}

Bu catch bloğundaki yorum ne anlama gelir? Yazar için bir anlam ifade etmiş olabilir; ancak o anlam bize yeterince aktarılmıyor. Aklımıza pek çok soru geliyor: What does that comment in the catch block mean? Clearly it meant something to the author, but the meaning does not come through all that well. Many questions arise:

  • Tüm varsayılanları kim yüklüyor? loadProperties.load çağrısından önce mi yüklendi?Who loads all the defaults? Were they loaded before the call to loadProperties.load?
  • Yoksa loadProperties.load istisnayı yakalayıp varsayılanları yükleyip ardından istisnayı mı iletdi?Or did loadProperties.load catch the exception, load the defaults, and then pass the exception on for us to ignore?
  • Yazar boş catch bloğu bırakmaktan vicdan rahatlatmak için mi yazdı?Was the author trying to comfort himself about the fact that he was leaving the catch block empty?
  • Ya da en ürkütücü ihtimal: varsayılanları yükleyecek kodu daha sonra yazmak için kendine hatırlatma mıydı bu?Or was the author trying to tell himself to come back here later and write the code that would load the defaults?
Tek çözüm, ne olduğunu anlamak için sistemin diğer bölümlerindeki kodu incelemektir. Sizi başka bir modüle bakmaya zorlayan her yorum, size iletişim kuramadığı için başarısız olmuştur ve harcadığı bitleri hak etmez. Our only recourse is to examine the code in other parts of the system to find out what's going on. Any comment that forces you to look in another module for the meaning of that comment has failed to communicate to you and is not worth the bits it consumes.
Gereksiz Yorum / Redundant Comments Redundant Comments
Listing 4-1, tamamen gereksiz bir başlık yorumuna sahip basit bir işlev gösteriyor. Yorum muhtemelen kodun kendisinden daha uzun süre okunuyor. Listing 4-1 shows a simple function with a header comment that is completely redundant. The comment probably takes longer to read than the code itself.
Listing 4-1 — waitForClose Java 
// Helper method that returns when this.closed goes true; throws an
// exception if the timeout is reached.
public synchronized void waitForClose(final long timeoutMillis)
  throws Exception
{
  if(!closed)
  {
    wait(timeoutMillis);
    if(!closed)
      throw new Exception("MockResponseSender could not be closed");
  }
}
Bu yorumun ne işlevi var? Kesinlikle koddan daha bilgilendirici değil. Kodu ne meşrulaştırıyor ne de niyet ya da gerekçe sunuyor. Koddan daha kolay okunmuyor. Tam tersine, koddan daha az kesin ve okuyucuyu gerçek anlamayı bırakıp bu belirsizliği kabullenmesine davet ediyor. Sizi arabaya gülen, "kaputun altına bakmanıza gerek yok" diyen bir ikinci el araba satıcısına benziyor. What purpose does this comment serve? It's certainly not more informative than the code. It does not justify the code, or provide intent or rationale. It is not easier to read than the code. Indeed, it is less precise than the code and entices the reader to accept that lack of precision in lieu of true understanding. It is rather like a gladhanding used-car salesman assuring you that you don't need to look under the hood.

Mırıldanma Yorumu (Mumbling)Mumbling Comment

Kavram AçıklamasıConcept Explanation

Mırıldanma, yazarın kendine yönelik yazdığı, okuyana hiçbir şey aktarmayan, birden fazla soruya yol açan yorumdur. Yazar için anlamlıyken okuyucu için bulmacaya dönüşür. Mumbling is a comment written for the author, communicating nothing to the reader, leading to multiple questions. While meaningful to the author, it becomes a puzzle for the reader.

Kötü Örnek (Before)Bad Example (Before)

KÖTÜ — Mırıldanan yorumBAD — Mumbling comment 
catch(FileNotFoundException e) {
    // Use defaults if config is missing
    // (left ambiguous on purpose!)
}

İyi Örnek (After)Good Example (After)

İYİ — İşlem açık, yorum gereksizGOOD — Action clear, comment unnecessary 
catch(FileNotFoundException e) {
    config = ConfigFactory.loadDefaults();
    logger.info("Config bulunamadı, varsayılanlar yüklendi");
}
Pratik İpucuPractical Tip Yazdığınız yorumu bir meslektaşınıza okuyun. "Peki kim bunu yapıyor?" sorusu gelirse, yorum mırıldanmaya dönmüştür — kodu konuşturun. Read your comment to a colleague. If they say "OK, but who does that?", the comment has become a mumble — let the code speak.

Gereksiz Yorum (Redundant Comment)Redundant Comment

Kavram AçıklamasıConcept Explanation

Gereksiz yorum, kodun yaptığını kelimesi kelimesine tekrar eden yorumdur. Okuyucuya hiçbir ek değer katmaz, aksine zihinsel yükü artırır. Bazen yorumu okumak koddan daha uzun sürer. A redundant comment repeats word for word what the code does. It adds no value to the reader, in fact it increases cognitive load. Sometimes reading the comment takes longer than reading the code.

Kötü Örnek (Before)Bad Example (Before)

KÖTÜ — Yorum kodu tekrar söylüyorBAD — Comment repeats the code 
// Return an empty list if the user list is empty
if (users.isEmpty()) return Collections.emptyList();

// Return user count
return users.size();

İyi Örnek (After)Good Example (After)

İYİ — Yorum yok, kod zaten açıkGOOD — No comment, code already clear 
if (users.isEmpty()) return Collections.emptyList();
return users.size();
Pratik İpucuPractical Tip Bir yorumu okuduktan sonra kendinize sorun: "Yorumu okumadan önce kodu anlamıyor muyduk?" Eğer kod zaten açıksa, yorum gereksizdir — silin. After reading a comment ask yourself: "Didn't we already understand the code without reading the comment?" If the code is already clear, the comment is redundant — delete it.
