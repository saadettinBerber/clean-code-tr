Yanıltıcı Yorum / Misleading Comments Misleading Comments
Bazen iyi niyetle de olsa, bir programcı yeterince kesin olmayan yorumlar yazar. Yukarıdaki Listing 4-1'deki yorumu yeniden düşünün. Metot, this.closed değişkeni true olduğunda "geri döner" — bu doğру. Fakat belirtilen zaman aşımı süresi geçmeden this.closed true olmadığı durumda ne olur? Yorum sessizce geçiştiriyor. Bu küçük yanıltıcı bilgi, kendisine inanan bir programcının kodu o metodu çağırıp neden yavaş çalıştığını merak etmesine neden olabilir. Sometimes, with best intentions, a programmer makes a statement in his comments that isn't precise enough to be accurate. Consider again the comment from Listing 4-1. The method does not return when this.closed becomes true. It returns if this.closed is true, and then waits for a thousand milliseconds, and then throws an exception if this.closed is still not true. This subtle bit of misinformation, couched in a comment that is harder to read than the code, could cause another programmer to blithely call this function in the expectation that it will return as soon as this.closed becomes true.
Zorunlu Yorum / Mandated Comments Mandated Comments
Her fonksiyonun bir Javadoc'a, her değişkenin bir yoruma sahip olmasını emreden kural tamamen saçmadır. Bu tür yorumlar kodu karmaşıklaştırır, yanlış anlaşılmalara zemin hazırlar ve genel bir düzensizliğe katkıda bulunur. Aşağıdaki Javadoc'u zorunlu kılan bir kuralın ürünü olmak dışında herhangi bir anlam ifade etmiyor: It is just plain silly to have a rule that says that every function must have a javadoc, or every variable must have a comment. Comments like this just clutter up the code, propagate lies, and lend to general confusion and disorganization. The following is an example of what can happen when a coding standard mandates that every function must have a javadoc:
Zorunluluktan yazılan anlamsız Javadoc Mandated meaningless Javadoc Java 
/**
 * @param title CD'nin başlığı
 * @param author CD'nin yaratıcısı
 * @param tracks CD'deki parça sayısı
 * @param durationInMinutes CD'nin dakika cinsinden süresi
 */
public void addCD(String title, String author,
                  int tracks, int durationInMinutes) {
  CD cd = new CD();
  cd.title = title;
  cd.author = author;
  cd.tracks = tracks;
  cd.duration = durationInMinutes;
  cdList.add(cd);
}
Günlük Yorumu / Journal Comments Journal Comments
Bazen insanlar bir modülü her düzenlediklerinde başına değişiklik kaydı ekler. Bu davranış uzun yıllar geriye gider. Kaynak kodu kontrol sistemleri yokken bu bir ölçüde anlam taşıyordu. Ancak günümüzde bu günlük yorumları — uzun, karmaşık ve giderek büyüyen kayıtlar — kodu baştan kirletmektedir. Silinmelidirler. Sometimes people add a comment to the start of a module every time they edit it. These comments accumulate as a kind of journal, or log, of every change that has ever been made. I have seen some modules with dozens of pages of these run-on journal entries. The running log of changes used to be important when source code control systems didn't exist. Nowadays, however, these long journals are just more clutter to obfuscate the module. They should be completely removed.
Silinmesi gereken günlük yorumu örneği Journal comment example — should be deleted 
* Değişiklikler (11-Oct-2001'den itibaren)
* ----------------------------------------
* 11-Oct-2001 : Sınıf yeniden düzenlendi ve com.jrefinery.date paketine taşındı (DG);
* 05-Nov-2001 : getDescription() metodu eklendi (DG); getMonthNames() kaldırıldı (DG);
* 12-Nov-2001 : IBD'nin talebi üzerine addDays() metoduna yorumlar eklendi (DG);
*               stringToWeekdayCode() metodu düzeltildi (DG);
* 05-Dec-2001 : Hatalar düzeltildi (DG);
* 29-May-2002 : Ay sabitlerini bir arabirime taşıdım (DG);

Yanıltıcı Yorum (Misleading Comment)Misleading Comment

Kavram AçıklamasıConcept Explanation

Yanıltıcı yorumlar iyi niyetle yazılabilir; ancak yeterince kesin değillerdir. Kodun gerçekte ne yaptığından farklı bir beklenti yaratırlar ve bu durum ince ve tehlikeli hatalara yol açar. Hiç yorum olmamaktan çok daha zararlıdırlar. Misleading comments can be written with good intent, but they are not precise enough. They create expectations that differ from what the code actually does, leading to subtle and dangerous bugs. They are far more harmful than having no comment at all.

Kötü Örnek (Before)Bad Example (Before)

KÖTÜ — Yorum tam doğru değil, yanıltıcıBAD — Comment not quite accurate, misleading 
// items listesini sıralar ve döndürür
public List<Item> getSortedItems() {
    if (items == null) return null; // yorum bunu söylemedi!
    Collections.sort(items);
    return items;
}

İyi Örnek (After)Good Example (After)

İYİ — Yorum yoksa kod doğrudan konuşuyorGOOD — No comment, code speaks directly 
public List<Item> getSortedItems() {
    if (items == null) return Collections.emptyList();
    return items.stream().sorted().collect(toList());
}
Pratik İpucuPractical Tip Yorum yazmadan önce şunu sorun: "Bu yorumun söylediği ile kodun yaptığı arasında hiçbir fark yok mu?" En küçük belirsizlik bile yanıltıcı yoruma dönüşebilir. Before writing a comment, ask: "Is there any discrepancy between what this comment says and what the code does?" Even the smallest ambiguity can turn into a misleading comment.

Zorunlu Yorum (Mandated Comment)Mandated Comment

Kavram AçıklamasıConcept Explanation

Kodlama standartlarının "her fonksiyon Javadoc içermeli" gibi katı kuralları, programcıları anlamsız yorumlar yazmaya zorlar. Bu yorumlar değer katmaz — sadece kod gürültüsü yaratır ve gerçek yorumları gizler. Coding standards with strict rules like "every function must have a Javadoc" force programmers to write meaningless comments. These comments add no value — they just create code noise and hide real comments.

Kötü Örnek (Before)Bad Example (Before)

KÖTÜ — Zoraki Javadoc, hiçbir şey eklemiyorBAD — Forced Javadoc, adds nothing 
/**
 * @param name Kullanıcı adı
 * @param email Kullanıcı e-postası
 * @return Bir kullanıcı nesnesi döndürür
 */
public User createUser(String name, String email) { ... }

İyi Örnek (After)Good Example (After)

İYİ — Açık isim, yorum gereksizGOOD — Clear name, comment unnecessary 
public User createUser(String name, String email) { ... }
Pratik İpucuPractical Tip Ekip olarak "her fonksiyon Javadoc içermeli" kuralını "herkese açık API metodları iyi belgelenmiş Javadoc içermeli" kuralıyla değiştirin. İkincisi gerçekten değer yaratır. As a team, replace the "every function must have Javadoc" rule with "public API methods must have well-documented Javadoc." The latter actually creates value.

Günlük Yorumu (Journal Comment)Journal Comment

Kavram AçıklamasıConcept Explanation

Dosyanın başına her değişiklikte eklenen tarihli kayıtlar, versiyon kontrol sistemlerinin olmadığı dönemde anlam taşırdı. Bugün Git gibi araçlar bu tarihi çok daha eksiksiz tutar. Bu yorumlar artık yalnızca kodu karıştırmaktadır — silinmelidirler. Dated entries added to the start of a file with each change made sense in the era before version control systems. Today tools like Git keep this history far more completely. These comments now only clutter code — they should be deleted.

Kötü Örnek (Before)Bad Example (Before)

KÖTÜ — Dosya başında değişiklik günlüğüBAD — Change log at top of file 
// 12-Mar-2023: auth modülü eklendi (ali)
// 15-Mar-2023: token süresi düzeltildi (ayse)
// 20-Mar-2023: refresh token eklendi (ali)
// 01-Apr-2023: rate limiting eklendi (mehmet)

İyi Örnek (After)Good Example (After)

İYİ — Git commit'leri tarihi tutarGOOD — Git commits maintain history 
$ git log --oneline auth.service.ts
a1b2c3d feat: add rate limiting
d4e5f6g feat: add refresh token support
7h8i9j0 fix: correct token expiry
k1l2m3n feat: add auth module
Pratik İpucuPractical Tip Ekibinizdeki günlük yorumları temizleyin ve anlamlı Git commit mesajları yazmayı bir kültür haline getirin. git blame ve git log zaten tüm tarihi saklıyor. Clean up journal comments in your team and make writing meaningful Git commit messages a culture. git blame and git log already store all the history.
