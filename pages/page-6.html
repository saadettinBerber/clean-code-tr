Most managers want good code, even when they are obsessing about the schedule. They may defend the schedule and requirements with passion; but that's their job. It's your job to defend the code with equal passion.
Bu noktayı pekiştirmek için şunu düşünün: Bir doktor olsaydınız ve hastanız ameliyat hazırlığındaki el yıkama işlemini durdumanızı isteseydi — çünkü çok fazla zaman alıyormuş2 — ne yapardınız? Açıkça hasta patrondur; ama doktor kesinlikle bu isteğe uymamalıdır. Neden? Çünkü doktor, hasta hakkında hastalık ve enfeksiyon riskleri konusunda hastadan daha fazla şey bilir. Doktorun hastanın isteğine uyması profesyonel olmayan (unprofessional) bir davranış olurdu (cezai sorumluluğu bir yana). To drive this point home, what if you were a doctor and had a patient who demanded that you stop all the silly hand-washing in preparation for surgery because it was taking too much time?2 Clearly the patient is the boss; and yet the doctor should absolutely refuse to comply. Why? Because the doctor knows more than the patient about the risks of disease and infection. It would be unprofessional (never mind criminal) for the doctor to comply with the patient.
Aynı şekilde, karmaşa (mess) yaratmanın risklerini anlamayan yöneticilerin (managers) iradesine boyun eğmek programcılar için de profesyonel olmayan bir davranıştır. So too it is unprofessional for programmers to bend to the will of managers who don't understand the risks of making messes.
Temel İkilem / The Primal Conundrum The Primal Conundrum
Programcılar temel değerlerle ilgili bir ikilemle (conundrum) karşı karşıyadır. Birkaç yıldan fazla deneyime sahip tüm geliştiriciler, önceki karmaşaların (messes) kendilerini yavaşlattığını bilir. Ancak tüm geliştiriciler, son teslim tarihlerini (deadlines) karşılamak için karmaşa yaratma baskısını hisseder. Kısacası, hızlı gitmek için zaman ayırmazlar! Programmers face a conundrum of basic values. All developers with more than a few years experience know that previous messes slow them down. And yet all developers feel the pressure to make messes in order to meet deadlines. In short, they don't take the time to go fast!
Gerçek profesyoneller (professionals), ikilemin ikinci kısmının yanlış olduğunu bilir. Karmaşa yaparak son teslim tarihine yetişemezsiniz. Aslında karmaşa sizi anında yavaşlatır ve son teslim tarihini kaçırmanıza neden olur. Son teslim tarihine yetişmenin tek yolu — hızlı gitmenin tek yolu — kodu her zaman mümkün olduğunca temiz tutmaktır. True professionals know that the second part of the conundrum is wrong. You will not make the deadline by making the mess. Indeed, the mess will slow you down instantly, and will force you to miss the deadline. The only way to make the deadline—the only way to go fast—is to keep the code as clean as possible at all times.
Temiz Kod Sanatı mı? / The Art of Clean Code? The Art of Clean Code?
Diyelim ki dağınık kodun (messy code) önemli bir engel olduğuna inandınız. Diyelim ki hızlı gitmenin tek yolunun kodunuzu temiz tutmak olduğunu kabul ettiniz. O zaman kendinize şunu sormalısınız: "Temiz kodu nasıl yazarım?" Eğer kodun temiz olmasının ne demek olduğunu bilmiyorsanız, temiz kod yazmaya çalışmanın bir anlamı yoktur! Let's say you believe that messy code is a significant impediment. Let's say that you accept that the only way to go fast is to keep your code clean. Then you must ask yourself: "How do I write clean code?" It's no good trying to write clean code if you don't know what it means for code to be clean!
Kötü haber şu ki, temiz kod yazmak bir resim yapmaya çok benzer. Çoğumuz bir resmin iyi mi kötü mü yapıldığını biliriz. Ancak iyi sanatı kötüden ayırabilmek, resim yapmayı bildiğimiz anlamına gelmez. Aynı şekilde temiz kodu kirli koddan (dirty code) ayırabilmek, temiz kod yazmayı bildiğimiz anlamına gelmez! The bad news is that writing clean code is a lot like painting a picture. Most of us know when a picture is painted well or badly. But being able to recognize good art from bad does not mean that we know how to paint. So too being able to recognize clean code from dirty code does not mean that we know how to write clean code!

Doktor Benzetmesi (Doctor Analogy)Doctor Analogy

Kavram AçıklamasıConcept Explanation

Martin, programcıların kodu savunma sorumluluğunu bir doktorun el yıkamayı savunmasına benzetir. Hasta (yönetici/müşteri) "zaman kaybı" diye şikâyet etse bile, doktor hijyen kurallarından ödün vermez — çünkü riskleri bilir. Aynı şekilde programcılar da "hızlı teslim et" baskısına rağmen kod kalitesinden ödün vermemelidir.Martin compares programmers' responsibility to defend code quality with a doctor defending hand-washing. Even when the patient (manager/client) complains about "wasting time," the doctor doesn't compromise on hygiene — because they know the risks. Similarly, programmers shouldn't compromise code quality despite "deliver fast" pressure.

İYİ — Profesyonel duruşGOOD — Professional stance 
// Yönetici: "Bu özellik yarına hazır olmalı!"
// Geliştirici: "İndirim hesaplama kritik bir iş kuralı.
//   Hatalı hesaplama gelir kaybına yol açar.
//   2 günde testleriyle birlikte teslim edebilirim."

function calculateDiscount(cart: CartItem[]): Money {
    this.validateCart(cart);

    const discountRules = [
        new BulkDiscountRule(threshold: 5, rate: 0.10),
        new VolumeBonusRule(threshold: 10, rate: 0.05),
        new VipDiscountRule(rate: 0.15),
    ];

    return discountRules
        .filter(rule => rule.appliesTo(cart))
        .reduce((total, rule) => total.add(rule.calculate(cart)),
                Money.zero());
}

Temel İkilem (The Primal Conundrum)The Primal Conundrum

Kavram AçıklamasıConcept Explanation

Programcıların temel ikilemi şudur: Karmaşanın (mess) bizi yavaşlattığını biliyoruz, ama son teslim tarihlerine (deadlines) yetişmek için karmaşa yaratma baskısı hissediyoruz. Martin bu ikilemin çözümünü açıkça belirtir: Hızlı gitmenin tek yolu kodu temiz tutmaktır. Karmaşa yapmak sizi hızlandırmaz — anında yavaşlatır.The fundamental conundrum: we know messes slow us down, yet we feel pressure to make messes to meet deadlines. Martin states the resolution clearly: the only way to go fast is to keep the code clean. Making a mess doesn't speed you up — it slows you down instantly.

Temiz Kod Sanatı (The Art of Clean Code)The Art of Clean Code

Kavram AçıklamasıConcept Explanation

Martin, temiz kod yazmayı resim yapmaya benzetir: İyi bir resmi kötüsünden ayırt edebilmek, resim yapmayı bildiğiniz anlamına gelmez. Aynı şekilde temiz kodu kirli koddan ayırt edebilmek, temiz kod yazabildiğiniz anlamına gelmez. Temiz kod yazmak bir zanaat (craftsmanship) becerisidir — öğrenilmesi zaman, pratik ve disiplin gerektirir.Martin compares writing clean code to painting: being able to tell good art from bad doesn't mean you can paint. Similarly, recognizing clean code from dirty code doesn't mean you can write clean code. Writing clean code is a craftsmanship skill — it requires time, practice, and discipline to learn.

Recognizing problems is the first step, but it's not enough.

İYİ — Kod duygusu ile yeniden yapılandırılmışGOOD — Restructured with code-sense 
// Strateji deseni (Strategy Pattern) + açık arayüz
public interface DataFormatter {
    FormattedData format(RawData data);
}

public class JsonFormatter implements DataFormatter {
    public FormattedData format(RawData data) {
        return new FormattedData(toJson(data));
    }
}

public class DataPipeline {
    private final DataFormatter formatter;
    private final RetryPolicy retryPolicy;

    public DataPipeline(DataFormatter formatter, RetryPolicy retryPolicy) {
        this.formatter = formatter;
        this.retryPolicy = retryPolicy;
    }

    public ProcessedData process(RawData data) {
        RawData validated = validate(data);
        FormattedData formatted = formatter.format(validated);
        return retryPolicy.execute(() -> save(formatted));
    }
}

"Bu neden böyle yazılmış?" diye sorun. Zamanla kalıpları tanımaya ve uygulamaya başlarsınız.To develop "code-sense": spend 15 minutes daily reading open source code.
