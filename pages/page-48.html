Kendini Tekrar Etme / Don't Repeat Yourself Don't Repeat Yourself
Liste 3-1'e dikkatlice bakın ve dört kez tekrarlanan bir algoritma olduğunu fark edeceksiniz: SetUp, SuiteSetUp, TearDown ve SuiteTearDown durumlarının her biri için bir kez. Bu tekrarı fark etmek kolay değildir çünkü dört örnek diğer kodlarla iç içe geçmiştir ve düzgün bir şekilde tekrar edilmemiştir. Yine de tekrar bir sorundur çünkü kodu şişirir ve algoritma değişirse dört kat değişiklik gerektirir. Ayrıca bir atlanma hatası için dört kat fırsat yaratır. Look back at Listing 3-1 carefully and you will notice that there is an algorithm that gets repeated four times, once for each of the SetUp, SuiteSetUp, TearDown, and SuiteTearDown cases. It's not easy to spot this duplication because the four instances are intermixed with other code and aren't uniformly duplicated. Still, the duplication is a problem because it bloats the code and will require four-fold modification should the algorithm ever have to change. It is also a four-fold opportunity for an error of omission.
Bu tekrar, Liste 3-7'deki include metodu tarafından giderilmiştir. Kodu tekrar okuyun ve tüm modülün okunabilirliğinin tekrarın azaltılmasıyla nasıl arttığına dikkat edin. This duplication was remedied by the include method in Listing 3-7. Read through that code again and notice how the readability of the whole module is enhanced by the reduction of that duplication.
Tekrar (duplication), yazılımdaki tüm kötülüklerin kökü olabilir. Onu kontrol etmek veya ortadan kaldırmak amacıyla birçok ilke ve uygulama oluşturulmuştur. Örneğin Codd'un veritabanı normal formlarının (normal forms) tümünün, verideki tekrarı ortadan kaldırmaya hizmet ettiğini düşünün. Nesne yönelimli programlamanın (OOP), aksi takdirde gereksiz olacak kodları temel sınıflarda (base classes) toplamaya nasıl hizmet ettiğini de düşünün. Yapısal programlama (Structured Programming), Görünüm Yönelimli Programlama (Aspect Oriented Programming), Bileşen Yönelimli Programlama (Component Oriented Programming) — bunların hepsi kısmen tekrarı ortadan kaldırma stratejileridir. Alt rutinin (subroutine) icadından bu yana, yazılım geliştirmedeki yenilikler, kaynak kodumuzdan tekrarı ortadan kaldırmaya yönelik süregelen bir çaba olmuştur. Duplication may be the root of all evil in software. Many principles and practices have been created for the purpose of controlling or eliminating it. Consider, for example, that all of Codd's database normal forms serve to eliminate duplication in data. Consider also how object-oriented programming serves to concentrate code into base classes that would otherwise be redundant. Structured programming, Aspect Oriented Programming, Component Oriented Programming, are all, in part, strategies for eliminating duplication. It would appear that since the invention of the subroutine, innovations in software development have been an ongoing attempt to eliminate duplication from our source code.
Yapısal Programlama / Structured Programming Structured Programming
Bazı programcılar Edsger Dijkstra'nın yapısal programlama (structured programming) kurallarını takip ederler.14 Dijkstra, her fonksiyonun ve fonksiyon içindeki her bloğun tek bir giriş ve tek bir çıkışa sahip olması gerektiğini söylemiştir. Bu kuralları takip etmek, bir fonksiyonda yalnızca bir return ifadesi olması, bir döngüde break veya continue ifadesi olmaması ve asla, asla goto ifadesi kullanılmaması anlamına gelir. Some programmers follow Edsger Dijkstra's rules of structured programming.14 Dijkstra said that every function, and every block within a function, should have one entry and one exit. Following these rules means that there should only be one return statement in a function, no break or continue statements in a loop, and never, ever, any goto statements.
Bu hedeflere ve yapısal programlamanın disiplinine sempatiyle yaklaşsak da, fonksiyonlar çok küçük olduğunda bu kurallar çok az fayda sağlar. Yalnızca daha büyük fonksiyonlarda bu kurallar önemli bir fayda sunar. While we are sympathetic to the goals and disciplines of structured programming, those rules serve little benefit when functions are very small. It is only in larger functions that such rules provide significant benefit.
Dolayısıyla, fonksiyonlarınızı küçük tutarsanız, ara sıra birden fazla return, break veya continue ifadesi kullanmak zarar vermez ve bazen tek-giriş, tek-çıkış kuralından bile daha ifade edici olabilir. Öte yandan goto yalnızca büyük fonksiyonlarda anlam taşır, bu nedenle ondan kaçınılmalıdır. So if you keep your functions small, then the occasional multiple return, break, or continue statement does no harm and can sometimes even be more expressive than the single-entry, single-exit rule. On the other hand, goto only makes sense in large functions, so it should be avoided.

Kendini Tekrar Etme İlkesi (DRY — Don't Repeat Yourself)DRY — Don't Repeat Yourself

Kavram AçıklamasıConcept Explanation

DRY (Don't Repeat Yourself — Kendini Tekrar Etme) ilkesi, her bilgi parçasının sistemde tek ve belirgin bir temsile sahip olması gerektiğini söyler. Kod tekrarı, bir değişiklik gerektiğinde birden fazla yerin güncellenmesini zorunlu kılar ve hata riskini artırır.The DRY (Don't Repeat Yourself) principle states that every piece of knowledge should have a single, unambiguous representation in the system. Code duplication forces multiple places to be updated when a change is needed, increasing the risk of errors.

Kötü Örnek (Before)Bad Example (Before)

KÖTÜ — Tekrarlanan doğrulama mantığıBAD — Duplicated validation logic 
function createUser(name, email) {
    if (!email || !email.includes("@")) {
        throw new Error("Invalid email");
    }
    // ... kullanıcı oluştur
}

function updateEmail(userId, email) {
    if (!email || !email.includes("@")) {
        throw new Error("Invalid email");
    }
    // ... email güncelle
}

function sendInvite(email) {
    if (!email || !email.includes("@")) {
        throw new Error("Invalid email");
    }
    // ... davetiye gönder
}

İyi Örnek (After)Good Example (After)

İYİ — Tek doğrulama fonksiyonuGOOD — Single validation function 
function validateEmail(email) {
    if (!email || !email.includes("@")) {
        throw new Error("Invalid email");
    }
}

function createUser(name, email) {
    validateEmail(email);
    // ... kullanıcı oluştur
}

function updateEmail(userId, email) {
    validateEmail(email);
    // ... email güncelle
}

function sendInvite(email) {
    validateEmail(email);
    // ... davetiye gönder
}
Pratik İpucuPractical Tip"Üç kuralı"nı uygulayın: Aynı kodu iki kez yazdıysanız not edin, üçüncü kez yazdığınızda refactor edin. Tekrarı tespit etmenin en kolay yolu: "Bu mantık değişse kaç yeri güncellemem gerekir?" sorusunu sormaktır.Apply the "Rule of Three": if you've written the same code twice, note it; the third time, refactor. The easiest way to detect duplication: ask "If this logic changes, how many places do I need to update?"

Yapısal Programlama (Structured Programming)Structured Programming

Kavram AçıklamasıConcept Explanation

Dijkstra'nın yapısal programlama (structured programming) kuralları, her fonksiyonun tek giriş ve tek çıkışa sahip olmasını önerir. Ancak modern yazılım geliştirmede, fonksiyonlar küçük tutulduğunda, birden fazla return veya break ifadesi zarar vermez. Asıl kaçınılması gereken goto ifadesidir.Dijkstra's rules of structured programming suggest that every function should have one entry and one exit. However, in modern software development, when functions are kept small, multiple return or break statements do no harm. The real thing to avoid is goto.

Kötü Örnek (Before)Bad Example (Before)

KÖTÜ — Tek return zorlama (gereksiz karmaşıklık)BAD — Forcing single return (unnecessary complexity) 
def find_user(users, user_id):
    result = None
    for user in users:
        if user.id == user_id:
            result = user
            # Buldu ama döngüye devam ediyor!
    return result

İyi Örnek (After)Good Example (After)

İYİ — Erken dönüş (küçük fonksiyonda sorun yok)GOOD — Early return (fine in small functions) 
def find_user(users, user_id):
    for user in users:
        if user.id == user_id:
            return user
    return None
Pratik İpucuPractical TipFonksiyonlarınız küçükse (5-10 satır), birden fazla return kullanmaktan çekinmeyin. Okunabilirliği artırır. Ancak 50+ satırlık fonksiyonlarda tek-giriş, tek-çıkış kuralı hâlâ değerlidir.If your functions are small (5-10 lines), don't hesitate to use multiple return statements. It improves readability. But in 50+ line functions, the single-entry, single-exit rule still has value.

Tek Giriş / Tek Çıkış (Single Entry / Single Exit)Single Entry / Single Exit

Kavram AçıklamasıConcept Explanation

Tek giriş / tek çıkış (single-entry, single-exit) kuralı, bir fonksiyonun sadece bir noktadan başlaması ve bir noktadan çıkması gerektiğini belirtir. goto ifadeleri bu kuralı kesinlikle ihlal eder. Küçük fonksiyonlarda ise birden fazla return, break veya continue kullanmak kabul edilebilir ve hatta daha ifade edici olabilir.The single-entry, single-exit rule states that a function should start at only one point and exit at only one point. goto statements definitively violate this rule. In small functions, however, using multiple return, break, or continue is acceptable and can even be more expressive.

Kötü Örnek (Before)Bad Example (Before)

KÖTÜ — goto kullanımı (ASLA kullanmayın)BAD — goto usage (NEVER use) 
// C dili örneği
int processData(int* data, int size) {
    int result = 0;
    if (data == NULL)
        goto error;
    if (size <= 0)
        goto error;
    // ... işleme kodu
    goto done;
error:
    result = -1;
done:
    return result;
}

İyi Örnek (After)Good Example (After)

İYİ — Koruma cümleleriyle erken dönüşGOOD — Early return with guard clauses 
int processData(int* data, int size) {
    if (data == NULL || size <= 0)
        return -1;

    // ... işleme kodu
    return result;
}
Pratik İpucuPractical Tipgoto'dan kesinlikle kaçının — modern dillerde zaten çoğunda yoktur. Küçük fonksiyonlarda erken return kullanın. Büyük fonksiyonları ise küçüğe böldüğünüzde tek-giriş, tek-çıkış sorunu kendiliğinden çözülür.Absolutely avoid goto — most modern languages don't even have it. Use early return in small functions. When you break large functions into small ones, the single-entry, single-exit problem solves itself.
