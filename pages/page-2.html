Clean Code — Robert C. Martin
Chapter 1: Clean Code / Temiz Kod › There Will Be Code / Kod Her Zaman Var Olacak
Sayfa
Bu kitap iyi programlama hakkındadır. Baştan sona kodla doludur. Koda her yönden bakacağız. Yukarıdan aşağıya, aşağıdan yukarıya ve içinden dışına doğru inceleyeceğiz. İşimiz bittiğinde, kod hakkında pek çok şey biliyor olacağız. Dahası, iyi kod ile kötü kod arasındaki farkı anlayabilecek durumda olacağız. İyi kod yazmayı bileceğiz. Ve kötü kodu iyi koda nasıl dönüştüreceğimizi de bileceğiz. This is a book about good programming. It is filled with code. We are going to look at code from every different direction. We'll look down at it from the top, up at it from the bottom, and through it from the inside out. By the time we are done, we're going to know a lot about code. What's more, we'll be able to tell the difference between good code and bad code. We'll know how to write good code. And we'll know how to transform bad code into good code.
Kod Her Zaman Var Olacak / There Will Be Code There Will Be Code
Kod hakkında bir kitabın çağın gerisinde kaldığını savunanlar olabilir — kodun artık sorun olmadığını, modeller (models) ve gereksinimlerle (requirements) ilgilenmemiz gerektiğini söyleyenler. Hatta bazıları kodun sonunun yaklaştığını ileri sürdü. Yakında tüm kodun yazılmak yerine üretileceğini (generated) söylediler. Programcılara artık ihtiyaç kalmayacağını, çünkü iş dünyasının programları doğrudan spesifikasyonlardan (specifications) oluşturacağını iddia ettiler. One might argue that a book about code is somehow behind the times—that code is no longer the issue; that we should be concerned about models and requirements instead. Indeed some have suggested that we are close to the end of code. That soon all code will be generated instead of written. That programmers simply won't be needed because business people will generate programs from specifications.
Saçmalık! Koddan asla kurtulamayacağız, çünkü kod gereksinimlerin ayrıntılarını temsil eder. Bir noktada bu ayrıntılar görmezden gelinemez veya soyutlanamaz (abstracted); belirtilmek zorundadırlar. Ve gereksinimleri bir makinenin çalıştırabileceği kadar ayrıntılı bir şekilde belirtmek programlamadır. Böyle bir spesifikasyon (specification) koddur. Nonsense! We will never be rid of code, because code represents the details of the requirements. At some level those details cannot be ignored or abstracted; they have to be specified. And specifying requirements in such detail that a machine can execute them is programming. Such a specification is code.
Dillerimizin soyutlama (Abstraction) seviyesinin artmaya devam edeceğini bekliyorum. Alana özgü dillerin (domain-specific languages) sayısının da artmaya devam edeceğini öngörüyorum. Bu iyi bir şey olacak. Ama kodu ortadan kaldırmayacak. Gerçekten de, bu üst düzey ve alana özgü dillerde yazılan tüm spesifikasyonlar kod olacak! Yine de titiz, doğru, öyle biçimsel ve ayrıntılı olmak zorunda olacak ki bir makine onu anlayıp çalıştırabilsin. I expect that the level of abstraction of our languages will continue to increase. I also expect that the number of domain-specific languages will continue to grow. This will be a good thing. But it will not eliminate code. Indeed, all the specifications written in these higher level and domain-specific language will be code! It will still need to be rigorous, accurate, and so formal and detailed that a machine can understand and execute it.
Kodun bir gün yok olacağını düşünenler, biçimsel (formal) olmak zorunda olmayan bir matematik keşfetmeyi uman matematikçilere benzerler. Ne söylediğimizden ziyade ne istediğimizi yapabilecek makineler yaratmanın bir yolunu bulacağımızı umuyorlar. Bu makineler bizi o kadar iyi anlayabilmeli ki, belirsizce ifade edilmiş ihtiyaçlarımızı, bu ihtiyaçları tam olarak karşılayan mükemmel çalışan programlara dönüştürebilsinler. The folks who think that code will one day disappear are like mathematicians who hope one day to discover a mathematics that does not have to be formal. They are hoping that one day we will discover a way to create machines that can do what we want rather than what we say. These machines will have to be able to understand us so well that they can translate vaguely specified needs into perfectly executing programs that precisely meet those needs.
Bu asla gerçekleşmeyecek. Tüm sezgileri ve yaratıcılıklarıyla birlikte insanlar bile, müşterilerinin belirsiz duygularından başarılı sistemler oluşturmayı başaramamıştır. Gerçekten de, gereksinim spesifikasyonu (requirements specification) disiplini bize bir şey öğrettiyse, o da iyi tanımlanmış gereksinimlerin kod kadar biçimsel olduğu ve o kodun çalıştırılabilir testleri (executable tests) olarak işlev görebildiğidir! This will never happen. Not even humans, with all their intuition and creativity, have been able to create successful systems from the vague feelings of their customers. Indeed, if the discipline of requirements specification has taught us anything, it is that well-specified requirements are as formal as code and can act as executable tests of that code!
Unutmayın ki kod, gereksinimleri nihai olarak ifade ettiğimiz dildir. Gereksinimlere daha yakın diller oluşturabiliriz. Bu gereksinimleri biçimsel yapılara ayrıştırmamıza (parse) ve birleştirmemize yardımcı araçlar geliştirebiliriz. Ancak gerekli hassasiyeti (precision) asla ortadan kaldıramayız — bu nedenle her zaman kod olacaktır. Remember that code is really the language in which we ultimately express the requirements. We may create languages that are closer to the requirements. We may create tools that help us parse and assemble those requirements into formal structures. But we will never eliminate necessary precision—so there will always be code.

Soyutlama (Abstraction) Abstraction

Kavram Açıklaması Concept Explanation

Soyutlama (Abstraction), karmaşık detayları gizleyerek daha yüksek seviyede, daha anlaşılır bir arayüz sunma sürecidir. Kitapta belirtildiği gibi, programlama dillerinin soyutlama seviyesi artsa da kodun kendisi asla ortadan kalkmayacaktır — çünkü bir noktada detaylar kesin olarak belirtilmek zorundadır. Abstraction is the process of hiding complex details to present a higher-level, more understandable interface. As stated in the book, even though the abstraction level of programming languages will increase, code itself will never disappear — because at some point details must be precisely specified.

Kötü Örnek (Before) Bad Example (Before)

KÖTÜ BAD 
class ReportGenerator:
    def generate(self, data, format_type):
        if format_type == "pdf":
            # 50 satır PDF oluşturma kodu...
            doc = PDFDocument()
            doc.set_font("Arial", 12)
            doc.add_page()
            for row in data:
                doc.add_row(row[0], row[1], row[2])
            doc.save("/tmp/report.pdf")
            smtp = smtplib.SMTP("mail.server.com")
            smtp.send("/tmp/report.pdf", "boss@company.com")
        elif format_type == "csv":
            # 30 satır CSV oluşturma kodu...
            with open("/tmp/report.csv", "w") as f:
                for row in data:
                    f.write(",".join(row))
            smtp = smtplib.SMTP("mail.server.com")
            smtp.send("/tmp/report.csv", "boss@company.com")
        elif format_type == "html":
            # 40 satır HTML oluşturma kodu...
            pass
Tüm detaylar tek bir metotta karışık biçimde sunuluyor. Rapor oluşturma, dosya kaydetme ve e-posta gönderme hepsi iç içe geçmiş. Soyutlama (Abstraction) yok, her şey en düşük seviyede açık. Yeni bir format eklemek tüm metodu değiştirmeyi gerektiriyor. All details are mixed in a single method. Report generation, file saving, and email sending are all tangled together. There's no abstraction, everything is exposed at the lowest level. Adding a new format requires modifying the entire method.

İyi Örnek (After) Good Example (After)

İYİ GOOD 
class ReportGenerator:
    def __init__(self, formatter, sender):
        self.formatter = formatter
        self.sender = sender

    def generate_and_send(self, data, recipient):
        report = self.formatter.format(data)
        self.sender.send(report, recipient)


class PdfFormatter:
    def format(self, data):
        # PDF'e özgü detaylar burada soyutlanmış
        ...

class CsvFormatter:
    def format(self, data):
        # CSV'ye özgü detaylar burada soyutlanmış
        ...
Her sorumluluk kendi sınıfına (Class) soyutlanmış. ReportGenerator sadece "formatla ve gönder" gibi üst seviye kavramlarla çalışıyor. Detaylar alt sınıflarda gizli. Yeni bir format eklemek için mevcut koda dokunmaya gerek yok — sadece yeni bir Formatter sınıfı oluşturulur. Each responsibility is abstracted into its own class. ReportGenerator works only with high-level concepts like "format and send." Details are hidden in subclasses. Adding a new format doesn't require touching existing code — just create a new Formatter class.
Pratik İpucu Practical Tip Bir fonksiyon veya sınıf yazarken kendinize sorun: "Bu kodu okuyan biri, iç detayları bilmeden ne yaptığını anlayabilir mi?" Eğer anlayamıyorsa, doğru soyutlama seviyesini yakalayamamışsınız demektir. When writing a function or class, ask yourself: "Can someone reading this code understand what it does without knowing the internal details?" If not, you haven't found the right level of abstraction.

Spesifikasyon / Kod Olarak Gereksinim Specification as Code

Kavram Açıklaması Concept Explanation

Martin'in bu bölümdeki temel argümanı şudur: Gereksinimler ne kadar detaylandırılırsa, koda o kadar yaklaşır. İyi tanımlanmış bir spesifikasyon (specification), özünde koddur. Bu nedenle "kod artık gerekli olmayacak" iddiası geçersizdir — çünkü hassasiyet (precision) her zaman gereklidir. Martin's core argument in this section is: The more detailed requirements become, the closer they get to code. A well-specified specification is essentially code. Therefore the claim "code will no longer be needed" is invalid — because precision is always required.

Kötü Örnek (Before) Bad Example (Before)

KÖTÜ — Belirsiz Gereksinim BAD — Vague Requirement 
// Gereksinim: "Kullanıcı indirim alsın"

function applyDiscount(user) {
    // hangi kullanıcı? ne kadar indirim?
    // hangi ürünlerde? ne zaman geçerli?
    return price * 0.9; // ???
}
Belirsiz bir gereksinim, belirsiz bir koda dönüşmüş. "Kullanıcı indirim alsın" gibi muğlak bir ifade, sihirli sayılar (magic numbers) ve cevaplanmamış sorularla dolu bir uygulamaya yol açıyor. Spesifikasyon (specification) yetersiz olunca kod da yetersiz kalıyor. A vague requirement turned into vague code. An ambiguous statement like "user should get discount" leads to implementation full of magic numbers and unanswered questions. When specification is insufficient, code is insufficient too.

İyi Örnek (After) Good Example (After)

İYİ — Kesin Spesifikasyon = Kesin Kod GOOD — Precise Specification = Precise Code 
PREMIUM_DISCOUNT_RATE = 0.15
STANDARD_DISCOUNT_RATE = 0.05
MIN_ORDER_FOR_DISCOUNT = 100.00

def apply_discount(order_total, membership_tier):
    if order_total < MIN_ORDER_FOR_DISCOUNT:
        return order_total

    if membership_tier == "premium":
        return order_total * (1 - PREMIUM_DISCOUNT_RATE)
    elif membership_tier == "standard":
        return order_total * (1 - STANDARD_DISCOUNT_RATE)

    return order_total
Kesin gereksinimler, kesin koda dönüşmüş. Her iş kuralı açık: minimum sipariş tutarı, üyelik seviyesine göre indirim oranları, tüm sabitler anlamlı isimlerle tanımlı. Bu kod aynı zamanda bir spesifikasyon gibi de okunabiliyor — çünkü Martin'in dediği gibi, iyi kod zaten spesifikasyondur. Precise requirements turned into precise code. Every business rule is explicit: minimum order amount, discount rates by membership tier, all constants with meaningful names. This code also reads like a specification — because as Martin says, good code is specification.
Pratik İpucu Practical Tip Bir gereksinimi kodlamaya başlamadan önce onu yeterince kesin biçimde ifade edebiliyor musunuz kontrol edin. Eğer gereksinimi bir cümleyle net şekilde yazamıyorsanız, kodu da net yazamazsınız. "Belirsiz gereksinim = belirsiz kod" formülünü unutmayın. Before starting to code a requirement, check if you can express it precisely enough. If you can't write the requirement clearly in a sentence, you can't write the code clearly either. Remember the formula: "vague requirement = vague code."

İyi Kod vs. Kötü Kod Good Code vs. Bad Code

Kavram Açıklaması Concept Explanation

Martin bu bölümün girişinde kitabın temel vaadini ortaya koyuyor: Bu kitabı okuduktan sonra iyi kod ile kötü kod arasındaki farkı anlayacaksınız, iyi kod yazmayı bileceksiniz ve kötü kodu iyi koda dönüştürebileceksiniz. Bu farkındalık, Temiz Kod (Clean Code) yolculuğunun ilk adımıdır. Martin lays out the book's core promise at the beginning of this chapter: after reading this book, you'll understand the difference between good and bad code, know how to write good code, and be able to transform bad code into good code. This awareness is the first step in the Clean Code journey.

Kötü Örnek (Before) Bad Example (Before)

KÖTÜ BAD 
public class U {
    public static List<int[]> g(List<int[]> l) {
        List<int[]> r = new ArrayList<>();
        for (int[] x : l)
            if (x[0] == 4)
                r.add(x);
        return r;
    }
}
Sınıf adı (U), metot adı (g), değişken adları (l, r, x) tamamen anlamsız. 4 sihirli sayısı (magic number) neyi temsil ediyor? x[0] ne anlama geliyor? Bu kodu yazan kişi dışında hiç kimse anlayamaz. Class name (U), method name (g), variable names (l, r, x) are completely meaningless. What does the magic number 4 represent? What does x[0] mean? Nobody except the author can understand this code.

İyi Örnek (After) Good Example (After)

İYİ GOOD 
public class GameBoard {
    private static final int FLAGGED = 4;
    private static final int STATUS_INDEX = 0;

    public List<Cell> getFlaggedCells(List<Cell> cells) {
        List<Cell> flaggedCells = new ArrayList<>();
        for (Cell cell : cells) {
            if (cell.isFlagged()) {
                flaggedCells.add(cell);
            }
        }
        return flaggedCells;
    }
}
Aynı iş mantığı, ama artık okunabilir (Readable Code). Sınıf adı (GameBoard) bağlamı veriyor. Metot adı (getFlaggedCells) ne yaptığını söylüyor. Sihirli sayı yerine anlamlı sabit (FLAGGED) kullanılıyor. Cell nesnesi (Object) int[] dizisinin yerini almış. Kodu ilk kez okuyan biri bile ne yaptığını anlayabiliyor. Same business logic, but now readable. Class name (GameBoard) provides context. Method name (getFlaggedCells) tells what it does. Meaningful constants (FLAGGED) replace magic numbers. A Cell object replaces int[]. Even a first-time reader can understand what this code does.
Pratik İpucu Practical Tip Kodunuzu bir iş arkadaşınıza 30 saniyede açıklayamıyorsanız, muhtemelen kötü koddur. İyi kod kendini açıklar — okuyucuyu düşünmeye değil, anlamaya yönlendirir. Her sihirli sayıya bir isim verin, her int[]'i bir nesneye (Object) dönüştürmeyi düşünün. If you can't explain your code to a colleague in 30 seconds, it's probably bad code. Good code explains itself — it guides the reader toward understanding, not puzzlement. Name every magic number, consider turning every int[] into an object.
