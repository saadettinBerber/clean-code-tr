HTML Yorumu / HTML Comments HTML Comments
Kaynak kodu yorumlarında HTML bir iğrençliktir. Aşağıdaki kodu okuyunca göreceğiniz gibi, yorumların okunmasını en kolay olması gereken yerde — editörde/IDE'de — zorlaştırır. Yorumlar bir araç (Javadoc gibi) tarafından çıkarılıp bir web sayfasında gösterilecekse, uygun HTML etiketleriyle donatmak programcının değil, o aracın sorumluluğu olmalıdır: HTML in source code comments is an abomination, as you can tell by reading the code below. It makes the comments hard to read in the one place where they should be easy to read—the editor/IDE. If comments are going to be extracted by some tool (like Javadoc) to appear in a Web page, then it should be the responsibility of that tool, and not the programmer, to adorn the comments with appropriate HTML.
HTML yorumu — okunması güç HTML comment — hard to read 
/**
 * FitNesse testlerini çalıştıran görev.
 * <p/>
 * <pre>
 * Kullanım:
 * &lt;taskdef name=&quot;execute-fitnesse-tests&quot;
 *     classname=&quot;fitnesse.ant.ExecuteFitnesseTestsTask&quot;
 *     classpathref=&quot;classpath&quot; /&gt;
 * VEYA
 * &lt;execute-fitnesse-tests
 *     suitepage=&quot;FitNesse.SuiteAcceptanceTests&quot;
 *     fitnesseport=&quot;8082&quot; /&gt;
 * </pre>
 */
Yerel Olmayan Bilgi / Nonlocal Information Nonlocal Information
Bir yorum yazacaksanız, yakınında bulunduğu kodu açıkladığından emin olun. Yerel bir yorumun içinde sistemin tamamına ait bilgi vermeyin. Örneğin aşağıdaki Javadoc yorumunu ele alalım. Yorumun aşırı gereksiz olması bir yana, varsayılan port hakkında da bilgi veriyor. Oysa fonksiyon bu varsayılan değer üzerinde hiçbir denetime sahip değil. Bu yorum fonksiyonu değil, sistemin çok uzaklarındaki başka bir parçayı anlatıyor: If you must write a comment, then make sure it describes the code it appears near. Don't offer systemwide information in the context of a local comment. Consider, for example, the javadoc comment below. Aside from the fact that it is horribly redundant, it also offers information about the default port. And yet the function has absolutely no control over what that default is. The comment is not describing the function, but some other, far distant part of the system.
Yerel olmayan bilgi içeren yorum Comment with nonlocal information 
/**
 * fitnesse'in çalışacağı port. Varsayılan: <b>8082</b>.
 *
 * @param fitnessePort
 */
public void setFitnessePort(int fitnessePort)
{
  this.fitnessePort = fitnessePort;
}
Bu yorum, varsayılan değeri içeren kod değiştiğinde yorumun da güncelleneceğini garanti etmiyor. Yorumun yerel olmayan bilgiye dayanması onu hem hatalı hem de bakımı güç kılıyor. Of course there is no guarantee that this comment will be changed when the code containing the default is changed. The nonlocal nature of the comment makes it both error-prone and hard to maintain.

HTML Yorumu (HTML Comment)HTML Comment

Kavram AçıklamasıConcept Explanation

Kaynak kodu yorumlarına HTML etiketi gömmek, o yorumu IDE'de neredeyse okunamaz hale getirir. Javadoc gibi araçlar HTML çıktı üretiyorsa, HTML formatlama sorumluluğu araca aittir, programcıya değil. Embedding HTML tags in source code comments makes those comments nearly unreadable in the IDE. If tools like Javadoc generate HTML output, the responsibility for HTML formatting belongs to the tool, not the programmer.

Kötü Örnek (Before)Bad Example (Before)

KÖTÜ — HTML etiketleri yorumdaBAD — HTML tags in comment 
/**
 * Kullanım:
 * <pre>
 * &lt;task name=&quot;myTask&quot; /&gt;
 * </pre>
 */
public void runTask() { ... }

İyi Örnek (After)Good Example (After)

İYİ — Sade yorum, harici doküman referansıGOOD — Plain comment, external doc reference 
/**
 * Görevi çalıştırır.
 * Kullanım detayları için: docs/tasks.md
 */
public void runTask() { ... }
Pratik İpucuPractical Tip Kullanım örnekleri ve uzun açıklamalar için harici dokümantasyon araçları (Docusaurus, MkDocs, Confluence) kullanın. Kaynak kodundaki yorumlar sade ve kısa olmalıdır. For usage examples and long explanations, use external documentation tools (Docusaurus, MkDocs, Confluence). Comments in source code should be plain and brief.

Yerel Olmayan Bilgi (Nonlocal Information)Nonlocal Information

Kavram AçıklamasıConcept Explanation

Bir yorum, sadece yakınındaki kodu açıklamalıdır. Başka bir sınıftaki bir sabitin değeri veya uzaktaki bir konfigürasyonun detayı gibi yerel olmayan bilgiler yorumda yer almamalıdır — çünkü bu bilgi değiştikçe yorum güncellenmez ve yanlış kalır. A comment should only explain the code near it. Nonlocal information such as the value of a constant in another class or details of a distant configuration should not be in comments — because as that information changes, the comment won't be updated and will remain wrong.

Kötü Örnek (Before)Bad Example (Before)

KÖTÜ — Uzak bir konfigürasyon bilgisi yorumdaBAD — Remote configuration info in comment 
// Default timeout is 30 seconds (see Config.TIMEOUT_SECONDS)
public void setTimer(Timer t) {
    this.timer = t;
}

İyi Örnek (After)Good Example (After)

İYİ — Kod kendi varsayılanını içeriyorGOOD — Code contains its own default 
public static final int DEFAULT_TIMEOUT_SECONDS = 30;

        public void setTimer(Timer t) {
        this.timer = t;
        }
Pratik İpucuPractical Tip Bir yorumda "başka bir yerde tanımlanmış bir değer" hakkında bilgi veriyorsanız, o değeri ya koda taşıyın ya da yorumu silin. Yorumun güncel kalmasını sağlamak pratik olarak olanaksızdır. If a comment provides information about "a value defined elsewhere," either move that value into the code or delete the comment. Keeping such a comment up to date is practically impossible.
