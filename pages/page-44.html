Yan Etkilere Dikkat / Have No Side Effects Have No Side Effects
Yan etkiler (side effects) yalandır. Fonksiyonunuz tek bir şey yapma sözü verir, ama aynı zamanda başka gizli şeyler de yapar. Bazen kendi sınıfının değişkenlerinde beklenmedik değişiklikler yapabilir. Bazen fonksiyona geçirilen parametrelerde veya sistem genelindeki değişkenlerde (system globals) değişiklik yapabilir. Her iki durumda da bunlar sinsi ve zarar verici, garip zamansal bağımlılıklara (temporal couplings) ve sıra bağımlılıklarına (order dependencies) yol açan yanlış yönlendirmelerdir. Side effects are lies. Your function promises to do one thing, but it also does other hidden things. Sometimes it will make unexpected changes to the variables of its own class. Sometimes it will make them to the parameters passed into the function or to system globals. In either case they are devious and damaging mistruths that often result in strange temporal couplings and order dependencies.
Örneğin, Liste 3-6'daki görünüşte zararsız fonksiyonu ele alalım. Bu fonksiyon bir userName'i bir password ile eşleştirmek için standart bir algoritma kullanır. Eşleşirlerse true, bir şey yanlış giderse false döndürür. Ama aynı zamanda bir yan etkisi de vardır. Görebilir misiniz? Consider, for example, the seemingly innocuous function in Listing 3-6. This function uses a standard algorithm to match a userName to a password. It returns true if they match and false if anything goes wrong. But it also has a side effect. Can you spot it?
Listing 3-6: UserValidator.java Java 
public class UserValidator {
    private Cryptographer cryptographer;

    public boolean checkPassword(String userName, String password) {
        User user = UserGateway.findByName(userName);
        if (user != User.NULL) {
            String codedPhrase = user.getPhraseEncodedByPassword();
            String phrase = cryptographer.decrypt(codedPhrase, password);
            if ("Valid Password".equals(phrase)) {
                Session.initialize();
                return true;
            }
        }
        return false;
    }
}
Yan etki elbette Session.initialize() çağrısıdır. checkPassword fonksiyonu, adından da anlaşılacağı gibi, şifreyi kontrol ettiğini söyler. Adı, oturumu başlattığını ima etmez. Bu yüzden fonksiyon adına güvenen bir çağırıcı, kullanıcının geçerliliğini kontrol etmeye karar verdiğinde mevcut oturum verilerini silme riskiyle karşılaşır. The side effect is the call to Session.initialize(), of course. The checkPassword function, by its name, says that it checks the password. The name does not imply that it initializes the session. So a caller who believes what the name of the function says runs the risk of erasing the existing session data when he or she decides to check the validity of the user.
Bu yan etki bir zamansal bağımlılık (temporal coupling) yaratır. Yani checkPassword yalnızca belirli zamanlarda (başka bir deyişle, oturumu başlatmanın güvenli olduğu zamanlarda) çağrılabilir. Sıra dışı çağrılırsa oturum verileri istemeden kaybolabilir. Zamansal bağımlılıklar kafa karıştırıcıdır, özellikle yan etki olarak gizlendiğinde. Eğer zamansal bir bağımlılığınız olması gerekiyorsa, bunu fonksiyonun adında açıkça belirtmelisiniz. Bu durumda fonksiyonu checkPasswordAndInitializeSession olarak yeniden adlandırabilirdik, ancak bu da "Tek Bir Şey Yap" kuralını ihlal eder. This side effect creates a temporal coupling. That is, checkPassword can only be called at certain times (in other words, when it is safe to initialize the session). If it is called out of order, session data may be inadvertently lost. Temporal couplings are confusing, especially when hidden as a side effect. If you must have a temporal coupling, you should make it clear in the name of the function. In this case we might rename the function checkPasswordAndInitializeSession, though that certainly violates "Do one thing."

Yan Etkiler (Side Effects)Side Effects

Kavram AçıklamasıConcept Explanation

Yan etki, bir fonksiyonun adının vaat ettiğinin ötesinde yaptığı gizli işlemlerdir. Bu, sınıf değişkenlerini değiştirmek, global durumu etkilemek veya geçirilen parametreleri bozmak olabilir.A side effect is a hidden operation that a function performs beyond what its name promises. This could be modifying class variables, affecting global state, or corrupting passed parameters.

Kötü Örnek (Before)Bad Example (Before)

KÖTÜ — Gizli yan etkiBAD — Hidden side effect 
class ShoppingCart:
    def calculateTotal(self):
        total = sum(item.price for item in self.items)
        self.lastCalculated = datetime.now()  # Gizli!
        self.sendAnalytics(total)              # Gizli!
        return total

İyi Örnek (After)Good Example (After)

İYİ — Şeffaf fonksiyonlarGOOD — Transparent functions 
class ShoppingCart:
    def calculateTotal(self):
        return sum(item.price for item in self.items)

    def calculateAndTrack(self):
        total = self.calculateTotal()
        self.lastCalculated = datetime.now()
        self.sendAnalytics(total)
        return total
Pratik İpucuPractical TipFonksiyonun adını okuyun ve kendinize sorun: "Bu isim, fonksiyonun yaptığı HER ŞEYİ açıklıyor mu?" Eğer hayırsa, ya yan etkiyi kaldırın ya da ismi güncelleyin.Read the function name and ask yourself: "Does this name explain EVERYTHING the function does?" If not, either remove the side effect or update the name.

Zamansal Bağımlılık (Temporal Coupling)Temporal Coupling

Kavram AçıklamasıConcept Explanation

Zamansal bağımlılık, bir fonksiyonun yalnızca belirli bir zamanda veya belirli bir sırayla çağrılabilmesini ifade eder. Yan etki olarak gizlendiğinde tehlikeli olur çünkü çağırıcı bu kısıtlamadan haberdar değildir.Temporal coupling means a function can only be called at certain times or in a certain order. When hidden as a side effect, it's dangerous because the caller is unaware of this constraint.

Kötü Örnek (Before)Bad Example (Before)

KÖTÜ — Gizli sıra bağımlılığıBAD — Hidden order dependency 
class ReportGenerator {
    public Report generate() {
        // Gizli: veri kaynağına bağlanır!
        this.connection = Database.connect();
        return new Report(this.connection.query("SELECT..."));
    }

    public void cleanup() {
        // generate() çağrılmadan cleanup() çağrılamaz!
        this.connection.close();
    }
}

İyi Örnek (After)Good Example (After)

İYİ — Bağımlılık açıkça belirtilmişGOOD — Dependency made explicit 
class ReportGenerator {
    public Report connectAndGenerate() {
        try (Connection conn = Database.connect()) {
            return new Report(conn.query("SELECT..."));
        }
    }
}
Pratik İpucuPractical TipEğer fonksiyonlarınızın belirli bir sırayla çağrılması gerekiyorsa, ya isimlerinde bunu açıkça belirtin ya da bu sırayı ortadan kaldıracak şekilde yeniden tasarlayın (try-with-resources gibi).If your functions need to be called in a specific order, either make this explicit in their names or redesign to eliminate the ordering requirement (like try-with-resources).

Dürüst İsimlendirme (Honest Naming)Honest Naming

Kavram AçıklamasıConcept Explanation

Bir fonksiyonun adı, o fonksiyonun yaptığı her şeyi dürüstçe yansıtmalıdır. Eğer ad ile davranış uyumsuzsa, bu bir yalandır ve kodunuzu okuyan herkes için tuzak oluşturur.A function's name should honestly reflect everything that function does. If the name and behavior are inconsistent, that's a lie and creates traps for everyone reading your code.

Kötü Örnek (Before)Bad Example (Before)

KÖTÜ — İsim ile davranış uyumsuzBAD — Name doesn't match behavior 
function validateEmail(email) {
    const isValid = EMAIL_REGEX.test(email);
    if (isValid) {
        sendWelcomeEmail(email);  // Doğrulama mı, e-posta gönderme mi?
        incrementSignupCount();   // Sayaç artırma?!
    }
    return isValid;
}

İyi Örnek (After)Good Example (After)

İYİ — Her fonksiyon adına sadıkGOOD — Each function true to its name 
function isValidEmail(email) {
    return EMAIL_REGEX.test(email);
}

function registerNewUser(email) {
    if (!isValidEmail(email)) throw new Error("Geçersiz e-posta");
    sendWelcomeEmail(email);
    incrementSignupCount();
}
Pratik İpucuPractical Tip"En Az Şaşırtma İlkesi"ni (Principle of Least Surprise) hatırlayın: Fonksiyonun adını okuyan biri ne bekliyorsa, fonksiyon TAM da onu yapmalıdır. Ne eksik, ne fazla.Remember the "Principle of Least Surprise": Whatever someone expects from reading the function name, the function should do EXACTLY that. No less, no more.
